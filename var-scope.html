var・スコープ・初期化

varの使える場所、推論できない右辺、スコープ、確定代入を、コンパイル可否で判定できる形に整理します。

この単元の結論

var は「型を書かなくてもよい便利機能」ではありますが、試験では使える場所・推論される型・初期化の有無・スコープで落とされます。

最初に押さえるべき判断は次の4つです。

  1. ローカル変数か。 フィールド、メソッド引数、戻り値型なら即コンパイルエラー。
  2. 初期化式があるか。 var x; は型を推論できないので不可。
  3. 右辺だけで型が決まるか。 null、配列初期化子だけ、ラムダ式だけは不可。
  4. 推論後の型でその後の代入・メソッド呼び出しが成立するか。 var は動的型ではない。

varは動的型ではない

var は、ローカル変数の型をコンパイル時に右辺から推論する構文です。実行時に型が変わるわけではありません。

var n = 10;       // nの型はintに確定
var s = "abc";    // sの型はStringに確定

// n = "abc";     // NG: int型の変数にStringは代入できない
s = "def";        // OK: String同士
誤解しやすい点: var はJavaScriptの var とは別物です。Javaでは、型はコンパイル時に確定します。

初期化式がないvarは不可

var は右辺から型を推論します。したがって、右辺がなければ型を決められません。

// var a;              // NG: 初期化式がない
var b = 10;            // OK: int
var c = "abc";         // OK: String

ここで混同しやすいのが、通常のローカル変数宣言です。通常の型を明示すれば、宣言だけはできます。ただし、使う前には必ず初期化が必要です。

int x;                 // 宣言だけならOK
// System.out.println(x); // NG: ローカル変数xが初期化されていない
x = 10;
System.out.println(x); // OK

nullだけでは推論できない

null はどの参照型にも代入できる特別な値です。逆に言えば、null だけでは具体的な型が決まりません。

// var a = null;        // NG: 型を推論できない
var b = (String) null;  // OK: Stringと推論できる
String c = null;        // OK: 型を明示している
判断: null は「参照なし」を表す値であり、String 型そのものではありません。型推論の材料としては弱すぎます。

配列初期化子だけでは不可

配列初期化子 {1, 2, 3} は、左辺の型があって初めて意味が決まります。var では左辺の型がないため使えません。

// var a = {1, 2, 3};       // NG
var b = new int[]{1, 2, 3}; // OK: int[]
int[] c = {1, 2, 3};        // OK: 左辺にint[]がある

また、var[] のように配列型の要素型として var を使うこともできません。

// var[] nums = new int[3]; // NG
var nums = new int[3];      // OK: int[]と推論

ラムダ式だけでは不可

ラムダ式は、代入先の関数型インタフェースがあって初めて型が決まります。var の右辺にラムダ式だけを書いても、目標型がないため推論できません。

// var f = () -> System.out.println("A"); // NG
Runnable r = () -> System.out.println("A"); // OK

メソッド参照も同じです。

// var f = System.out::println; // NG
java.util.function.Consumer<String> c = System.out::println; // OK

複数変数を同時宣言できない

var は1つの宣言につき1つの変数です。通常の int a = 1, b = 2; のような書き方はできません。

int a = 1, b = 2;       // OK
// var x = 1, y = 2;    // NG
var x = 1;              // OK
var y = 2;              // OK

推論される型は「見た目より具体的」になりやすい

var は、右辺の型そのものに近い型で推論します。インタフェース型に自動で寄せてくれるわけではありません。

var list = new java.util.ArrayList<String>();
// listの型は ArrayList<String>。List<String> ではない。

java.util.List<String> list2 = new java.util.ArrayList<>();
// list2の型は List<String>。
Silverでは、厳密な総称型推論よりも「varが使える場所か」「その後の代入が成立するか」を優先して見れば十分です。

varとダイヤモンド演算子

new ArrayList<>() のように型引数を省略した右辺でも、var 自体は使えます。ただし、推論結果が意図より広くなることがあります。

var list1 = new java.util.ArrayList<String>(); // ArrayList<String>
var list2 = new java.util.ArrayList<>();       // ArrayList<Object> に近い扱いになりやすい

list1.add("A");
// list1.add(10);      // NG

list2.add("A");       // OK
list2.add(10);         // OK

学習用には、var とダイヤモンド演算子を同時に使うと型が読みづらくなる、と覚えておくと安全です。

final var

final と var は併用できます。final は再代入禁止、var は型推論です。役割が違います。

final var x = 10; // xはfinalなint
// x = 20;        // NG: finalなので再代入不可

スコープの基本

ローカル変数は、宣言されたブロック内でだけ使えます。ブロックとは、基本的に { } で囲まれた範囲です。

if (true) {
    var x = 10;
    System.out.println(x); // OK
}
// System.out.println(x);  // NG: xはスコープ外

for文の初期化部で宣言した変数も、for文の中だけで有効です。

for (var i = 0; i < 3; i++) {
    System.out.println(i);
}
// System.out.println(i);  // NG

同じ名前の再宣言

同じスコープ、または外側のスコープの変数がまだ見えている範囲では、同じ名前のローカル変数を再宣言できません。

void test() {
    var x = 10;
    // var x = 20;       // NG: 同じスコープで再宣言

    if (true) {
        // var x = 30;   // NG: 外側のxが見えている
    }
}

ただし、互いにスコープが重ならないブロックなら、同じ名前を使えます。

void test() {
    if (true) {
        var x = 10;
    }
    if (true) {
        var x = 20;      // OK: 先ほどのxはもう見えない
    }
}

フィールドとローカル変数の名前が同じ場合

ローカル変数がフィールドと同じ名前の場合、単に名前を書くとローカル変数が優先されます。フィールドにアクセスしたい場合は this. を使います。

class Sample {
    int value = 10;

    void test() {
        var value = 20;
        System.out.println(value);      // 20
        System.out.println(this.value); // 10
    }
}

確定代入: ローカル変数は使う前に必ず初期化

フィールドにはデフォルト値がありますが、ローカル変数にはデフォルト値がありません。var 以前の問題として、ローカル変数は使う前に初期化が必要です。

class Sample {
    int field; // 0で初期化される

    void test(boolean flag) {
        int local;
        if (flag) {
            local = 10;
        }
        // System.out.println(local); // NG: flagがfalseなら未初期化
    }
}

条件分岐がある場合は、すべての経路で初期化されるかを確認します。

void test(boolean flag) {
    int local;
    if (flag) {
        local = 10;
    } else {
        local = 20;
    }
    System.out.println(local); // OK

試験での判断順

  1. varがローカル変数として使われているか。 フィールド・引数・戻り値なら即エラー。
  2. 右辺があるか。 var x; は不可。
  3. 右辺だけで型が決まるか。 null、配列初期化子、ラムダ式単体は不可。
  4. 複数変数宣言になっていないか。 var a=1,b=2; は不可。
  5. 推論後の型で後続コードを読む。 以降は通常の型チェック問題として処理する。
  6. スコープ外参照・未初期化参照がないか。 特にif/for内の変数に注意する。

ミニ演習

Q1. var x = 10; x = 10L; は？

コンパイルエラーです。x は最初の代入で int と推論されています。long から int への暗黙変換はできません。

Q2. var x = (Object) "abc"; のxの型は？

Object です。右辺全体の型が Object なので、String ではありません。

Q3. for (var i=0; i<3; i++) { } System.out.println(i); は？

コンパイルエラーです。i のスコープはfor文の中だけです。

Q4. var list = new java.util.ArrayList<>(); list.add(1); list.add("A"); は？

コンパイルできます。型引数を明示していないため、要素型が広く推論されます。試験では、var と <> を同時に使うと型が読みづらくなる点に注意します。
