var・スコープ・初期化

varは動的型ではなく、ローカル変数の型推論です。禁止場所、初期化、スコープを整理します。

varの正体

var はローカル変数の型を右辺から推論するための構文です。コンパイル時点で型は確定します。JavaScriptのvarのような動的型ではありません。

var n = 10;      // int
var s = "abc";   // String
// n = "abc";    // NG: nはintとして確定済み

varが使える場所・使えない場所

場所可否理由
ローカル変数可初期化式から型を推論できる
拡張forの変数可要素型を推論できる
try-with-resourcesの変数可初期化式から型を推論できる
フィールド不可ローカル変数ではない
メソッド引数不可右辺がない
コンストラクタ引数不可右辺がない
戻り値型不可戻り値型推論ではない
catchパラメータ不可例外型を明示する必要がある

初期化式が必要

var は右辺から型を決めるため、初期化式なし、null だけ、配列初期化子だけでは型を決められません。

// var a;          // NG
// var b = null;   // NG
// var c = {1,2,3}; // NG
var d = new int[]{1,2,3}; // OK
var e = (String) null;    // OK: Stringと推論できる

スコープはブロックで決まる

ローカル変数は宣言されたブロック内でだけ使えます。ifやforの中で宣言した変数は、外側から見えません。

if (true) {
    var x = 10;
}
// System.out.println(x); // NG: スコープ外

for (var i = 0; i < 3; i++) {
    System.out.println(i);
}
// System.out.println(i); // NG

varとダイヤモンド演算子

var list = new ArrayList<String>(); なら型は明確です。一方、右辺が new ArrayList<>() だけだと型引数の文脈が弱くなります。試験では「何型として扱われるか」よりも、varが使える場所かどうかが優先論点です。

var list1 = new java.util.ArrayList<String>(); // ArrayList<String>
var list2 = new java.util.ArrayList<>();       // 型推論は可能。ただし意図が読みづらい

ミニ演習

Q. private var value; は？

コンパイルエラーです。フィールドに var は使えません。

Q. public void test(var x) は？

コンパイルエラーです。メソッド引数に var は使えません。

関連: 判定フロー / コンパイル・実行時例外 / ひっかけ集

varの推論結果を見抜く

varで宣言した変数の型は、右辺の式から決まります。以後、その型として固定されます。代入のたびに型が変わるわけではありません。

var x = 10;       // int
// x = 10L;       // NG: longをintへ暗黙変換できない

var y = 10L;      // long
y = 10;           // OK: intからlongへ拡大変換

ラムダとvar

Java Silverでは深掘りされすぎる可能性は低いですが、var をラムダ引数に使う場合はすべての仮引数で統一する必要があります。混在はできません。

// (var a, var b) -> a + b  // OKな形
// (var a, b) -> a + b      // NG: varあり/なし混在
