継承・interface・overload/override関係整理

継承で実装されるもの、interface実装、overloadとoverrideの違いを横断的に整理します。

まず全体像

継承、インタフェース、オーバーロード、オーバーライドは名前が似ていますが、見ている軸が違います。

用語何の話か試験で見る場所
継承 extends親クラスの機能を子クラスが引き継ぐフィールド・メソッド・コンストラクタ・アクセス修飾子
実装 implementsインタフェースの契約をクラスが満たすpublic実装、抽象メソッド、defaultメソッド
オーバーロード同じ名前で引数リストが違うメソッドを複数作るコンパイル時、引数型、曖昧な呼び出し
オーバーライド親のインスタンスメソッドを子で再定義する実行時、宣言型と実体、アクセス修飾子、戻り値型
ポリモーフィズム親型・インタフェース型で子の実体を扱う呼べるメソッドは宣言型、動くメソッドは実体

継承で引き継がれるもの・引き継がれないもの

対象サブクラスから使えるか注意
public/protectedメソッド基本使えるアクセス可能なら継承される
defaultアクセスのメソッド同じパッケージなら使える別パッケージのサブクラスからは不可
privateメソッド・privateフィールド直接使えない継承関係でもprivateは同じクラス内だけ
フィールドアクセス可能なら使える同名フィールドは隠蔽。多態的ではない
コンストラクタ引き継がれない子のコンストラクタ先頭でsuper(...)を呼ぶ
staticメソッド呼べるがオーバーライドではない隠蔽。宣言型やクラス名で解決される

メソッドは実体、フィールドは宣言型

class A {
  String name = "A";
  void print() { System.out.println("A"); }
}
class B extends A {
  String name = "B";
  @Override
  void print() { System.out.println("B"); }
}

A x = new B();
System.out.println(x.name); // A
x.print();                  // B

フィールドは変数の宣言型で決まります。オーバーライドされたインスタンスメソッドは実体で決まります。

interfaceを実装すると何が必要になるか

interface Worker {
  void work();              // public abstract
}

class Engineer implements Worker {
  public void work() {      // public必須
    System.out.println("work");
  }
}

インタフェースの抽象メソッドはpublicです。実装クラスで void work() のようにpublicを省くと、アクセスを弱めることになるためコンパイルエラーです。

抽象クラスがインタフェースを実装する場合は、メソッドを実装しなくてもよいです。ただし、その抽象クラスを継承した具象クラスは、最終的に未実装の抽象メソッドを実装する必要があります。

オーバーロードとオーバーライドの違い

比較オーバーロードオーバーライド
決まるタイミングコンパイル時実行時
見るものメソッド名と引数リスト親子関係、同じシグネチャ
戻り値だけ違う不可共変戻り値なら可
アクセス修飾子区別条件にならない親より狭くできない
static通常通り定義可能オーバーライドではなく隠蔽
class Sample {
  void test(int x) {}
  void test(long x) {}       // overload
  // int test(int x) { return 0; } // NG。戻り値だけではoverload不可
}

オーバーロード解決の典型順序

void test(long x) { System.out.println("long"); }
void test(Integer x) { System.out.println("Integer"); }
void test(int... x) { System.out.println("varargs"); }

test(10);

この場合は long が選ばれます。基本は「完全一致 → widening → boxing → varargs」の優先で考えます。ただし複数候補が同程度に見えると曖昧でコンパイルエラーになることがあります。

オーバーライド成立条件

  • インスタンスメソッドであること
  • メソッド名と引数リストが同じこと
  • 戻り値型が同じ、またはサブクラス型の共変戻り値であること
  • アクセス修飾子を親より狭くしないこと
  • チェック例外を親より広くしないこと
  • final メソッドではないこと
  • private メソッドはオーバーライド対象ではないこと

抽象クラス・interface・通常クラスの関係

interface A { void run(); }

abstract class B implements A {
  // run()を実装しなくてもよい。Bはabstractだから。
}

class C extends B {
  public void run() {}
}

通常クラスは、継承・実装によって残った抽象メソッドをすべて実装しなければいけません。抽象クラスは、未実装のまま残せます。

super() と this()

class A {
  A(int x) {}
}
class B extends A {
  B() {
    super(10); // コンストラクタの先頭に必要
  }
}

親に引数なしコンストラクタがない場合、子のコンストラクタで明示的に super(...) を書かなければいけません。this(...) と super(...) はどちらもコンストラクタの先頭にしか書けないため、同じコンストラクタ内で両方は呼べません。

sealedとinterfaceの組み合わせ

sealed interface Shape permits Circle, Square {}
final class Circle implements Shape {}
non-sealed class Square implements Shape {}

sealedなクラス・インタフェースを直接継承・実装する型は、許可リストに入り、さらに final / sealed / non-sealed のどれかを宣言します。
