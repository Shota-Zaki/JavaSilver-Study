例外処理

try/catch/finally、チェック例外、RuntimeException、try-with-resources、throw/throwsを整理します。

例外処理の判定順

例外問題は、まずコンパイル可否、次にtry内でどこまで実行されるか、最後にcatch/finally/closeの順番を追います。

try

例外が発生する可能性がある処理を書きます。例外発生後、try内の残りは実行されません。

catch

対応する例外型を受け取ります。上から順番に判定されます。

finally

例外の有無に関係なく基本的に実行されます。

throw / throws

throw は例外を投げる文、throws はメソッド宣言で例外送出可能性を示します。

チェック例外・非チェック例外

チェック例外はcatchまたはthrowsが必要です。RuntimeException とそのサブクラスは非チェック例外で、catchやthrowsを強制されません。

チェック例外

Exception のサブクラスのうち、RuntimeException 系ではないもの。例: IOException。

非チェック例外

RuntimeException とそのサブクラス。例: NullPointerException、NumberFormatException。

Error

OutOfMemoryError など。通常のアプリで回復処理の対象にしません。

Exception

広く受け取れますが、先に書くとサブクラスcatchが到達不能になります。

catchの順番

catchは上から判定されます。親クラスのcatchを先に書くと、子クラスのcatchは到達不能になりコンパイルエラーです。

try {
    throw new java.io.FileNotFoundException();
} catch (java.io.IOException e) {
    System.out.println("IO");
// } catch (java.io.FileNotFoundException e) { // NG。到達不能
}
catch順は「具体的な例外 → 抽象的な例外」です。親を先に置く選択肢は、実行時ではなくコンパイル段階で落ちます。

finally

finally は、例外が発生しても発生しなくても実行されます。catchが実行されない場合でもfinallyは実行されます。

try {
    System.out.println("A");
} catch (NullPointerException e) {
    System.out.println("B");
} finally {
    System.out.println("C");
}
// A C

構文上、try の後に finally を書き、その後に catch を書くことはできません。catch は finally より前です。

try-with-resources

try-with-resourcesで宣言されたリソースは、tryブロックを抜けると自動的に close() されます。close順は宣言と逆順です。

class A implements AutoCloseable {
    private final String name;
    A(String name) { this.name = name; }
    public void close() {
        System.out.println(name);
    }
}

try (A a = new A("A"); A b = new A("B")) {
    System.out.println("body");
} finally {
    System.out.println("finally");
}
// body B A finally

RuntimeExceptionの頻出

NullPointerException

null に対してメソッドやフィールドへアクセスしたとき。

ArrayIndexOutOfBoundsException

配列の有効範囲外にアクセスしたとき。

NumberFormatException

Integer.parseInt("abc") のような変換失敗。

ClassCastException

実体と互換性のない型へダウンキャストしたとき。
