recordとsealedの関係

Java 17専用の復習記事です。recordが自動生成するものと、sealedが制限する継承先を別々に整理し、両者を組み合わせたコードを読めるようにします。

recordは値のまとまりを表す、暗黙にfinalなクラス

record Point(int x, int y) {}

コンポーネントxとyから、private finalフィールド、同名アクセサ、正規コンストラクタ、equals、hashCode、toStringが用意されます。recordクラスは暗黙にfinalなので、別クラスが継承することはできません。

sealedは直接のサブタイプを列挙する

sealed interface Shape permits Circle, Rectangle {}
record Circle(double radius) implements Shape {}
record Rectangle(double width, double height) implements Shape {}

recordは暗黙にfinalなので、sealed型の直接サブタイプに必要なfinal/sealed/non-sealedの条件を満たします。上の2つのrecordはShapeを実装できます。

コンパイル時に見る場所

論点recordsealed
宣言側コンポーネントの型と名前permitsまたは同一コンパイル単位の直接サブタイプ
継承他クラスを明示的にextendsできず、record自体も継承不可許可された直接サブタイプだけが継承・実装可能
子側の指定暗黙にfinalfinal/sealed/non-sealedのいずれか
コンストラクタ正規またはコンパクト正規コンストラクタ通常のクラス・インタフェースの規則

誤りになる例

sealed interface Result permits Success {}
record Success(String value) implements Result {}

// record Failure(String message) implements Result {}
// コンパイルエラー: permitsに含まれていない

// class Child extends Success {}
// コンパイルエラー: recordは暗黙にfinal

記事末尾の確認

  • recordを「短い普通のクラス」とだけ説明せず、自動生成と制約を挙げられる。
  • recordがsealed型の直接サブタイプになれる理由を説明できる。
  • permits違反とrecord継承違反を別のコンパイルエラーとして判定できる。
