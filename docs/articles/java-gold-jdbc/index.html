JDBCの接続・SQL実行・トランザクション

JDBCコードは、SQLの細かい文法よりも、APIの役割、戻り値、例外処理、リソース解放、トランザクションを切り分けられるかで判断を誤りやすくなります。
ここではJava Goldで必要な読み方に絞って整理します。

広告スポンサーリンク表示エリア

JDBCとは

JDBCは、Javaプログラムからリレーショナルデータベースへ接続し、SQLを送信し、結果を受け取るための標準APIです。アプリケーションはJDBC APIを使い、実際のDB製品との差分はJDBCドライバが吸収します。

コードを読むときは「JDBCを使えばSQL文が自動で正しくなる」とは考えません。JDBCは接続・送信・取得の仕組みであり、SQL文自体の妥当性は別の論点です。

最小コード

JDBCの基本形は、接続を作り、SQLを準備し、実行し、結果を読む流れです。

import java.sql.*;

class Main {
    public static void main(String[] args) throws SQLException {
        String url = "jdbc:example://localhost/db";
        try (Connection con = DriverManager.getConnection(url);
             PreparedStatement ps = con.prepareStatement(
                 "select name from users where id = ?")) {

            ps.setInt(1, 10);

            try (ResultSet rs = ps.executeQuery()) {
                while (rs.next()) {
                    System.out.println(rs.getString("name"));
                }
            }
        }
    }
}
ResultSetは作られた直後に1行目を指しません。読む前にnext()を呼びます。

executeQuery / executeUpdate / execute

JDBCコードでは、実行メソッドの戻り値を先に判定します。

メソッド主な用途戻り値注意
executeQuery()SELECTResultSet更新件数ではない
executeUpdate()INSERT / UPDATE / DELETE / DDLint影響行数。ResultSetではない
execute()結果の種類が固定できないSQLbooleantrueなら最初の結果がResultSet
getResultSet()execute()後の結果取得ResultSetexecute()の判定後に使う
getUpdateCount()execute()後の更新件数取得int更新結果がない場合は-1

PreparedStatementの読み方

PreparedStatementでは、SQL中の?へ値を設定します。インデックスは0番始まりではなく1番始まりです。

PreparedStatement ps = con.prepareStatement(
    "select * from users where name = ? and age > ?");
ps.setString(1, "Alice");
ps.setInt(2, 20);
誤解正しい見方
setString(0, ...)で最初の?を指定するJDBCのパラメータ番号は1番始まり
?の数より多く値を設定してもよい対応しない位置指定は実行時にSQLException
SQL文字列に値を直接連結しても同じ直接連結にはSQLインジェクションの危険がある

SQLインジェクションとPreparedStatement

ユーザー入力をSQL文字列へ直接連結すると、入力内容がSQL構造として解釈される危険があります。

// 悪い例: 入力値をSQLに直接連結している
String sql = "select * from users where name = '" + name + "'";
Statement st = con.createStatement();
ResultSet rs = st.executeQuery(sql);
// 良い例: 値はプレースホルダへ渡す
PreparedStatement ps = con.prepareStatement(
    "select * from users where name = ?");
ps.setString(1, name);
ResultSet rs = ps.executeQuery();
PreparedStatementは「文字列結合を短く書く道具」ではありません。SQL構造と値を分離するために使います。

ResultSetはカーソルを進めて読む

ResultSetは表の結果を持ちますが、最初から1行目を指しません。next()で次の行へ進み、行があればtrueを返します。

try (ResultSet rs = ps.executeQuery()) {
    while (rs.next()) {
        int id = rs.getInt("id");
        String name = rs.getString("name");
        System.out.println(id + ":" + name);
    }
}
形意味注意
rs.next()次の行へ進む戻り値がfalseなら行がない
rs.getString(1)1列目を文字列で取得列番号も1番始まり
rs.getString("name")列名で取得列名の有無はSQL結果次第
rs.getInt(...)数値として取得SQL NULLの扱いはwasNull()も関係

トランザクション

複数の更新をひとまとまりとして成功・失敗させる仕組みがトランザクションです。JDBCではautoCommitが判定の軸です。

try (Connection con = DriverManager.getConnection(url)) {
    con.setAutoCommit(false);
    try {
        // update 1
        // update 2
        con.commit();
    } catch (SQLException e) {
        con.rollback();
        throw e;
    }
}
API意味注意
setAutoCommit(false)自動コミットを止める自分でcommitまたはrollbackする
commit()変更を確定する更新成功時
rollback()変更を戻す例外発生時など
setSavepoint()途中地点を作るより細かい戻しに使う

try-with-resourcesと閉じる順番

Connection、PreparedStatement、ResultSetはリソースです。基本的にtry-with-resourcesで閉じます。複数リソースは宣言と逆順に閉じられます。

try (Connection con = DriverManager.getConnection(url);
     PreparedStatement ps = con.prepareStatement("select * from users");
     ResultSet rs = ps.executeQuery()) {
    while (rs.next()) {
        System.out.println(rs.getString(1));
    }
}
この例では、終了時にResultSet、PreparedStatement、Connectionの順に閉じます。

SQLExceptionはchecked例外

JDBCの多くのメソッドはSQLExceptionを投げます。SQLExceptionはchecked例外なので、catchするかthrowsで宣言します。

void test(Connection con) throws SQLException {
    PreparedStatement ps = con.prepareStatement("select * from users");
}

RuntimeExceptionではないため、何も処理せずに呼び出すとコンパイルエラーになります。

不可能・誤りパターン

コード・考え方結果理由
ResultSet rs = ps.executeUpdate();コンパイルエラーexecuteUpdate()の戻り値はint
int n = ps.executeQuery();コンパイルエラーexecuteQuery()の戻り値はResultSet
ps.setString(0, "A");実行時にSQLExceptionパラメータ番号は1番始まり
rs.getString(1)をrs.next()前に呼ぶ実行時にSQLExceptionになり得るカーソルが行を指していない
SQLExceptionを処理しないコンパイルエラーchecked例外
SQL文字列へユーザー入力を直接連結するコンパイルは通るSQLインジェクションの危険がある
commit()すれば必ず成功保証されないSQLExceptionが発生し得る

コードの読み方

  1. 戻り値型を見る。executeQueryはResultSet、executeUpdateはint。
  2. SQLExceptionを処理しているか見る。
  3. ResultSetを読む前にnext()しているか見る。
  4. PreparedStatementのパラメータ番号が1番始まりか見る。
  5. トランザクションではautoCommit、commit、rollbackを見る。
  6. try-with-resourcesの閉じる順番を見る。

よくある誤解を整理する

executeQueryは更新件数を返さない

executeQuery()はResultSetを返します。更新件数が必要ならexecuteUpdate()を使います。

ResultSetは最初から1行目を指さない

next()でカーソルを進めてから列値を読みます。

PreparedStatementの番号は0から始まらない

setString(1, ...)のように1番から指定します。

SQLExceptionは未処理で呼び出せない

checked例外なので、catchまたはthrowsが必要です。

確認ポイントの補足

JDBCコードでは、SQL文そのものよりも、JDBC APIの戻り値、パラメータ番号、例外処理、トランザクションを先に見ます。SQLExceptionはchecked例外なので、catchまたはthrowsが必要です。

見る対象判定誤読例
executeQueryResultSetを返す。主にSELECTで使う更新件数が返ると読む
executeUpdate更新件数をintで返すResultSetが返ると読む
PreparedStatement#setXxxパラメータ番号は1始まり配列感覚で0始まりにする
setAutoCommit(false)commitまたはrollbackで確定・取り消しを制御する更新SQLを実行した時点で必ず確定すると読む

try-with-resourcesでは、宣言した逆順にリソースが閉じられます。Connection、PreparedStatement、ResultSetのどれを宣言しているかで、close対象が変わります。
