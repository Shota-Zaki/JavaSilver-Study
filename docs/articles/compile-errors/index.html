コンパイルエラーの読み方と原因まとめ

出力を追うコードは、出力を追う前にコンパイルで止まることが多いです。
ここでは、原因を「何がエラーか」「なぜエラーか」「どう直すか」「javacのメッセージをどう読むか」「どこで混同しやすいか」まで分けて整理します。

広告スポンサーリンク表示エリア

Java実行・宣言ルールの全体像

Javaの実行ルールは、main メソッドだけを見ると断片的になります。まず、ソースコードが実行されるまでの流れを固定します。

Sample.java  --javac-->  Sample.class  --java-->  JVMがmainを呼び出す
用語役割判定ポイント
ソースファイル.java のファイル。人間が書くコード。public トップレベルクラス名とファイル名が一致するか。
コンパイルjavac がソースを検査し、classファイルを作る処理。型、構文、未初期化、アクセス制御、例外処理漏れを検査する。
classファイルJVMが実行できるバイトコード。コンパイルエラーなら作られない。したがって実行も出力もない。
JVMclassファイルを読み込み、実行する仕組み。実行時例外は、JVMが実行中にコードへ到達したときに発生する。
JDK開発に必要な道具一式。javac や java を含む。javac と java の役割を混同しない。
重要: 「コンパイルできること」と「javaコマンドで起動できること」は別です。構文上正しいクラスでも、正しい main がなければ起動用クラスにはなりません。

ファイル名・クラス名・mainメソッドを分ける

観点確認すること判定
ファイル名public トップレベルクラスと一致しているか。一致しなければコンパイルエラー。
クラス宣言トップレベルクラスを複数書けるか。書ける。ただし public は1つまで。
mainメソッドjavaコマンドの入口として使える形か。public static void main(String[] args) または String... args。
起動引数args に何が入るか。渡された文字列の配列。引数なしなら長さ0で、null ではない。
 
// A.java
public class A { }

class B {
    public static void main(String[] args) {
        System.out.println("B");
    }
}

この例では、ファイル名は A.java で正しいためコンパイルできます。B は public ではありませんが、正しい main を持っていれば実行対象になれます。つまり、publicクラス名の規則 と 実行入口の規則 は別です。

3分類を混ぜない

Javaの候補は、しばしば「コンパイルエラー」「実行時例外」「出力結果」を混ぜてきます。ここを混同すると、解説を読んでも同じ間違いを繰り返します。

分類発生タイミング例
コンパイルエラー実行前。classファイルを作れない型不一致、未初期化ローカル変数、override違反、catch到達不能
実行時例外コンパイル後、実行中NullPointerException、ArrayIndexOutOfBoundsException、ClassCastException
正常出力コンパイルも実行も成功printlnの結果、または何も表示されない

コンパイルエラーの典型

int a;
System.out.println(a); // 未初期化ローカル変数

byte b = 1;
b = b + 1;             // b + 1 はint

class A { private void test() {} }
class B extends A { @Override void test() {} } // privateはoverride対象外

コンパイルエラーは、JVMが実行する前に発生します。したがって「途中まで表示される」は原則ありません。

実行時例外の典型

String s = null;
System.out.println(s.length()); // NullPointerException

int[] a = new int[0];
a[0] = 10;                     // ArrayIndexOutOfBoundsException

Object obj = "abc";
Integer n = (Integer) obj;      // ClassCastException

実行時例外は、該当行に到達して初めて起きます。到達しない行の例外は発生しません。

到達不能コードとcatch順序

catchは上から順に評価されます。スーパークラスの例外を先にcatchすると、サブクラスのcatchは到達不能になりコンパイルエラーです。

try {
    throw new java.io.IOException();
} catch (Exception e) {
} catch (java.io.IOException e) { // NG: 先にExceptionで捕まる
}

何も表示されないケース

「何も表示されない」は、コンパイルエラーや例外とは別です。正常に実行されたが println に到達しない、または空文字を出力するなどのケースです。

String s = "";
System.out.print(s); // 正常終了。ただし見た目は何も出ない

判定練習

Q. String s = null; System.out.println(s); は例外？

例外ではありません。 println(Object) はnullを文字列 null として表示します。 s.length() のようにインスタンスメソッドを呼ぶと例外

Q. コンパイルエラーのコードは、途中のprintlnだけ実行される？

されません。そもそも実行用のclassファイルを作れないので、出力はありません

関連: 判定フロー / コンパイル・実行時例外 / 混同しやすい点

「途中まで出る」かどうかの境界

コンパイルエラーなら何も実行されません。実行時例外なら、例外発生前の出力は残ります。この違いが候補を切る鍵です。

// コンパイルエラー: 何も表示されない
int a;
System.out.println("A");
System.out.println(a);

// 実行時例外: Aは表示される
System.out.println("A");
String s = null;
s.length();
System.out.println("B");

例外がcatchされる場合とされない場合

発生した例外型がcatchの型に代入可能ならcatchされます。代入可能でなければ、そこで呼び出し元へ伝播します。

try {
    Object o = "abc";
    Integer i = (Integer) o;
} catch (NullPointerException e) {
    System.out.println("NPE");
}
// ClassCastExceptionはcatchされない。異常終了。

コンパイルエラー：例外処理

原因典型コード理由見抜き方
チェック例外未処理throw new IOException();catchするかthrows宣言が必要。Exception 直下や IOException を見る。
catch順序不正catch(Exception) の後に catch(IOException)後続catchが到達不能。親例外を先にcatchしていないか。

実行時例外：null・範囲外・キャスト

例外典型コード原因注意点
NullPointerExceptions.length() where s == nullnull参照でメソッド/フィールドアクセス。System.out.println(s) だけなら null 表示で例外を投げない。
ArrayIndexOutOfBoundsExceptionarray[array.length]配列範囲外。最後の有効添字は length - 1。
StringIndexOutOfBoundsExceptionstr.charAt(str.length())文字列範囲外。substring(length) は空文字で可能、charAt(length) は不可。
IndexOutOfBoundsExceptionlist.get(list.size())List範囲外。配列とは例外クラス名が違う。
ClassCastExceptionString s = (String) obj;実体が変換先型ではない。コンパイル可能でも実体が変換先型でなければClassCastExceptionが発生する。
ArrayStoreExceptionObject[] a = new String[1]; a[0] = new Object();実体配列の要素型に合わない。配列の共変性で出る。
ArithmeticException10 / 0整数の0除算。小数の0除算とは挙動が違う。
NumberFormatExceptionInteger.parseInt("A")数値へ変換できない。起動引数の変換コードでよく使う。
UnsupportedOperationExceptionList.of("A").add("B")変更不可リストへの変更操作。List.of や固定サイズリストに注意。

実行時例外：catchされる場合・されない場合

例外が発生しても、対応する catch があれば処理は継続します。対応する catch がなければ、そこで異常終了します。finally がある場合は、異常終了前にも実行されます。

try {
    System.out.println("A");
    String s = null;
    s.length();
    System.out.println("B");
} catch (NullPointerException e) {
    System.out.println("C");
} finally {
    System.out.println("D");
}
System.out.println("E");

この場合、出力は A C D E です。B は例外発生後なので実行されません。catchで処理できたため、finally後に後続処理へ進みます。

チェック例外と非チェック例外の見分け

分類代表例コンパイラの処理要求覚え方
チェック例外Exception, IOExceptioncatch または throws が必要。RuntimeException 系ではない Exception。
非チェック例外RuntimeException, NullPointerException, ClassCastException, NumberFormatException処理しなくてもコンパイル可能。実行時のプログラムミス系。
ErrorOutOfMemoryError など通常処理対象にしない。コードでは例外階層として区別。
RuntimeExceptionかどうかを見る。
非チェック例外は、発生する式と例外クラスを対応させて整理します。NullPointerException、ArrayIndexOutOfBoundsException、ClassCastException、NumberFormatException、ArithmeticException は非チェック例外として即判定できるようにしてください。

エラー判定の実戦フロー

  1. 宣言を見る。 ファイル名、package/import、クラス修飾子、mainメソッド、変数宣言を判定する。
  2. 型を見る。 代入、戻り値、引数、演算結果、キャストの可否を判定する。
  3. アクセスを見る。 private/default/protected/public、staticからの参照、継承関係を見る。
  4. override/interfaceを見る。 アクセス、戻り値、throws、abstract未実装、default衝突を判定する。
  5. チェック例外を見る。 IOExceptionなどがcatch/throwsされているか判定する。
  6. ここまで通ったら実行時へ進む。 null、範囲外、キャスト、0除算、数値変換を左から追う。
  7. 例外処理を見る。 catchされるか、finallyが実行されるか、後続処理へ進むかを判断する。

static context：staticからインスタンスを直接触っている

non-static variable ... cannot be referenced from a static context は、static メソッドからインスタンスフィールドやインスタンスメソッドを直接参照したときに出ます。

public class Sample {
    private int value = 10;

    public static void main(String[] args) {
        System.out.println(value); // NG
    }
}

main はstaticなので、Sample のインスタンスがなくても呼び出されます。インスタンスごとに値を持つ value を直接読めません。

public class Sample {
    private int value = 10;

    public static void main(String[] args) {
        Sample s = new Sample();
        System.out.println(s.value); // OK
    }
}
書き方可否理由
static メソッド → static フィールド可能クラスに属するため。
static メソッド → インスタンスフィールドを直接参照不可どのインスタンスの値か決まらない。
static メソッド → インスタンス生成後に参照可能対象インスタンスが明確。
インスタンスメソッド → static フィールド可能インスタンス側からクラスメンバへアクセス可能。

unreported exception：チェック例外の処理漏れ

unreported exception ...; must be caught or declared to be thrown は、チェック例外を処理していないという意味です。

import java.io.IOException;

public class Main {
    public static void main(String[] args) {
        throw new IOException(); // NG
    }
}

修正方法は2つです。catch するか、呼び出し元へ投げることを throws で宣言します。

// 方法1: catchする
try {
    throw new IOException();
} catch (IOException e) {
    System.out.println("handled");
}

// 方法2: throwsする
public static void main(String[] args) throws IOException {
    throw new IOException();
}
例外チェック要否処理しないとどうなるか
IOException必要コンパイルエラー。
Exception必要コンパイルエラー。
RuntimeException不要コンパイル可能。
NullPointerException不要コンパイル可能。発生すれば実行時例外。
Error不要通常は処理対象にしない。

missing return / unreachable statement：制御フローのエラー

戻り値が必要なメソッドでは、すべての実行経路で値を返す必要があります。人間が「たぶん通る」と思っても、コンパイラが保証できなければエラーです。

int score(boolean ok) {
    if (ok) {
        return 100;
    }
    // NG: okがfalseのとき戻り値がない
}
int score(boolean ok) {
    if (ok) {
        return 100;
    }
    return 0; // OK
}

unreachable statement は、絶対に実行できない文がある場合です。

void test() {
    return;
    System.out.println("A"); // NG: 到達不能
}
パターン結果注意点
return の直後到達不能原則コンパイルエラー。
throw の直後到達不能例外を投げた後には進まない。
while(false) の中到達不能ループ本体に入れない。
if(false) の中特別扱いされる場合あり到達不能判定の細部に踏み込みすぎない。コードを読むときは典型形を優先。

構文エラー：波カッコ・セミコロン・宣言位置

illegal start of expression、reached end while parsing、class, interface, enum, or record expected は、波カッコや宣言位置の崩れで出やすいエラーです。

メッセージありがちな原因見る場所
reached end while parsing} が足りないファイル末尾からブロック対応を見る。
class, interface, enum, or record expectedクラス外に文を書いている余計な } でクラスが閉じていないか。
illegal start of expressionメソッド内にメソッドを書いた、宣言位置が不正直前の {} とセミコロンを見る。
';' expectedセミコロン不足指摘行だけでなく前の行も見る。
not a statement単独で文にならない式a + b; のような意味のない式。
 
public class Main {
    public static void main(String[] args) {
        System.out.println("A");
}
System.out.println("B"); // NG: クラス外に文が出ている

package / import / classpath由来のコンパイルエラー

構文が正しくても、package、import、コンパイル位置、-d、-cp がずれるとエラーになります。コマンド系は javac・javaコマンド の記事と合わせて見ると効果があります。

メッセージ原因例修正
package ... does not existimport先が見つからないimport ex.Item;コンパイル対象、-cp、ディレクトリ構成を見る。
cannot find symbol class XクラスXが見つからないnew Item()同時にコンパイルするか、classの場所をclasspathへ入れる。
bad source fileファイル位置・package・クラス名が不一致package ex; なのに配置がずれているjavac -d out を使って出力先を整理。
duplicate class同じ完全修飾名のクラスが複数ある同じpackage・同じclass名重複ファイルを除外するか名前を変える。
class X is public, should be declared in a file named X.javapublicクラス名とファイル名不一致public class A を B.java に保存A.java にする。
 
// src/ex/Item.java
package ex;
public class Item { }

// src/Main.java
import ex.Item;
public class Main {
    public static void main(String[] args) {
        System.out.println(new Item());
    }
}
# OK例
javac -d out src/ex/Item.java src/Main.java
java -cp out Main

package ex; のあるクラスを利用するなら、コンパイラとJVMが ex.Item を見つけられる状態にする必要があります。

文字コード由来のエラー

Windows環境では、ファイルがUTF-8なのに javac がMS932として読もうとして、次のようなエラーが出ることがあります。

エラー: この文字は、エンコーディングMS932にマップできません

この場合は、コンパイル時に文字コードを指定します。

javac -encoding UTF-8 Sample.java
javac -encoding UTF-8 -d out src/ex/Sample.java
状況対応注意
UTF-8のソースをWindowsのコマンドでコンパイルjavac -encoding UTF-8文字化けしたまま直さない。
パッケージ付きソース-d out も併用class出力先とpackage階層を合わせる。
日本語コメントだけが問題コメントでも読み取り時点でエラーになるコメントだから無関係、とは限らない。

@Overrideで出るエラーはむしろ手がかり

@Override を付けてエラーになった場合、「オーバーライドできていない」ことが分かります。消して通すのは悪手です。@Override があることでコンパイルエラーを判断しやすくなる場合があります。

NG原因例なぜoverrideにならないか
引数型違い親 test(int) / 子 test(long)シグネチャが違う。オーバーロード。
戻り値型不正親 Number get() / 子 Object get()戻り値を広げている。
アクセスを狭くする親 public / 子 protected呼び出し可能範囲を狭めている。
privateメソッド親 private void m()継承されないためoverride対象外。
staticメソッド親子で同名staticオーバーライドではなく隠蔽。
throwsを広げる親 throws IOException / 子 throws Exceptionチェック例外を広げられない。
 
class A {
    public void test(int n) {}
}
class B extends A {
    @Override
    public void test(long n) {} // NG: 引数型が違う
}

warningとerrorを混同しない

javac は、コンパイルを止める error と、注意喚起の warning を分けます。Goldではジェネリクスのunchecked warningが出やすいですが、warningだけならclassファイルは作られます。

import java.util.*;

public class Main {
    public static void main(String[] args) {
        List list = new ArrayList();
        list.add("A");
        List<Integer> nums = list; // unchecked warning
    }
}
javac -Xlint:unchecked Main.java
分類class生成扱い
error原則生成されないコンパイルエラーを選ぶ。
warning生成されるwarningだけなら実行結果まで追う。
unchecked warning生成されるraw typeとジェネリクス混在で出やすい。
deprecation warning生成される非推奨APIの警告。コンパイル失敗ではない。

実戦用：エラー名から原因を逆引き

エラー名・文言まず疑う場所先に見る場所
cannot find symbol名前・スコープ・import・変数型symbol と location。
incompatible types代入・戻り値・引数右辺型と左辺型。
method ... cannot be appliedメソッド呼び出し引数の個数・型・順番。
constructor ... cannot be appliednewしている箇所コンストラクタ定義と呼び出し。
non-static ... static contextmain やstaticメソッド内インスタンス生成の有無。
unreported exceptionチェック例外catch/throws。
missing return statement戻り値ありメソッド全経路でreturnしているか。
unreachable statementreturn / throw の後絶対に実行されない文。
reached end while parsing波カッコ不足{ と } の対応。
class, interface, enum, or record expectedクラス外の文余計な }。
should be declared in a file named ...ファイル名publicトップレベルクラス名。

ミニ演習

演習1：コンパイルか実行時か

String s = null;
System.out.println(s.length());

結果: コンパイルは可能。実行時に NullPointerException

演習2：ローカル変数

int x;
if (args.length > 0) {
    x = 10;
}
System.out.println(x);

結果: コンパイルエラー。 if に入らない経路では x が未初期化

演習3：catch順序

try {
    throw new java.io.IOException();
} catch (Exception e) {
} catch (java.io.IOException e) {
}

結果: コンパイルエラー。先に Exception で捕まえるため、後続の IOException は到達不能

最後に見るチェックリスト

  • 出力を追う前にコンパイルエラー候補を除外したか。
  • ローカル変数とフィールドの初期値ルールを混同していないか。
  • var をフィールド・引数・戻り値に使っていないか。
  • 小さい整数型の演算結果を int として見ているか。
  • overrideでアクセス・戻り値・throwsを判定したか。
  • チェック例外をcatchまたはthrowsしているか。
  • nullのメソッド呼び出し、範囲外、キャスト失敗を左から追ったか。
  • 例外発生後、tryブロックの残りを実行していないか。
  • catchされた後にfinallyと後続処理へ進むかを見たか。
