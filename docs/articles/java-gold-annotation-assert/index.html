アノテーションとassert

アノテーションとassertは、コードの通常処理そのものよりも、コンパイラ・実行時・ツールに伝える情報として扱われます。
Java Goldでは、どこに付けられるか、いつ有効か、実行結果に影響するかを分けて読みます。

広告スポンサーリンク表示エリア

アノテーションとは

アノテーションは、クラス、メソッド、フィールド、引数などに付ける補足情報です。処理を直接書く構文ではなく、コンパイラ、実行環境、ライブラリ、ツールが読むための情報です。

@Override
public String toString() {
    return "sample";
}
アノテーションを付けただけで処理が自動実行されるとは限りません。誰がそのアノテーションを読むかで意味が決まります。

標準アノテーション

アノテーション主な役割確認する点
@Overrideオーバーライドしていることをコンパイラに判定させる成立しなければコンパイルエラー
@Deprecated非推奨APIであることを示す使用すると警告。通常はコンパイルエラーではない
@SuppressWarnings特定の警告を抑制するエラーは抑制できない
@FunctionalInterface関数型インタフェースであることを判定させる抽象メソッドが1つでなければコンパイルエラー
@SafeVarargsgeneric varargsの安全性を表明する付けられる場所に制限がある

@Override

@Overrideは、メソッドがスーパークラスまたはインタフェースのメソッドを上書きしていることをコンパイラに判定させます。オーバーライドにならない場合はコンパイルエラーです。

class A {
    void test(int n) {}
}
class B extends A {
    @Override
    void test(String s) {} // コンパイルエラー: 引数が違う
}

このコードでは、test(String)はtest(int)のオーバーライドではなくオーバーロードです。そのため@Overrideを付けるとエラーになります。

原因結果
引数型が違うオーバーライドにならない
戻り値型が不正コンパイルエラー
アクセス修飾子を狭くするコンパイルエラー
staticメソッドをインスタンスメソッドで上書きするコンパイルエラー
privateメソッドを上書きしようとするオーバーライドにならない

@Deprecated

@Deprecatedは、今後の利用を推奨しないAPIであることを示します。使っただけで即コンパイルエラーになるわけではありません。

class OldApi {
    @Deprecated
    void oldMethod() {}
}
class Main {
    public static void main(String[] args) {
        new OldApi().oldMethod(); // 警告対象
    }
}
警告とエラーを混同すると誤りになります。@Deprecatedは通常、コンパイルを止めません。

@SuppressWarnings

@SuppressWarningsは警告を抑制します。警告を除外するためのものであり、コンパイルエラーを除外するものではありません。

@SuppressWarnings("unchecked")
void test(java.util.List list) {
    java.util.List<String> names = list; // unchecked warning対象
}
指定値対象の例
uncheckedraw typeとジェネリクス混在など
deprecation非推奨APIの利用
varargsvarargs関連警告

@FunctionalInterface

@FunctionalInterfaceは、そのインタフェースが関数型インタフェースであることをコンパイラに判定させます。抽象メソッドが1つなら成立します。

@FunctionalInterface
interface Runner {
    void run();
    default void stop() {}
    static Runner of(Runner r) { return r; }
}

defaultメソッドやstaticメソッドは、抽象メソッド数に数えません。Object由来のtoStringなどを宣言しても、通常は関数型インタフェースの抽象メソッド数には数えません。

@FunctionalInterface
interface Bad {
    void a();
    void b(); // コンパイルエラー: 抽象メソッドが2つ
}

@SafeVarargs

@SafeVarargsは、ジェネリックな可変長引数に対してヒープ汚染につながる操作を行わないことを表明するアノテーションです。どこにでも付けられるわけではありません。

class Util {
    @SafeVarargs
    static <T> void printAll(T... values) {
        for (T v : values) {
            System.out.println(v);
        }
    }
}

代表的に使える場所は、staticメソッド、finalメソッド、privateメソッド、コンストラクタです。オーバーライドされ得る通常のインスタンスメソッドには付けられません。

class Bad {
    @SafeVarargs
    <T> void test(T... values) {} // コンパイルエラー
}

カスタムアノテーション

アノテーション型は@interfaceで定義します。通常のinterfaceとは書き方と目的が異なります。

public @interface Info {
    String value();
    int version() default 1;
}

要素にはメソッドのような形を書きます。defaultで既定値を指定できます。

@Info(value = "sample", version = 2)
class Sample {}
ルール注意
要素に引数を書かないString name(String s);のようにはしない
戻り値型は制限されるプリミティブ、String、Class、enum、annotation、それらの配列など
default値を指定できる指定しない場合、利用側で値が必要
要素名がvalueだけなら省略記法が使える@Info("sample")のように書ける

メタアノテーション

アノテーションに付けるアノテーションをメタアノテーションと呼びます。どこに付けられるか、いつまで残るかを決めます。

メタアノテーション意味確認する点
@Target付けられる場所を指定TYPE、METHOD、FIELD、PARAMETERなど
@Retentionどの段階まで残すかSOURCE、CLASS、RUNTIME
@DocumentedJavadocに含める実行結果には直接関係しない
@Inheritedクラスへのアノテーションをサブクラスへ継承メソッドやフィールドには効かない
@Repeatable同じアノテーションを複数付けられるコンテナアノテーションが必要
import java.lang.annotation.*;

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
@interface TestInfo {
    String value();
}

@Target

@Targetは、アノテーションを付けられる場所を制限します。指定と異なる場所に付けるとコンパイルエラーです。

ElementType対象
TYPEclass、interface、enum、annotation
METHODメソッド
FIELDフィールド
PARAMETER仮引数
CONSTRUCTORコンストラクタ
LOCAL_VARIABLEローカル変数
TYPE_USE型が現れる場所
TYPE_PARAMETER型パラメータ

@Retention

@Retentionは、アノテーション情報をどの段階まで残すかを決めます。

RetentionPolicy意味注意
SOURCEソースコード上だけコンパイル後には残らない
CLASSclassファイルまで残る実行時リフレクションで読めるとは限らない
RUNTIME実行時まで残るリフレクションで読めます
実行時にリフレクションで読みたいならRUNTIMEが必要です。

assertとは

assertは、開発中の前提条件を見るための構文です。通常の入力チェックや業務例外処理の代わりに使うものではありません。

int n = -1;
assert n >= 0;
System.out.println("end");

assertはデフォルトでは無効です。無効なら条件式は評価されず、そのまま次へ進みます。有効にすると、条件がfalseのときAssertionErrorが発生します。

assertの有効化

assertを有効にするには、実行時に-eaまたは-enableassertionsを指定します。

java Main
java -ea Main
java -enableassertions Main
実行方法assert条件がfalseの場合
java Main無効何も起きない
java -ea Main有効AssertionError
javacではなくjava実行時に有効化します。

assertの2つの形

形意味
assert 条件式;条件がfalseならAssertionError
assert 条件式 : 式;条件がfalseなら式の値をメッセージとして持つAssertionError
int value = -1;
assert value >= 0 : "value must be positive";

コロンの右側の式は、assertが有効で、条件がfalseの場合に評価されます。

assertで避けるべき使い方

assertは実行時オプションで無効になり得ます。引数検証や状態更新など、通常動作に必須の処理をassertだけに置いてはいけません。

int i = 0;
assert ++i > 0;
System.out.println(i);
実行出力理由
java Main0assert無効。++iが評価されない
java -ea Main1assert有効。条件式が評価される
assertの条件式に副作用を入れると、有効・無効で実行結果が変わります。

不可能・誤りパターン

コード・考え方結果理由
@Overrideを付ければオーバーライドになるならない条件を満たさないとコンパイルエラー
@SuppressWarningsでコンパイルエラーを除外する除外できない抑制できるのは警告
@FunctionalInterfaceを付ければラムダ対象になるならない抽象メソッドが1つである必要がある
@SafeVarargsを通常のインスタンスメソッドへ付けるコンパイルエラー付けられる場所に制限がある
@Retention(CLASS)なら実行時に必ず読めます読めません実行時に読むならRUNTIME
java Mainでassertが必ず動く動かない通常は無効。-eaが必要
assertを入力チェックの代わりにする代わりにならない無効化できるため通常の検証処理に使わない

よくある誤解を整理する

アノテーションを付ければ処理が自動実行されるわけではない

アノテーションは情報です。コンパイラ、実行環境、ライブラリなどが読んで初めて意味を持ちます。

@Overrideはオーバーライドを成立させない

成立しているかを検査します。成立しない場合はコンパイルエラーです。

@Deprecatedはコンパイルを止めない

通常は警告です。エラーとは分けて判定します。

@SuppressWarningsはエラーを除外しない

警告を抑制するためのものです。型不一致や構文エラーは消せません。

assertは通常実行では動かない

-eaを付けて実行した場合に有効になります。

assertに副作用を書くと結果が変わる

assertが無効な場合、条件式自体が評価されません。

コードの読み方

  1. まずアノテーションが付けられる場所として正しいかを見る。
  2. @Override、@FunctionalInterface、@SafeVarargsはコンパイル可否に直結しやすい。
  3. @Deprecatedと@SuppressWarningsは、警告とエラーを分ける。
  4. @Retentionは、実行時に読めるかを判定する。
  5. assertコードでは、-eaが指定されているかを最初に見る。
  6. assert条件式に副作用がある場合、有効・無効で出力が変わる。
