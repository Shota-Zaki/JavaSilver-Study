Java 17における継承の完全整理

ここでは、通常クラス、抽象クラス、インターフェースを中心に、Java 17の継承規則を整理します。

広告スポンサーリンク表示エリア

まず、最重要の区別は次の3つです。

対象 子クラス側の動作 呼び出しの決まり
インスタンスメソッド オーバーライド 実行時のオブジェクト型
staticメソッド 隠蔽（ハイディング） 参照変数・型名のコンパイル時型
フィールド 隠蔽 参照変数のコンパイル時型
コンストラクタ 継承されない this(...)またはsuper(...)で呼ぶ

1. 継承とは

クラスの継承はextendsで表します。

class Parent {
}

class Child extends Parent {
}

ChildはParentのサブクラス、ParentはChildのスーパークラスです。

Child child = new Child();
Parent parent = child;

子クラスのオブジェクトは、親クラス型の変数に代入できます。これをアップキャストと呼びます。

Javaのクラスは、直接継承できるクラスが1つだけです。

class Child extends Parent1, Parent2 { // コンパイルエラー
}

一方、インターフェースは複数実装できます。

class Child extends Parent implements Runnable, AutoCloseable {
    @Override
    public void run() {
    }

    @Override
    public void close() {
    }
}

明示的にextendsを書かなかった通常クラスは、直接または間接的にjava.lang.Objectを継承します。finalクラスは継承できません。Java 17では、sealedクラスによって直接継承できるクラスを限定することもできます。

2. 継承されるもの・されないもの

基本表

要素 継承されるか
publicフィールド 継承される
protectedフィールド 継承される
同一パッケージのパッケージアクセスフィールド 継承される
privateフィールド 継承されない
publicインスタンスメソッド 継承される
protectedインスタンスメソッド 継承される
同一パッケージのパッケージアクセスメソッド 継承される
privateメソッド 継承されない
利用可能なstaticメソッド クラスからは継承され得るが、オーバーライドではない
コンストラクタ 継承されない
インスタンス初期化子 継承されない
static初期化子 継承されない

コンストラクタ、インスタンス初期化子、static初期化子はクラスの「メンバー」ではないため、継承されません。privateメンバーも子クラスには継承されません。

3. アクセス修飾子と継承

アクセス範囲

修飾子 同じクラス 同じパッケージ 別パッケージの子クラス その他
public ○ ○ ○ ○
protected ○ ○ 条件付き○ ×
修飾子なし ○ ○ × ×
private ○ × × ×

3.1 private

class Parent {
    private int value = 10;

    private void print() {
        System.out.println(value);
    }
}

class Child extends Parent {
    void test() {
        // System.out.println(value); // コンパイルエラー
        // print();                   // コンパイルエラー
    }
}

privateメンバーは親クラスのオブジェクト内部には存在しますが、子クラスへ継承されず、子クラスから直接アクセスできません。

必要なら親クラスがアクセサメソッドを提供します。

class Parent {
    private int value = 10;

    protected int getValue() {
        return value;
    }
}

3.2 修飾子なしのパッケージアクセス

package p1;

public class Parent {
    int value = 10;
}

package p2;

import p1.Parent;

public class Child extends Parent {
    void test() {
        // System.out.println(value); // コンパイルエラー
    }
}

別パッケージの子クラスには、パッケージアクセスのメンバーは継承されません。

3.3 protectedの注意点

protectedは単純な「子クラスならどのオブジェクト経由でもアクセスできる」という意味ではありません。

別パッケージの子クラスでは、基本的に次の対象を通してアクセスします。

  • this
  • super
  • 自分のクラス型
  • 自分のサブクラス型

次はコンパイルエラーです。

package p1;

public class Parent {
    protected int value = 10;
}

package p2;

import p1.Parent;

public class Child extends Parent {
    void test(Parent parent, Child child) {
        System.out.println(this.value);  // OK
        System.out.println(super.value); // OK
        System.out.println(child.value); // OK

        // System.out.println(parent.value);
        // コンパイルエラー
    }
}

parentの実体がChildだったとしても、式のコンパイル時型がParentなのでアクセスできません。

別パッケージでのprotectedアクセスは、アクセスを行う子クラス型またはそのサブクラス型を通す必要があります。protectedコンストラクタも、別パッケージから通常のnew Parent()で自由に呼べるわけではありません。

4. インスタンスメソッドのオーバーライド

4.1 基本形

class Parent {
    void execute() {
        System.out.println("Parent");
    }
}

class Child extends Parent {
    @Override
    void execute() {
        System.out.println("Child");
    }
}

Parent obj = new Child();
obj.execute();

出力は次のとおりです。

Child

参照変数の型はParentですが、実体はChildです。インスタンスメソッドは実行時のオブジェクト型によって選ばれます。

これを動的ディスパッチ、動的メソッド探索、ポリモーフィズムなどと呼びます。

4.2 オーバーライドの成立条件

一般的には次を満たす必要があります。

  1. 親子関係がある
  2. インスタンスメソッドである
  3. メソッド名が同じ
  4. 引数の型と並びが同じ
  5. 戻り値型が同じか、共変戻り値型である
  6. アクセス範囲を狭くしていない
  7. チェック例外を広げていない
  8. 親メソッドがfinalではない
  9. 親メソッドがprivateではない

実際の仕様では、ジェネリクスとの互換性のため「同じシグネチャ」だけでなく「サブシグネチャ」という規則が使われます。

4.3 @Overrideを付ける

class Parent {
    void execute(int value) {
    }
}

class Child extends Parent {
    @Override
    void execute(Integer value) { // コンパイルエラー
    }
}

intとIntegerは別の引数型です。自動ボクシングがあっても、オーバーライド判定では同じシグネチャになりません。

@Overrideを付けなければ、単なるオーバーロードとしてコンパイルできる場合があります。

class Child extends Parent {
    void execute(Integer value) {
    }
}

そのため、オーバーライドするつもりのメソッドには原則として@Overrideを付けます。条件を満たさなければコンパイラが検出します。

5. 戻り値型のルール

5.1 プリミティブ型

プリミティブ型の戻り値は同一でなければなりません。

class Parent {
    int getValue() {
        return 10;
    }
}

class Child extends Parent {
    @Override
    long getValue() { // コンパイルエラー
        return 10L;
    }
}

intからlongへ暗黙変換できても、オーバーライドの戻り値型としては認められません。

5.2 参照型と共変戻り値

参照型では、親メソッドの戻り値型のサブタイプを返せます。

class Animal {
}

class Dog extends Animal {
}

class Parent {
    Animal create() {
        return new Animal();
    }
}

class Child extends Parent {
    @Override
    Dog create() {
        return new Dog();
    }
}

DogはAnimalのサブタイプなので有効です。

逆はできません。

class Parent {
    Dog create() {
        return new Dog();
    }
}

class Child extends Parent {
    @Override
    Animal create() { // コンパイルエラー
        return new Animal();
    }
}

6. アクセス修飾子を狭くできない

オーバーライド後のメソッドは、親メソッドと同じか、より広いアクセス範囲にします。

class Parent {
    protected void execute() {
    }
}

class Child extends Parent {
    @Override
    public void execute() { // OK
    }
}

逆はコンパイルエラーです。

class Parent {
    public void execute() {
    }
}

class Child extends Parent {
    @Override
    protected void execute() { // コンパイルエラー
    }
}

概ね次の方向だけが許可されます。

パッケージアクセス → protected → public

ただし、パッケージアクセスメソッドはパッケージ境界によってそもそも継承・オーバーライド関係が成立しないことがあります。

7. 例外とオーバーライド

7.1 チェック例外

子クラスのメソッドは、親メソッドより広いチェック例外を宣言できません。

import java.io.IOException;

class Parent {
    void execute() throws IOException {
    }
}

class Child extends Parent {
    @Override
    void execute() throws Exception { // コンパイルエラー
    }
}

ExceptionはIOExceptionより広いためです。

次は有効です。

import java.io.FileNotFoundException;
import java.io.IOException;

class Parent {
    void execute() throws IOException {
    }
}

class Child extends Parent {
    @Override
    void execute() throws FileNotFoundException {
    }
}

次も有効です。

class Child extends Parent {
    @Override
    void execute() {
    }
}

整理すると、子メソッドは次を選べます。

子メソッドのthrows 可否
親と同じチェック例外 ○
親例外のサブクラス ○
チェック例外なし ○
親例外のスーパークラス ×
親にない別系統のチェック例外 ×

7.2 非チェック例外

RuntimeException系は、親メソッドに宣言がなくても子メソッドで宣言できます。

class Parent {
    void execute() {
    }
}

class Child extends Parent {
    @Override
    void execute() throws IllegalStateException {
    }
}

非チェック例外はコンパイル時の例外検査の対象外です。オーバーライド側は、親メソッドが許可していないチェック例外を追加できません。

8. privateメソッドはオーバーライドされない

class Parent {
    private void execute() {
        System.out.println("Parent");
    }
}

class Child extends Parent {
    void execute() {
        System.out.println("Child");
    }
}

この2つはオーバーライド関係ではありません。

Parent.execute()はChildに継承されていないため、Child.execute()は新しい別メソッドです。

class Child extends Parent {
    @Override
    void execute() { // コンパイルエラー
    }
}

@Overrideを付けると、オーバーライド対象が存在しないためコンパイルエラーになります。

9. finalメソッド

class Parent {
    final void execute() {
    }
}

class Child extends Parent {
    @Override
    void execute() { // コンパイルエラー
    }
}

finalメソッドはオーバーライドできません。

privateメソッドも実質的にはオーバーライドされませんが、理由が異なります。

  • final：継承されるが、変更禁止
  • private：そもそも継承されない

finalメソッドをオーバーライドまたは隠蔽しようとすると、コンパイルエラーになります。

10. staticメソッドはオーバーライドされない

staticメソッドはオブジェクトではなくクラスに所属します。

class Parent {
    static void print() {
        System.out.println("Parent");
    }
}

class Child extends Parent {
    static void print() {
        System.out.println("Child");
    }
}

これはオーバーライドではなく、隠蔽です。

Parent obj = new Child();
obj.print();

出力は次です。

Parent

参照変数objのコンパイル時型がParentだからです。

Child child = new Child();
child.print();

こちらは次を出力します。

Child

推奨される書き方は型名を使う方法です。

Parent.print();
Child.print();

インスタンス経由でも呼べますが、動的ディスパッチされるように見えて誤解を招きます。

staticと非staticは相互に置き換えられない

class Parent {
    static void execute() {
    }
}

class Child extends Parent {
    void execute() { // コンパイルエラー
    }
}

逆もコンパイルエラーです。

class Parent {
    void execute() {
    }
}

class Child extends Parent {
    static void execute() { // コンパイルエラー
    }
}

同じシグネチャの継承関係にあるメソッドを、片方だけstaticにすることはできません。

staticメソッドは参照変数の型によって選ばれ、インスタンスメソッドは実行時のオブジェクト型によって選ばれます。

11. フィールドはオーバーライドされない

class Parent {
    int value = 10;
}

class Child extends Parent {
    int value = 20;
}

これはフィールドのオーバーライドではなく隠蔽です。

Parent p = new Child();
Child c = new Child();

System.out.println(p.value);
System.out.println(c.value);

出力は次です。

10
20

フィールドは参照変数のコンパイル時型で決まります。

class Child extends Parent {
    int value = 20;

    void print() {
        System.out.println(value);       // 20
        System.out.println(this.value);  // 20
        System.out.println(super.value); // 10
    }
}

親子の同名フィールドは別々に存在します。

フィールドの型が違っていてもよい

class Parent {
    int value = 10;
}

class Child extends Parent {
    String value = "Java";
}

これは合法です。

Child child = new Child();

System.out.println(child.value);            // Java
System.out.println(((Parent) child).value); // 10

同名フィールドを別の型で宣言できるため、設計上は極力避けるべきです。

12. オーバーロードとオーバーライド

オーバーロード

同じ名前で引数リストが異なるメソッドです。

class Example {
    void execute(int value) {
    }

    void execute(String value) {
    }
}

どのメソッドを呼ぶかは、原則としてコンパイル時の引数型で決まります。

オーバーライド

親クラスのインスタンスメソッドを子クラスで再定義します。

class Parent {
    void execute(int value) {
    }
}

class Child extends Parent {
    @Override
    void execute(int value) {
    }
}

最終的にどの実装を実行するかは、実行時のオブジェクト型で決まります。

両方が混在する例

class Parent {
    void execute(Object value) {
        System.out.println("Parent Object");
    }
}

class Child extends Parent {
    @Override
    void execute(Object value) {
        System.out.println("Child Object");
    }

    void execute(String value) {
        System.out.println("Child String");
    }
}

Parent p = new Child();
p.execute("Java");

出力は次です。

Child Object

処理は2段階です。

  1. コンパイル時、pの型はParentなのでexecute(Object)が選ばれる
  2. 実行時、Childのexecute(Object)へ動的ディスパッチされる

Child.execute(String)は、変数pの型からは見えないため候補になりません。オーバーロード選択はコンパイル時、オーバーライド実装の選択は実行時です。

13. super

親メソッドを呼ぶ

class Parent {
    void execute() {
        System.out.println("Parent");
    }
}

class Child extends Parent {
    @Override
    void execute() {
        super.execute();
        System.out.println("Child");
    }
}

出力は次です。

Parent
Child

super.execute()は、直接の親クラス側にある実装を基準に探索します。

親フィールドを参照する

class Parent {
    int value = 10;
}

class Child extends Parent {
    int value = 20;

    void print() {
        System.out.println(super.value);
    }
}

superはstaticコンテキストで使えない

class Child extends Parent {
    static void test() {
        // super.execute(); // コンパイルエラー
    }
}

superは現在のオブジェクトを前提とするため、staticメソッドやstatic初期化子では使えません。

抽象メソッドをsuperで呼べない

abstract class Parent {
    abstract void execute();
}

class Child extends Parent {
    @Override
    void execute() {
        // super.execute(); // コンパイルエラー
    }
}

親側に呼び出せる実装本体が存在しないためです。

14. 抽象クラス

14.1 基本

abstract class Animal {
    abstract void sound();

    void sleep() {
        System.out.println("sleep");
    }
}

抽象クラスは直接インスタンス化できません。

Animal animal = new Animal(); // コンパイルエラー

ただし、抽象クラス型の変数は使用できます。

class Dog extends Animal {
    @Override
    void sound() {
        System.out.println("bow");
    }
}

Animal animal = new Dog();

14.2 抽象クラスには通常メソッドも書ける

abstract class Animal {
    private String name;

    Animal(String name) {
        this.name = name;
    }

    public String getName() {
        return name;
    }

    abstract void sound();
}

抽象クラスには次を宣言できます。

  • フィールド
  • コンストラクタ
  • 通常のインスタンスメソッド
  • 抽象メソッド
  • staticメソッド
  • finalメソッド
  • 初期化子

「抽象クラスは抽象メソッドしか持てない」は誤りです。

14.3 抽象クラスにもコンストラクタがある

abstract class Parent {
    Parent() {
        System.out.println("Parent constructor");
    }
}

class Child extends Parent {
    Child() {
        System.out.println("Child constructor");
    }
}

new Child();

出力は次です。

Parent constructor
Child constructor

抽象クラス自体は直接生成できませんが、子クラスの生成過程で抽象クラスのコンストラクタが実行されます。

14.4 具象クラスは抽象メソッドを実装する

abstract class Parent {
    abstract void execute();
}

class Child extends Parent {
}

Childが抽象クラスでないならコンパイルエラーです。

次のどちらかが必要です。

class Child extends Parent {
    @Override
    void execute() {
    }
}

または、

abstract class Child extends Parent {
}

抽象クラスは未実装の抽象メソッドを次の子クラスへ持ち越せます。

14.5 通常メソッドを抽象化し直せる

class Parent {
    public void execute() {
        System.out.println("Parent");
    }
}

abstract class Middle extends Parent {
    @Override
    public abstract void execute();
}

これは合法です。

Middleは親の具体的実装を抽象メソッドで上書きし、Middleの具象サブクラスに再実装を要求しています。

ただし、親メソッドがfinalならできません。

14.6 抽象メソッドに付けられない修飾子

次の組合せは基本的に矛盾します。

abstract final void execute();        // エラー
abstract private void execute();      // エラー
abstract static void execute();       // エラー
abstract synchronized void execute(); // エラー
abstract native void execute();       // エラー

理由は次のとおりです。

  • abstract：子クラスで実装する
  • final：子クラスで変更禁止
  • private：子クラスに継承されない
  • static：インスタンスメソッドとしてオーバーライドされない
  • synchronized、native：実装本体を前提とする

14.7 abstract final classは不可

abstract final class Example { // コンパイルエラー
}

abstractはサブクラスによる完成を前提とし、finalはサブクラスを禁止するため矛盾します。

15. インターフェースの継承

interface Movable {
    void move();
}

上のメソッドは暗黙的に次と同じです。

interface Movable {
    public abstract void move();
}

実装クラスではpublicが必要です。

class Car implements Movable {
    @Override
    public void move() {
    }
}

次はコンパイルエラーです。

class Car implements Movable {
    @Override
    void move() { // publicより狭い
    }
}

インターフェースの抽象メソッドはpublicなので、実装メソッドもpublicでなければなりません。

インターフェースのフィールド

interface Config {
    int MAX = 100;
}

これは暗黙的に次と同じです。

interface Config {
    public static final int MAX = 100;
}

そのため再代入できません。

Config.MAX = 200; // コンパイルエラー

16. defaultメソッド

16.1 基本

interface Printable {
    default void print() {
        System.out.println("default");
    }
}

実装クラスは、オーバーライドしなくても利用できます。

class Document implements Printable {
}

new Document().print();

出力：

default

defaultは、インターフェースにインスタンスメソッドの標準実装を持たせる仕組みです。

16.2 実装クラスでオーバーライド

class Document implements Printable {
    @Override
    public void print() {
        System.out.println("Document");
    }
}

インターフェースメソッドなので、オーバーライド側はpublicです。

17. defaultメソッドの優先順位

基本的な判断順は次です。

  1. クラス側の具体的インスタンスメソッド
  2. より具体的なサブインターフェースのdefault
  3. 優先順位を決められなければ明示的にオーバーライド

17.1 クラス側の具体的メソッドが優先される

class Parent {
    public void execute() {
        System.out.println("Parent");
    }
}

interface Worker {
    default void execute() {
        System.out.println("Worker");
    }
}

class Child extends Parent implements Worker {
}

new Child().execute();

出力：

Parent

クラス階層から継承した具体的メソッドが、インターフェースのdefaultより優先されます。

17.2 より具体的なインターフェースが優先される

interface A {
    default void execute() {
        System.out.println("A");
    }
}

interface B extends A {
    @Override
    default void execute() {
        System.out.println("B");
    }
}

class C implements A, B {
}

new C().execute();

出力：

B

BはAのサブインターフェースなので、より具体的なBが優先されます。

17.3 無関係な2つのdefaultは衝突する

interface A {
    default void execute() {
        System.out.println("A");
    }
}

interface B {
    default void execute() {
        System.out.println("B");
    }
}

class C implements A, B {
}

これはコンパイルエラーです。

Cで明示的に解決します。

class C implements A, B {
    @Override
    public void execute() {
        A.super.execute();
    }
}

両方を呼ぶこともできます。

class C implements A, B {
    @Override
    public void execute() {
        A.super.execute();
        B.super.execute();
    }
}

同じシグネチャの無関係なdefaultメソッドを複数継承すると、コンパイラが自動選択できないため明示的な実装が必要です。

17.4 InterfaceName.superの制限

原則として、直接のスーパーインターフェースを指定します。

interface A {
    default void execute() {
    }
}

interface B extends A {
}

class C implements B {
    @Override
    public void execute() {
        // A.super.execute(); // コンパイルエラー
        B.super.execute();    // Bがdefaultを継承していれば利用可能
    }
}

継承階層を飛び越えて任意の祖先インターフェースを指定する機能ではありません。

17.5 Objectメソッドをdefaultにできない

次はコンパイルエラーです。

interface Example {
    default String toString() {
        return "Example";
    }
}

toString()、equals(Object)、hashCode()など、Objectの非privateインスタンスメソッドと同等のシグネチャを持つdefaultメソッドは宣言できません。実装クラスが持つObject由来のメソッドとの競合を避けるためです。

抽象メソッドとして宣言することはできます。

interface Example {
    @Override
    String toString();
}

18. インターフェースのstaticとprivate

staticメソッド

interface Utility {
    static void execute() {
        System.out.println("Utility");
    }
}

型名で呼びます。

Utility.execute();

実装クラスには継承されません。

class Impl implements Utility {
}

Impl.execute(); // コンパイルエラー

privateメソッド

interface Service {
    default void start() {
        log();
    }

    private void log() {
        System.out.println("log");
    }
}

privateインターフェースメソッドは、同じインターフェース内のdefaultメソッドや他のprivateメソッドから共通処理として利用します。実装クラスには継承されません。

クラスは、スーパーインターフェースのprivateまたはstaticメソッドを継承しません。

19. コンストラクタ

19.1 コンストラクタは継承されない

class Parent {
    Parent(int value) {
    }
}

class Child extends Parent {
}

Child(int)が自動的に作られることはありません。

new Child(10); // コンパイルエラー

コンストラクタはメンバーではなく、継承、オーバーライド、隠蔽の対象になりません。

19.2 コンストラクタ名

コンストラクタ名はクラス名と同じで、戻り値型を書きません。

class Sample {
    Sample() {
    }
}

次はコンストラクタではなくメソッドです。

class Sample {
    void Sample() {
    }
}

この場合、戻り値型voidがあるため通常メソッドです。

19.3 デフォルトコンストラクタ

クラスにコンストラクタを1つも宣言しなかった場合だけ、コンパイラがデフォルトコンストラクタを補います。

class Sample {
}

概念的には次に相当します。

class Sample {
    Sample() {
        super();
    }
}

ただし、アクセス修飾子はクラスのアクセス性に対応します。

19.4 1つでも宣言すると自動生成されない

class Sample {
    Sample(int value) {
    }
}

この場合、次はコンパイルエラーです。

new Sample();

必要なら自分で宣言します。

class Sample {
    Sample() {
    }

    Sample(int value) {
    }
}

20. super(...)とthis(...)

Java 17では、コンストラクタから別のコンストラクタを呼ぶ明示的コンストラクタ呼出しは、コンストラクタ本体の先頭に置きます。

親コンストラクタ

class Parent {
    Parent(int value) {
    }
}

class Child extends Parent {
    Child() {
        super(10);
    }
}

同じクラスの別コンストラクタ

class Sample {
    Sample() {
        this(10);
    }

    Sample(int value) {
        System.out.println(value);
    }
}

this(...)とsuper(...)を両方書けない

class Child extends Parent {
    Child() {
        this(10);
        super(20); // コンパイルエラー
    }

    Child(int value) {
        super(value);
    }
}

1つのコンストラクタが直接呼べるのは、同じクラスの別コンストラクタか、直接の親コンストラクタのどちらかです。

ただし、this(...)で呼ばれた先のコンストラクタは、最終的にsuper(...)を呼びます。

再帰的なコンストラクタ呼出しは不可

class Sample {
    Sample() {
        this(10);
    }

    Sample(int value) {
        this();
    }
}

これはコンストラクタ呼出しが循環するためコンパイルエラーです。

21. 暗黙のsuper()

コンストラクタの先頭にthis(...)もsuper(...)も書かなかった場合、コンパイラは暗黙的にsuper()を挿入します。

class Parent {
    Parent() {
    }
}

class Child extends Parent {
    Child() {
        // 暗黙的にsuper();
    }
}

しかし、親に引数なしコンストラクタがなければエラーです。

class Parent {
    Parent(int value) {
    }
}

class Child extends Parent {
    Child() {
        // 暗黙のsuper()がParent()を探す
        // Parent()は存在しないのでコンパイルエラー
    }
}

修正：

class Child extends Parent {
    Child() {
        super(10);
    }
}

親クラスにアクセス可能な引数なしコンストラクタがない場合、子コンストラクタから適切なsuper(...)を明示する必要があります。

22. 親コンストラクタがprivate

class Parent {
    private Parent() {
    }
}

class Child extends Parent { // コンパイルエラー
}

Childには暗黙のデフォルトコンストラクタが作られ、その中でsuper()を呼ぼうとします。しかしParent()がprivateなのでアクセスできません。

すべての親コンストラクタがprivateであれば、通常は外部クラスから継承できません。

ただし、同じトップレベルクラス内部のネストしたクラスなど、privateへアクセス可能な特殊な配置では成立することがあります。

23. コンストラクタとチェック例外

import java.io.IOException;

class Parent {
    Parent() throws IOException {
    }
}

子クラスのコンストラクタは、親コンストラクタが投げるチェック例外を処理する必要があります。

class Child extends Parent {
    Child() throws IOException {
        super();
    }
}

Java 17ではsuper()より前に通常のtry文を置けないため、親コンストラクタのチェック例外を同じ子コンストラクタ内で囲んで処理することはできません。

class Child extends Parent {
    Child() {
        try {
            // super(); // ここには書けない
        } catch (IOException e) {
        }
    }
}

通常は次のいずれかになります。

  • 子コンストラクタでもthrowsする
  • 例外を投げない別の親コンストラクタを選ぶ
  • 親クラス側の設計を変更する

24. 初期化順序

次のコードを考えます。

class Parent {
    static int parentStatic = print("1. Parent static field");

    static {
        print("2. Parent static block");
    }

    int parentField = print("5. Parent instance field");

    {
        print("6. Parent instance block");
    }

    Parent() {
        print("7. Parent constructor");
    }

    static int print(String message) {
        System.out.println(message);
        return 0;
    }
}

class Child extends Parent {
    static int childStatic = print("3. Child static field");

    static {
        print("4. Child static block");
    }

    int childField = print("8. Child instance field");

    {
        print("9. Child instance block");
    }

    Child() {
        print("10. Child constructor");
    }
}

初めて次を実行したとします。

new Child();

おおまかな順序は次です。

  1. 親クラスのstaticフィールド初期化
  2. 親クラスのstatic初期化子
  3. 子クラスのstaticフィールド初期化
  4. 子クラスのstatic初期化子
  5. オブジェクト領域をデフォルト値で初期化
  6. 親クラスのインスタンスフィールド初期化
  7. 親クラスのインスタンス初期化子
  8. 親クラスのコンストラクタ本体
  9. 子クラスのインスタンスフィールド初期化
  10. 子クラスのインスタンス初期化子
  11. 子クラスのコンストラクタ本体

同じクラス内のフィールド初期化式と初期化子は、基本的にソースコード上の記述順です。

クラス初期化では、子クラスより先に親クラスが初期化され、staticフィールド初期化式とstatic初期化子が記述順に実行されます。

25. コンストラクタからオーバーライド可能メソッドを呼ぶ危険

class Parent {
    Parent() {
        print();
    }

    void print() {
        System.out.println("Parent");
    }
}

class Child extends Parent {
    private String message = "Child";

    @Override
    void print() {
        System.out.println(message);
    }
}

new Child();

出力は次です。

null

理由は次のとおりです。

  1. Childオブジェクトの領域が確保され、messageはデフォルト値null
  2. Parentコンストラクタが実行される
  3. print()は動的ディスパッチによりChild.print()を呼ぶ
  4. しかしChildのフィールド初期化はまだ終わっていない
  5. messageはnull

したがって、コンストラクタから次のメソッドを呼ぶのは危険です。

  • オーバーライド可能なインスタンスメソッド
  • 抽象メソッド
  • 子クラスの初期化済み状態を前提とするメソッド

対策は次です。

final void print() {
}

または、

private void print() {
}

あるいは、コンストラクタ中では外部からオーバーライドされない処理だけを呼びます。

26. ポリモーフィズムと参照可能なメンバー

class Parent {
    void parentMethod() {
    }

    void commonMethod() {
        System.out.println("Parent");
    }
}

class Child extends Parent {
    void childMethod() {
    }

    @Override
    void commonMethod() {
        System.out.println("Child");
    }
}

Parent obj = new Child();

呼べるメソッドは、まず変数のコンパイル時型Parentで判定されます。

obj.parentMethod(); // OK
obj.commonMethod(); // OK。実行されるのはChild版
// obj.childMethod(); // コンパイルエラー

objの実体がChildでも、Parent型に宣言されていないchildMethod()は直接呼べません。

27. ダウンキャスト

Parent parent = new Child();
Child child = (Child) parent;

これは成功します。

しかし、次は実行時例外です。

Parent parent = new Parent();
Child child = (Child) parent;

コンパイルはできますが、実行時にClassCastExceptionが発生します。

安全に判定するにはinstanceofを使います。

if (parent instanceof Child child) {
    child.childMethod();
}

一方、型関係が完全に成立しないキャストはコンパイルエラーになることがあります。

class A {
}

final class B {
}

A a = new A();
// B b = (B) a; // 型関係によってはコンパイル時に不可能と判定

キャストはオブジェクト自体を別の型へ変換する処理ではありません。参照をどの型として扱うかを指定する処理です。

28. finalクラス

final class Parent {
}

class Child extends Parent { // コンパイルエラー
}

finalクラスは継承できません。

代表的な用途は次です。

  • クラスの不変条件を子クラスに破壊させたくない
  • セキュリティ上、動作の差し替えを禁止したい
  • 値オブジェクトとして振る舞いを固定したい
  • 継承ではなく合成を使わせたい

レコードクラスは暗黙的にfinalなので、通常のクラスから継承できません。

record Point(int x, int y) {
}

// class ChildPoint extends Point { } // コンパイルエラー

29. sealedクラス

Java 17では、継承可能な直接のサブクラスを限定できます。

sealed class Shape
        permits Circle, Rectangle {
}

final class Circle extends Shape {
}

non-sealed class Rectangle extends Shape {
}

Shapeを直接継承できるのはpermitsで指定されたクラスだけです。

直接のサブクラスは原則として次のいずれかを宣言します。

  • final
  • sealed
  • non-sealed
sealed class Shape permits Circle {
}

class Circle extends Shape { // コンパイルエラー
}

修正：

final class Circle extends Shape {
}

non-sealedを指定すると、その位置から通常の自由な継承へ戻ります。sealedクラスが許可していないクラスによる直接継承はコンパイルエラーです。

30. 同名メソッドを複数インターフェースから継承する場合

両方とも抽象メソッド

interface A {
    void execute();
}

interface B {
    void execute();
}

class C implements A, B {
    @Override
    public void execute() {
    }
}

1つの実装で両方の契約を満たせます。

戻り値に互換性がある

interface A {
    Number getValue();
}

interface B {
    Integer getValue();
}

class C implements A, B {
    @Override
    public Integer getValue() {
        return 10;
    }
}

IntegerはNumberのサブタイプなので、両方を満たせます。

戻り値に互換性がない

interface A {
    String getValue();
}

interface B {
    Integer getValue();
}

class C implements A, B { // コンパイルエラー
}

引数が同じで戻り値だけが互換性のないメソッドを、1つのクラスで両立できません。

戻り値型だけではオーバーロードできないため、次も不可です。

class C implements A, B {
    public String getValue() {
        return "";
    }

    public Integer getValue() { // コンパイルエラー
        return 0;
    }
}

31. フィールド名の競合

interface A {
    int VALUE = 10;
}

interface B {
    int VALUE = 20;
}

class C implements A, B {
    void print() {
        // System.out.println(VALUE); // コンパイルエラー
        System.out.println(A.VALUE);
        System.out.println(B.VALUE);
    }
}

同名フィールドを複数継承すること自体は直ちにエラーではありません。しかし、単純名VALUEで参照すると曖昧なのでコンパイルエラーです。

型名で限定すれば参照できます。複数の同名フィールドを継承した場合、単純名での曖昧な参照がコンパイルエラーになります。

32. ジェネリクスと継承の注意点

32.1 ジェネリック型は基本的に不変

List<Integer> integers = new ArrayList<>();
// List<Number> numbers = integers; // コンパイルエラー

IntegerがNumberのサブクラスでも、List<Integer>はList<Number>のサブタイプではありません。

必要に応じてワイルドカードを使います。

List<? extends Number> numbers = integers;

32.2 型消去による名前衝突

class Parent {
    void execute(List<String> values) {
    }
}

class Child extends Parent {
    void execute(List<Integer> values) {
    }
}

型消去後はどちらも概ねexecute(List)となるため、名前衝突としてコンパイルエラーになります。

ジェネリック型引数だけを変えて別メソッドとして共存させることはできません。

34. 実務上の設計指針

継承を使うべき場合

継承は、子クラスが親クラスとして完全に扱える場合に向いています。

Child is-a Parent

例：

Dog is an Animal

ただし、「コードを再利用したいだけ」で継承すると、親クラスの実装変更に強く依存します。

合成を優先する場合

class Engine {
    void start() {
    }
}

class Car {
    private final Engine engine;

    Car(Engine engine) {
        this.engine = engine;
    }

    void start() {
        engine.start();
    }
}

これはCar extends Engineではなく、Car has-an Engineです。

合成には次の利点があります。

  • 実装を交換しやすい
  • 親クラスの内部実装に依存しにくい
  • 継承階層が深くならない
  • テストしやすい
  • 不要なメソッドまで公開しにくい

コード再利用だけが目的なら、継承より合成を優先する方が保守しやすい場合が多いです。

オーバーライド時の推奨事項

@Override
public Result execute(Input input) throws SpecificException {
    return super.execute(input);
}

確認する項目は次です。

  1. @Overrideを付けたか
  2. アクセス範囲を狭めていないか
  3. 戻り値型に互換性があるか
  4. チェック例外を広げていないか
  5. 親クラスの契約を壊していないか
  6. 親実装を呼ぶ必要があるか
  7. コンストラクタから呼ばれる可能性がないか
  8. equals、hashCode、toStringの整合性が保たれているか

35. 最終的な判定手順

継承問題では次の順番で確認すると整理できます。

メソッドの場合

  1. staticかインスタンスか
  2. 親からアクセス可能か
  3. privateではないか
  4. finalではないか
  5. メソッド名と引数型が一致するか
  6. 戻り値型に互換性があるか
  7. アクセス範囲を狭めていないか
  8. チェック例外を広げていないか
  9. オーバーロード選択はコンパイル時型で判定
  10. インスタンスメソッドの実装は実行時型で判定

フィールドの場合

  1. フィールドはオーバーライドされない
  2. 同名なら隠蔽
  3. 参照変数のコンパイル時型で決まる
  4. 必要ならthis、super、型名で区別する

コンストラクタの場合

  1. コンストラクタは継承されない
  2. 自分で1つも宣言しなければデフォルトコンストラクタが作られる
  3. 自分で1つでも宣言すれば自動生成されない
  4. Java 17では先頭でthis(...)またはsuper(...)
  5. 省略時はsuper()
  6. 親にアクセス可能な対応コンストラクタが必要
  7. this(...)の連鎖は最終的にsuper(...)へ到達する
  8. 親から子の順に初期化される

この判定手順を使えば、継承に関するコンパイル可否、呼び出されるメソッド、出力順序の大部分を機械的に判断できます。
