ラムダ式と関数型インタフェース

ラムダ式は「短い書き方」ではなく、関数型インタフェースの抽象メソッドを実装する式です。
Goldでは、見た目ではなく、代入先の型、抽象メソッドの引数、戻り値、例外、キャプチャされる変数を順番に読む必要があります。

最初に見るべき判断軸

ラムダ式は単独では型を持ちません。必ず代入先、メソッド引数、戻り値、キャストなどの「ターゲット型」から型が決まります。
ラムダ式本体より先にターゲット型を判定します。

判定順見る場所判定すること
1代入先・引数・戻り値ターゲット型が関数型インタフェースか。
2抽象メソッド引数の数・型・戻り値・throws。
3ラムダの仮引数型省略、明示型、var、括弧のルール。
4ラムダ本体式形式かブロック形式か。値を返す必要があるか。
5外側の変数ローカル変数がfinalまたはeffectively finalか。
6例外ラムダ内のchecked例外が抽象メソッドのthrowsに合うか。

ラムダ式はターゲット型で決まる

x -> x + 1だけでは、xが何型で、戻り値が何型かは決まりません。Function<Integer, Integer>に代入されるなら、抽象メソッドはInteger apply(Integer)です。IntUnaryOperatorに代入されるなら、抽象メソッドはint applyAsInt(int)です。

import java.util.function.*;

Function<Integer, Integer> f = x -> x + 1;
IntUnaryOperator op = x -> x + 1;
Predicate<String> p = s -> s.length() > 0;
Consumer<String> c = s -> System.out.println(s);
ラムダ式本体より先に、ターゲット型となる関数型インタフェースの抽象メソッドを確認します。

関数型インタフェースとは

関数型インタフェースは、抽象メソッドを1つだけ持つインタフェースです。@FunctionalInterfaceは必須ではありませんが、付けると抽象メソッドが2つ以上になったときにコンパイルエラーで検出できます。

@FunctionalInterface
interface Converter {
    int convert(String s);
}

Converter c = s -> Integer.parseInt(s);
System.out.println(c.convert("10")); // 10
メンバ抽象メソッド数に数えるか理由
通常の抽象メソッド数える実装対象になる。
defaultメソッド数えない実装を持つため。
staticメソッド数えないインスタンスの抽象動作ではないため。
privateメソッド数えないJava 9以降、interface内の補助実装として使える。
Object由来のtoString、equalsなど原則数えないObjectのpublicメソッドと対応するため。
 
@FunctionalInterface
interface Checker {
    boolean test(String s);
    default Checker negate() { return s -> !test(s); }
    static Checker nonEmpty() { return s -> !s.isEmpty(); }
    String toString(); // Object由来のpublicメソッドなので抽象メソッド数には数えない
}

@FunctionalInterfaceで検出できる不正

抽象メソッドが2つあると、関数型インタフェースではありません。@FunctionalInterfaceを付けるとその時点でコンパイルエラーになります。付けなくても、ラムダ式のターゲット型として使うとコンパイルエラーです。

// @FunctionalInterface
interface Bad {
    void a();
    void b();
}

// Bad x = () -> System.out.println("x"); // コンパイルエラー

標準関数型インタフェース

Goldでは、java.util.functionの代表的な型を、名前ではなく抽象メソッドの形で覚えます。特にPredicate、Function、Consumer、Supplier、UnaryOperator、BinaryOperatorはよく使うです。

型抽象メソッド意味典型用途
Predicate<T>boolean test(T t)Tを受け取り真偽値filter
Function<T,R>R apply(T t)TをRへ変換map
Consumer<T>void accept(T t)Tを受け取って処理forEach
Supplier<T>T get()引数なしでTを返すorElseGet
UnaryOperator<T>T apply(T t)Tを受け取りTを返す同じ型の変換
BinaryOperator<T>T apply(T a, T b)Tを2つ受け取りTを返すreduce
 
import java.util.function.*;

Predicate<String> nonEmpty = s -> !s.isEmpty();
Function<String, Integer> length = s -> s.length();
Consumer<String> print = s -> System.out.println(s);
Supplier<String> supply = () -> "java";
UnaryOperator<String> upper = s -> s.toUpperCase();
BinaryOperator<Integer> add = (a, b) -> a + b;

Bi系・プリミティブ特化型

BiPredicate、BiFunction、BiConsumerは引数を2つ取ります。また、IntPredicateやIntFunctionなどのプリミティブ特化型は、ボクシングを避けるために用意されています。抽象メソッド名が通常版と違うものがあるので注意します。

型抽象メソッド注意点
BiPredicate<T,U>boolean test(T t, U u)2引数の条件。
BiFunction<T,U,R>R apply(T t, U u)2引数からRを返す。
BiConsumer<T,U>void accept(T t, U u)2引数を受け取り戻り値なし。
IntPredicateboolean test(int value)int専用。
IntFunction<R>R apply(int value)intをRへ変換。
IntUnaryOperatorint applyAsInt(int operand)intを受け取りintを返す。
ToIntFunction<T>int applyAsInt(T value)Tからintへ変換。
 
BiFunction<String, String, Integer> sumLength =
    (a, b) -> a.length() + b.length();

IntPredicate even = n -> n % 2 == 0;
ToIntFunction<String> length = s -> s.length();
IntUnaryOperator twice = n -> n * 2;

ラムダ仮引数の書き方

仮引数の型は、すべて省略するか、すべて明示するか、すべてvarにします。混在はできません。引数が1つで型を省略する場合だけ、丸括弧を省略できます。

書き方可否理由
s -> s.length()可1引数、型省略なので括弧省略可能。
(s) -> s.length()可括弧ありも可能。
(String s) -> s.length()可型を明示。
(var s) -> s.length()可Java 11以降、ラムダ仮引数にvarを使用可能。
String s -> s.length()不可型を明示するなら括弧が必要。
(String a, b) -> a + b不可明示型と省略型の混在。
(var a, String b) -> a + b不可varと明示型の混在。
(var a, b) -> a + b不可varと省略型の混在。
 
Function<String, Integer> a = s -> s.length();
Function<String, Integer> b = (String s) -> s.length();
Function<String, Integer> c = (var s) -> s.length();

BiFunction<Integer, Integer, Integer> d = (x, y) -> x + y;
BiFunction<Integer, Integer, Integer> e = (Integer x, Integer y) -> x + y;
BiFunction<Integer, Integer, Integer> f = (var x, var y) -> x + y;

式形式とブロック形式

ラムダ本体が式形式なら、その式の結果が戻り値になります。ブロック形式で値を返すならreturnが必要です。戻り値がvoidの抽象メソッドでは、式文として成立する呼び出しを式形式で書けます。

Function<String, Integer> a = s -> s.length();
Function<String, Integer> b = s -> { return s.length(); };
// Function<String, Integer> c = s -> { s.length(); }; // 戻り値がないため不可

Consumer<String> d = s -> System.out.println(s);
Consumer<String> e = s -> { System.out.println(s); };
// Consumer<String> f = s -> { return System.out.println(s); }; // printlnはvoid

ブロック形式では、すべての実行経路で戻り値が必要です。これは通常のメソッドと同じです。

// Function<Integer, String> f = n -> {
//     if (n > 0) {
//         return "plus";
//     }
// }; // n <= 0 のとき戻り値がないのでコンパイルエラー

void互換と値互換

ラムダ本体の式は、戻り値を使う文脈にも、戻り値を捨てる文脈にも合う場合があります。たとえばlist.add(x)はbooleanを返しますが、メソッド呼び出し式として文にもできるため、PredicateにもConsumerにも見えることがあります。オーバーロードでは曖昧になる可能性があります。

import java.util.*;
import java.util.function.*;

List<String> list = new ArrayList<>();
Predicate<String> p = s -> list.add(s); // booleanを返す
Consumer<String> c = s -> list.add(s);  // 戻り値を捨てる文として扱う

オーバーロードとラムダの曖昧さ

ラムダ式はターゲット型が必要です。
そのため、オーバーロードされたメソッドにラムダ式を渡すと、どの関数型インタフェースを使うか決められずコンパイルエラーになる場合があります。
キャストでターゲット型を明示すると解消できます。

import java.util.function.*;

static void call(Consumer<String> c) {}
static void call(Function<String, String> f) {}

// call(s -> s.trim()); // 曖昧でコンパイルエラー
call((Function<String, String>) (s -> s.trim()));

一方、戻り値の有無や引数の数で明確に区別できる場合は、オーバーロード解決できます。

static void run(Runnable r) {}
static void run(Supplier<String> s) {}

run(() -> System.out.println("A")); // Runnable
run(() -> "A");                    // Supplier<String>

外側のローカル変数はeffectively finalのみ使える

ラムダ式の外側にあるローカル変数は、finalまたはeffectively finalでなければラムダ内から参照できません。effectively finalとは、finalを書いていなくても、初期化後に値を変更していない状態です。

int base = 10;
Function<Integer, Integer> f = x -> x + base;
System.out.println(f.apply(5)); // 15

// base++; // これを書くと、上のラムダでbaseを使えなくなる

参照型変数の場合、変数が参照しているオブジェクトの中身を変更することと、変数自体へ別の参照を代入することは別です。
変数自体を再代入しなければ、effectively finalは維持されます。

List<String> list = new ArrayList<>();
Consumer<String> c = s -> list.add(s);
c.accept("A");
System.out.println(list); // [A]

// list = new ArrayList<>(); // これを書くとラムダからlistを参照できない
悪い判断

「listの中身を変更したからeffectively finalではない」と判断する。

良い判断

effectively finalはローカル変数への再代入の話。オブジェクトの内部状態の変更とは別。

ラムダ仮引数は外側のローカル変数名と衝突できない

ラムダ仮引数は、同じスコープにあるローカル変数名と同じ名前にできません。フィールド名とは衝突しても、ローカル変数とは扱いが違います。

String s = "outer";
// Function<String, String> f = s -> s.toUpperCase(); // ローカル変数sと衝突して不可
Function<String, String> g = x -> x.toUpperCase();

ラムダ内のthisは外側のインスタンス

ラムダ式は匿名クラスとは違い、新しいthisを作りません。ラムダ内のthisは、ラムダ式を書いた外側のインスタンスを指します。匿名クラス内のthisは匿名クラス自身です。

class Sample {
    private String name = "outer";

    Runnable lambda() {
        return () -> System.out.println(this.name);
    }

    Runnable anonymous() {
        return new Runnable() {
            private String name = "inner";
            public void run() {
                System.out.println(this.name);
            }
        };
    }
}

checked例外とラムダ式

ラムダ内でchecked例外を投げる場合、ターゲット型の抽象メソッドがその例外を投げられる宣言を持っている必要があります。Runnable.run()はchecked例外を投げられないため、ラムダ内でIOExceptionをそのまま投げることはできません。

import java.io.*;

@FunctionalInterface
interface IOAction {
    void run() throws IOException;
}

IOAction a = () -> { throw new IOException(); };
// Runnable r = () -> { throw new IOException(); }; // コンパイルエラー

メソッド参照の基本

メソッド参照はラムダ式を短く書ける構文ですが、名前が一致するだけでは成立しません。ターゲット型の抽象メソッドについて、引数の受け渡しと戻り値の互換性を確認します。

形式例対応する考え方
staticメソッド参照Integer::parseInts -> Integer.parseInt(s)
特定インスタンスのメソッド参照System.out::printlnx -> System.out.println(x)
任意インスタンスのメソッド参照String::lengths -> s.length()
コンストラクタ参照ArrayList::new() -> new ArrayList<>()
 
Function<String, Integer> parse = Integer::parseInt;
Consumer<String> print = System.out::println;
Function<String, Integer> length = String::length;
Supplier<ArrayList<String>> supplier = ArrayList::new;

任意インスタンスのメソッド参照は第1引数がレシーバになる

String::compareToIgnoreCaseのような参照では、抽象メソッドの第1引数がメソッド呼び出し対象になります。つまり(a, b) -> a.compareToIgnoreCase(b)です。

BiFunction<String, String, Integer> cmp = String::compareToIgnoreCase;
System.out.println(cmp.apply("a", "A")); // 0

同じString::valueOfでも、ターゲット型によって選ばれるオーバーロードが変わります。

Function<Integer, String> a = String::valueOf;
Function<char[], String> b = String::valueOf;

コンストラクタ参照

コンストラクタ参照もターゲット型から引数の数と型が決まります。Supplierなら引数なしコンストラクタ、Functionなら1引数コンストラクタ、BiFunctionなら2引数コンストラクタに対応します。

class Item {
    private final String name;
    private final int price;
    Item() { this("none", 0); }
    Item(String name) { this(name, 0); }
    Item(String name, int price) { this.name = name; this.price = price; }
    public String toString() { return name + ":" + price; }
}

Supplier<Item> s = Item::new;
Function<String, Item> f = Item::new;
BiFunction<String, Integer, Item> b = Item::new;

配列コンストラクタ参照

配列の生成もコンストラクタ参照で表せます。String[]::newは、長さを受け取って配列を返すIntFunction<String[]>として使えます。

IntFunction<String[]> maker = String[]::new;
String[] array = maker.apply(3);
System.out.println(array.length); // 3

Predicateの合成

Predicateにはand、or、negateがあります。これらは新しいPredicateを返します。元のPredicateを書き換えるわけではありません。

Predicate<String> nonEmpty = s -> !s.isEmpty();
Predicate<String> startsWithA = s -> s.startsWith("A");

Predicate<String> rule = nonEmpty.and(startsWithA);
System.out.println(rule.test("Apple")); // true
System.out.println(rule.test("Java"));  // false

Functionの合成

FunctionにはcomposeとandThenがあります。順番が逆なので、出力を追うコードでは大まかに読まないこと。f.andThen(g)はfの後にg、f.compose(g)はgの後にfです。

Function<String, String> trim = s -> s.trim();
Function<String, Integer> length = s -> s.length();

System.out.println(trim.andThen(length).apply(" java ")); // 4
System.out.println(length.compose(trim).apply(" java ")); // 4

Consumerの合成

Consumer.andThenは処理を順番に実行するConsumerを返します。戻り値はありません。途中で例外が発生すると、それ以降のConsumerは実行されません。

Consumer<String> a = s -> System.out.print("A" + s);
Consumer<String> b = s -> System.out.print("B" + s);

a.andThen(b).accept("X"); // AXBX

Comparatorとラムダ式

Comparator<T>も関数型インタフェースです。抽象メソッドはint compare(T a, T b)です。comparing、thenComparing、reversedなどは、StreamやCollectionsの並び替えコードでよく使うです。

record User(String name, int age) {}

List<User> users = new ArrayList<>(List.of(
    new User("B", 20),
    new User("A", 30),
    new User("A", 10)
));

users.sort(Comparator
    .comparing(User::name)
    .thenComparingInt(User::age));

System.out.println(users); // [User[name=A, age=10], User[name=A, age=30], User[name=B, age=20]]

匿名クラスとの違い

ラムダ式は関数型インタフェースにしか使えません。匿名クラスは、抽象メソッドが複数ある抽象クラスやインタフェースにも使えます。また、匿名クラスは新しいクラスを作るため、thisの意味やスコープがラムダ式と違います。

比較ラムダ式匿名クラス
対象関数型インタフェースのみインタフェース・抽象クラス・具象クラスの拡張など
this外側のインスタンス匿名クラス自身
抽象メソッドが複数不可実装すれば可能
可読性短い処理に向く状態や複数メソッドの実装に向く

Stream APIでの読み方

Streamのfilter、map、forEachは、それぞれ要求する関数型インタフェースが違います。どのメソッドが何を要求しているかを見れば、ラムダ式の戻り値も決まります。

Stream操作受け取る型ラムダの形
filterPredicate<? super T>Tを受け取りbooleanを返す。
mapFunction<? super T, ? extends R>Tを受け取りRを返す。
flatMapFunction<? super T, ? extends Stream<? extends R>>Tを受け取りStreamを返す。
forEachConsumer<? super T>Tを受け取り戻り値なし。
reduceBinaryOperator<T>など2つの値をまとめる。
 
List<String> list = List.of("a", "bb", "ccc");

list.stream()
    .filter(s -> s.length() > 1)
    .map(s -> s.toUpperCase())
    .forEach(s -> System.out.println(s));

不可能・誤りパターン

次のパターンは、ラムダ式・関数型インタフェースのコードでよく判断を誤りやすくなります。

コード結果理由
Object o = x -> x;不可Objectは関数型インタフェースではない。
var f = x -> x + 1;不可ラムダ式だけでは型推論できない。
Function<String,Integer> f = s -> { s.length(); };不可ブロック形式で戻り値がない。
Consumer<String> c = s -> { return s; };不可Consumer.acceptはvoid。
(String a, b) -> a + b不可明示型と型省略の混在。
(var a, String b) -> a + b不可varと明示型の混在。
ラムダ内で再代入された外側ローカル変数を参照不可effectively finalではない。
Runnable r = () -> { throw new IOException(); };不可Runnable.runはchecked例外を宣言していない。

読解ミニ演習1：ターゲット型

次のコードはどれがコンパイルできますか。

import java.util.function.*;

Predicate<String> a = s -> s.isEmpty();
Function<String, Boolean> b = s -> s.isEmpty();
Consumer<String> c = s -> s.isEmpty();
// Supplier<String> d = s -> s.toUpperCase();

aはboolean test(String)に合います。bはBoolean apply(String)に合います。cは戻り値を捨てる式文として扱えるためコンパイルできます。dはSupplier.get()が引数なしなので、引数sを取るラムダ式とは合いません。

読解ミニ演習2：effectively final

次のコードはコンパイルできません。理由を判定します。

int count = 0;
Runnable r = () -> System.out.println(count);
// count++; // この再代入により、countはeffectively finalではなくなる

ラムダ式が外側のローカル変数を使う場合、その変数はfinalまたはeffectively finalである必要があります。
参照後に変更しても、変更後に参照しても、同じく不可です。

コードを読む順番

ラムダ式は、文法だけでなく型推論、オーバーロード、例外、スコープが関係します。次の順で確認します。

  1. 代入先またはメソッド引数の型を見る。
  2. その型が関数型インタフェースか判定する。
  3. 抽象メソッドの引数リスト、戻り値、throwsを判定する。
  4. ラムダ仮引数の数・型・括弧・varの使い方を判定する。
  5. 本体が式形式かブロック形式かを判定する。
  6. 戻り値が必要な経路で必ず返しているか判定する。
  7. 外側のローカル変数を使っているならeffectively finalか判定する。
  8. メソッド参照なら、第1引数がレシーバになる形式かどうか判定する。
  9. checked例外が抽象メソッドのthrowsで許可されているか判定する。
ラムダ式は、ターゲット型の抽象メソッドへ戻して読むと判定できます。仮引数型、戻り値、checked例外の互換性を一つずつ確認してください。

ラムダ式の確認

ラムダ式は、単体では型が決まりません。先に代入先や引数の型から、対象の関数型インタフェースを決めます。

  • 抽象メソッドが1つだけかを見ます。
  • 引数の数・型・戻り値型が合うかを見ます。
  • 外側のローカル変数を使う場合は、実質的finalかを見ます。
  • メソッド参照は、見た目ではなく対象の関数型インタフェースに合わせて判定します。
