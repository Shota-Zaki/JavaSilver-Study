sealedとpermits

sealed class / interfaceはJava 17で正式導入されました。Java 11では利用できません。sealedは、継承または実装できる相手を制限する仕組みです。許可する側と許可される側の両方を見ます。

広告スポンサーリンク表示エリア

sealedの基本

sealed を付けたクラスやインタフェースは、permits に書いた型だけに継承・実装を許可します。

public sealed class Animal permits Dog, Cat {}

public final class Dog extends Animal {}
public non-sealed class Cat extends Animal {}
箇所見る内容理由
親クラスsealed と permits誰に継承を許可しているかを決める。
子クラスfinal / sealed / non-sealedsealedの直接サブクラスには継承方針の明示が必要。
名前permitsに含まれるか含まれない型は継承できない。

直接サブクラスの修飾子

sealedクラスを直接継承したクラスは、次のどれかで修飾します。

修飾子意味読み方
finalここで継承を止めるさらにサブクラスを作れない。
sealedさらに継承先を制限するこのクラスも継承先を制限する。直接サブタイプが同じコンパイル単位にない場合はpermitsを明示する。
non-sealed以降の継承制限を解除する通常の継承に戻る。
sealed class A permits B, C, D {}
final class B extends A {}
sealed class C extends A permits E {}
non-sealed class D extends A {}
final class E extends C {}

同じコンパイル単位ならpermitsを省略できる場合

すべての直接サブクラスがsealed型と同じコンパイル単位で宣言されている場合、permits を省略できるケースがあります。ただし明示されている permits とサブクラス側の修飾子を優先して判定します。

sealed class A {}
final class B extends A {}
non-sealed class C extends A {}

この場合も、直接サブクラス側には final、sealed、non-sealed のいずれかが必要です。

理解の確認

  • permits に書かれていないクラスは直接継承できない。
  • 直接サブクラスには final / sealed / non-sealed が必要。
  • non-sealed は、sealed階層の直接サブクラスで使う。
  • sealed interfaceでも、実装クラス側の修飾子の考え方は同じ。
