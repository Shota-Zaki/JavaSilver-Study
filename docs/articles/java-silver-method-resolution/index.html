オーバーロードの呼び出し解決

Silver向けの記事です。同名メソッドのどれを呼ぶかは、実行時の実体ではなく、コンパイル時に見える引数の型と変換規則で決まります。

最初に候補を集め、適用できる中で最も具体的なものを選ぶ

class OverloadSample {
    static void test(int value)     { System.out.print("int"); }
    static void test(long value)    { System.out.print("long"); }
    static void test(Integer value) { System.out.print("Integer"); }
    static void test(int... value)  { System.out.print("varargs"); }

    public static void main(String[] args) {
        test(1); // int
    }
}

整数リテラル1の型はintです。完全一致するtest(int)があるため、拡大変換、ボクシング、可変長引数の候補へ進みません。

代表的な優先関係

優先変換例
1完全一致int → int
2プリミティブの拡大変換int → long
3ボクシングまたはアンボクシングを含む候補int → Integer
4可変長引数int → int...

これは典型を扱うコード用の判断順です。複数引数やジェネリックメソッドでは追加の規則が関わるため、各候補が適用可能かを引数ごとに確認します。

戻り値型だけではオーバーロードできない

class ReturnTypeOnly {
    int read() { return 1; }
    // String read() { return "A"; } // コンパイルエラー
}

メソッドシグネチャの中心はメソッド名と引数型です。呼び出し式read()だけでは戻り値によって候補を区別できません。

nullは参照型候補のうち、より具体的な型を選ぶ

class NullResolution {
    static void test(Object value) { System.out.print("Object"); }
    static void test(String value) { System.out.print("String"); }

    public static void main(String[] args) {
        test(null); // String
    }
}

StringはObjectより具体的なのでtest(String)が選ばれます。互いに親子関係がない型が残ると曖昧です。

static void test(String value) {}
static void test(StringBuilder value) {}
// test(null); // コンパイルエラー: どちらがより具体的か決められない

オーバーロードは宣言型、オーバーライドは実体型

class Parent {
    void show(Object value) { System.out.print("P-Object"); }
}
class Child extends Parent {
    @Override void show(Object value) { System.out.print("C-Object"); }
    void show(String value) { System.out.print("C-String"); }
}
public class DispatchOrder {
    public static void main(String[] args) {
        Parent ref = new Child();
        ref.show("A"); // C-Object
    }
}
  1. コンパイル時、変数refの宣言型はParentなので、候補はshow(Object)だけです。
  2. 実行時、そのシグネチャをChildがオーバーライドしているため、Child.show(Object)が実行されます。
  3. Child.show(String)は、コンパイル時の候補に入りません。

可変長引数は配列としても呼べる

static void sum(int... values) {
    System.out.println(values.length);
}

sum();               // 0
sum(1, 2);           // 2
sum(new int[]{1, 2}); // 2

可変長引数はメソッド内では配列です。同じ位置に通常の配列引数と可変長引数を別々に定義すると、シグネチャが同じになり共存できません。

コードを読む順番

  1. 呼び出し元から見えるメソッドだけを候補にする。
  2. 実引数のコンパイル時の型を確定する。
  3. 引数ごとに適用可能な変換を確認する。
  4. 最も具体的な候補を一つに決める。決まらなければコンパイルエラー。
  5. 選ばれたシグネチャがオーバーライドされている場合だけ、実体型の実装へ進む。

記事末尾の確認

  • 戻り値型だけの違いをオーバーロードと判断しない。
  • nullで残る候補の型関係を比較できる。
  • 可変長引数を最初の候補として選ばない。
  • オーバーロード選択とオーバーライド実行を二段階で追える。
