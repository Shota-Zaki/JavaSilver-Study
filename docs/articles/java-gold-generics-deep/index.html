最初に固定する結論

ジェネリクスは、主にコンパイル時の型安全性を高める仕組みです。実行時には型引数の多くは消去されます。そのため、List<String>とList<Integer>を実行時に完全に別物として扱えるわけではありません。

観点覚えること読み方
型引数List<String> の Stringコンパイル時の取り出し型・追加可能型を制限する
ワイルドカード?, ? extends T, ? super T引数として受け取れる範囲を広げる代わりに、操作が制限される
型消去実行時には型引数が基本的に残らないinstanceof List<String> などは不可
raw typeList のように型引数を書かない型コンパイルは通ることがあるが、警告や実行時例外の原因になる
Silverまでの感覚で「継承関係があるから代入できるはず」と読むと、ジェネリクスで崩れます。String extends Object でも、List<String> は List<Object> のサブタイプではありません。

基本形：型を固定すると、取り出し時のキャストが不要になる

ジェネリクスを使わない場合、コレクションから取り出す値は基本的にObjectとして扱われます。型を指定すると、コンパイラが「このListにはStringだけを入れる」という前提でチェックできます。

import java.util.*;

public class Main {
    public static void main(String[] args) {
        List<String> names = new ArrayList<>();
        names.add("Alice");
        names.add("Bob");

        String first = names.get(0);  // キャスト不要
        System.out.println(first.length());
    }
}

この時点で重要なのは、List<String>は「String専用のListクラスが新しく作られる」という意味ではないことです。コンパイル時に型チェックされ、必要ならキャストが挿入される、と読む方が正確です。

raw type：コンパイルは通るが、型安全性を捨てている

List のように型引数を書かないものをraw typeと呼びます。古いJavaとの互換性のために残っています。raw typeの使用は多くの場合、警告付きでコンパイルされますが、型安全性が失われるため実行時にClassCastExceptionが発生することがあります。

import java.util.*;

public class Main {
    public static void main(String[] args) {
        List<String> strings = new ArrayList<>();
        strings.add("A");

        List raw = strings;       // unchecked warning の原因
        raw.add(100);             // コンパイルは通ることがある

        for (String s : strings) {
            System.out.println(s.length()); // 100をStringとして取り出そうとして実行時例外
        }
    }
}

ここで大まかに「raw typeはコンパイルエラー」と覚えるのは誤りです。多くの場合は警告です。ただし型安全性が失われるため、取り出し時の暗黙キャストでClassCastExceptionが発生する可能性があります。

ジェネリクスの不変性：List<String>はList<Object>ではない

配列には共変性がありますが、ジェネリクスは基本的に不変です。ここを間違えると、ワイルドカードの必要性も理解できません。

import java.util.*;

public class Main {
    public static void main(String[] args) {
        List<String> strings = new ArrayList<>();

        // List<Object> objects = strings; // コンパイルエラー
        // objects.add(100);               // もし許されると、List<String>にIntegerが入ってしまう
    }
}

この代入が許されると、List<String>の中にIntegerを入れられてしまいます。これを防ぐために、List<String>はList<Object>として扱えません。

配列は共変：だから実行時例外が起きる

対比として配列を見ます。String[]はObject[]に代入できます。しかし、不正な値を入れようとすると実行時にArrayStoreExceptionが発生します。

public class Main {
    public static void main(String[] args) {
        String[] strings = {"A", "B"};
        Object[] objects = strings;

        objects[0] = 100; // 実行時例外 ArrayStoreException
    }
}

配列は実行時に要素型を知っています。一方、ジェネリクスは型消去されます。この違いが、配列とジェネリクスの挙動差です。

List<?>：何のListか不明

List<?>は「何かの型のList」です。任意の型のListを受け取れますが、要素型が不明なので、具体的な値は追加できません。安全に取り出せる型はObjectです。

import java.util.*;

public class Main {
    public static void main(String[] args) {
        List<String> names = List.of("A", "B");
        print(names);
    }

    static void print(List<?> list) {
        Object value = list.get(0); // OK
        // list.add("C");          // コンパイルエラー
        // list.add(10);           // コンパイルエラー
        list.add(null);            // コンパイル上はOK。ただしList.ofなら実行時にUnsupportedOperationException
        System.out.println(value);
    }
}

List<?>にnullは追加できます。これはnullがどの参照型にも代入可能だからです。ただし、List.of()で作ったListは変更不可なので、実行時にはUnsupportedOperationExceptionになります。コンパイル可否と実行時可否を分けて見ると分かりやすいです。

? extends T：読む側に強い

? extends Numberは「NumberまたはNumberのサブタイプの何か」です。中身はIntegerかもしれないし、Doubleかもしれません。取り出すときは少なくともNumberとして扱えますが、追加は基本的にできません。

import java.util.*;

public class Main {
    public static void main(String[] args) {
        List<Integer> ints = List.of(1, 2, 3);
        double total = sum(ints);
        System.out.println(total);
    }

    static double sum(List<? extends Number> nums) {
        double total = 0;
        for (Number n : nums) {
            total += n.doubleValue();
        }

        // nums.add(10);    // コンパイルエラー
        // nums.add(1.5);   // コンパイルエラー
        return total;
    }
}

追加できない理由は単純です。仮に実体がList<Double>だった場合、Integerを追加できたら壊れます。? extends Numberは「Numberとして読めます」が、「Numberを入れられる」ではありません。

? super T：書く側に強い

? super Integerは「IntegerまたはIntegerのスーパータイプの何か」です。実体はList<Integer>かもしれないし、List<Number>やList<Object>かもしれません。どの場合でもIntegerは追加できます。

import java.util.*;

public class Main {
    public static void main(String[] args) {
        List<Number> nums = new ArrayList<>();
        addNumbers(nums);
        System.out.println(nums);
    }

    static void addNumbers(List<? super Integer> list) {
        list.add(10);              // OK
        list.add(Integer.valueOf(20)); // OK

        Object value = list.get(0); // OK
        // Integer i = list.get(0); // コンパイルエラー
    }
}

取り出すときにIntegerとして扱えないのは、実体がList<Object>かもしれないからです。Objectとしてなら安全です。

PECS：Producer extends, Consumer super

PECSはワイルドカードの役割を整理するための覚え方です。記号だけを丸暗記せず、値を読み取る側か受け取る側かを確認します。値を取り出して使うだけならextends、値を受け取らせるならsuperです。

コード上の役割使う型できること制限
値を供給する側? extends TTとして読めます具体的な値を追加できない
値を受け取る側? super TTを追加できる読み取りは基本Object
読み書き両方List<T>Tとして読め、Tを追加できる受け取れる型の幅は狭い

copyメソッドで判断する

典型例は、srcから読み、destへ書くメソッドです。srcはproducerなのでextends、destはconsumerなのでsuperを使います。

import java.util.*;

public class Main {
    public static void main(String[] args) {
        List<Integer> src = List.of(1, 2, 3);
        List<Number> dest = new ArrayList<>();

        copy(src, dest);
        System.out.println(dest);
    }

    static <T> void copy(List<? extends T> src, List<? super T> dest) {
        for (T value : src) {
            dest.add(value);
        }
    }
}

この形はGoldでよく使うです。List<? extends T>は取り出し、List<? super T>は追加、と役割で読むべきです。

型パラメータ：class Box<T>

クラスに型パラメータを持たせると、インスタンスごとに扱う型を固定できます。

public class Box<T> {
    private T value;

    public Box(T value) {
        this.value = value;
    }

    public T get() {
        return value;
    }

    public void set(T value) {
        this.value = value;
    }
}

class Main {
    public static void main(String[] args) {
        Box<String> box = new Box<>("Java");
        String s = box.get();
        System.out.println(s.toUpperCase());
    }
}

Tは型名ではなく、型を受け取るための仮の名前です。Box<String>なら、そのインスタンスではTがStringとして扱われます。

ジェネリックメソッド：戻り値や引数だけに型変数を使う

クラス全体ではなく、メソッド単位で型パラメータを宣言できます。戻り値型の前に<T>を書く点がコードを読むときに違いを確認します。

public class Main {
    public static void main(String[] args) {
        String s = pick("A", "B");
        Integer i = pick(10, 20);
        System.out.println(s + i);
    }

    static <T> T pick(T a, T b) {
        return a;
    }
}

static T pick(...)のように、<T>なしでいきなりTを使うことはできません。メソッドの型パラメータは戻り値型の前に宣言します。

staticメンバでTを使えない

クラスの型パラメータは、インスタンスに紐づく型情報として考えます。staticメンバはインスタンスに属さないため、クラスの型パラメータTを直接使えません。

class Box<T> {
    private T value;       // OK

    // private static T cache; // コンパイルエラー

    static <U> U identity(U value) { // staticメソッド独自の型パラメータならOK
        return value;
    }
}

ただし、staticメソッドが独自に<U>を宣言することはできます。クラスのTと、メソッドのUは別物です。

上限境界：T extends Number

<T extends Number>と書くと、Tとして使える型をNumber以下に制限できます。これにより、doubleValue()などNumberのメソッドを呼べます。

public class Main {
    public static void main(String[] args) {
        System.out.println(toDouble(10));
        System.out.println(toDouble(3.5));
    }

    static <T extends Number> double toDouble(T value) {
        return value.doubleValue();
    }
}

ここでのextendsは、クラス継承だけでなくインタフェース実装にも使います。型パラメータの境界では、インタフェースでもextendsと書きます。

複数境界：クラスを先に書く

複数の境界を指定する場合は&でつなぎます。クラスとインタフェースを混ぜる場合、クラスは最初に書く必要があります。

class MyNumber extends Number implements Comparable<MyNumber> {
    private final int value;
    MyNumber(int value) { this.value = value; }
    public int compareTo(MyNumber other) { return Integer.compare(value, other.value); }
    public int intValue() { return value; }
    public long longValue() { return value; }
    public float floatValue() { return value; }
    public double doubleValue() { return value; }
}

class Util {
    static <T extends Number & Comparable<T>> T max(T a, T b) {
        return a.compareTo(b) >= 0 ? a : b;
    }

    // static <T extends Comparable<T> & Number> T bad(T a) { return a; }
    // クラス境界 Number をインタフェース境界の後ろに置くためコンパイルエラー
}

境界にクラスを複数指定することはできません。Javaはクラスの多重継承を許可しないためです。

Comparable<? super T> の意味

GoldではComparable<? super T>のような形が出ます。これは「T自身、またはTのスーパータイプを比較対象にできる型」を受け入れるための書き方です。

import java.util.*;

class Person implements Comparable<Person> {
    final String name;
    Person(String name) { this.name = name; }
    public int compareTo(Person other) { return name.compareTo(other.name); }
}

public class Main {
    public static void main(String[] args) {
        List<Person> people = List.of(new Person("B"), new Person("A"));
        Person min = min(people);
        System.out.println(min.name);
    }

    static <T extends Comparable<? super T>> T min(List<T> list) {
        T result = list.get(0);
        for (T value : list) {
            if (value.compareTo(result) < 0) {
                result = value;
            }
        }
        return result;
    }
}

最初から完全理解しようとしすぎる必要はありません。Comparable<T>よりもComparable<? super T>の方が受け入れ範囲が広い、と読めれば十分な場面が多いです。

型推論とダイヤモンド演算子

new ArrayList<>()の<>はダイヤモンド演算子です。左辺や文脈から型を推論します。

import java.util.*;

public class Main {
    public static void main(String[] args) {
        List<String> a = new ArrayList<>(); // ArrayList<String> と推論
        a.add("A");

        var b = new ArrayList<>();          // 文脈が薄いので ArrayList<Object> 寄り
        b.add("B");
        b.add(10);

        System.out.println(a);
        System.out.println(b);
    }
}

varは「何でもよい型」ではありません。右辺から具体的な型が推論されます。var b = new ArrayList<>();のように型情報が少ないと、意図より広い型になることがあります。

型消去：実行時にはList<String>かどうかを直接判定できない

ジェネリクスの型引数は、基本的にコンパイル後に消去されます。そのため、実行時にList<String>かどうかをinstanceofで直接判定できません。

import java.util.*;

public class Main {
    public static void main(String[] args) {
        Object obj = List.of("A", "B");

        if (obj instanceof List<?> list) {  // OK
            System.out.println(list.size());
        }

        // if (obj instanceof List<String> list) { } // コンパイルエラー
    }
}

List<?>はreifiableな形として扱えるためinstanceofに使えます。一方、List<String>は実行時に型引数を判定できないため使えません。

型消去でオーバーロードできない例

型引数だけが異なるメソッドは、型消去後に同じシグネチャになるため、同時に定義できません。

import java.util.*;

class Sample {
    // void test(List<String> list) { }
    // void test(List<Integer> list) { }
    // 型消去後はどちらも test(List) になるためコンパイルエラー
}

「引数のジェネリクス部分が違うからオーバーロードできる」ではありません。型消去後に同じ形になるかを見ます。

new T() はできない

型パラメータTは実行時に具体的な型として残るとは限りません。そのため、new T()はできません。

class Factory<T> {
    // T create() {
    //     return new T(); // コンパイルエラー
    // }
}

同じ理由で、T.classのような書き方もできません。必要ならClass<T>を外から受け取る設計にします。

class Factory<T> {
    private final Class<T> type;

    Factory(Class<T> type) {
        this.type = type;
    }

    Class<T> type() {
        return type;
    }
}

new T[] もできない

型パラメータの配列を直接作ることもできません。

class Store<T> {
    // private T[] values = new T[10]; // コンパイルエラー
}

配列は実行時に要素型を保持します。しかし、Tは型消去されるため、実行時にどの型の配列を作ればよいか確定できません。

List<String>[] は作れない

ジェネリック型の配列生成も代表的な不可パターンです。

import java.util.*;

class Main {
    public static void main(String[] args) {
        // List<String>[] array = new List<String>[10]; // コンパイルエラー

        List<?>[] ok = new List<?>[10]; // これは可能
        ok[0] = List.of("A");
        ok[1] = List.of(10);
    }
}

List<?>[]は作れますが、配列とジェネリクスを組み合わせると型安全性を確認しにくくなります。必要性がなければ、ジェネリックなコレクションをそのまま使う方が明確です。

プリミティブ型は型引数に使えない

型引数に使えるのは参照型です。List<int>は書けません。必要ならラッパークラスを使います。

import java.util.*;

public class Main {
    public static void main(String[] args) {
        // List<int> nums = new ArrayList<>(); // コンパイルエラー

        List<Integer> nums = new ArrayList<>();
        nums.add(10); // オートボクシング
        int n = nums.get(0); // アンボクシング
        System.out.println(n);
    }
}

ジェネリック例外クラスは作れない

Throwableのサブクラスをジェネリックにすることはできません。

// class MyException<T> extends Exception { }
// コンパイルエラー：generic class may not extend Throwable

例外処理は実行時の型判定とcatchの仕組みに強く関係します。型消去されるジェネリック型を例外型にすると、catch判定と整合しないため禁止されています。

ワイルドカードに値を戻すときの注意

List<? extends Number>から取り出した値はNumberとして受け取れます。しかしIntegerとしては受け取れません。

import java.util.*;

class Main {
    static void test(List<? extends Number> list) {
        Number n = list.get(0); // OK
        // Integer i = list.get(0); // コンパイルエラー
    }
}

実体がList<Integer>なら動きそうに見えますが、メソッドの中ではList<Double>かもしれません。宣言型で安全に言えることだけが許可されます。

superの読み取りはObjectまで

List<? super Integer>はIntegerを追加できますが、取り出すときはObjectです。

import java.util.*;

class Main {
    static void test(List<? super Integer> list) {
        list.add(1);              // OK
        Object o = list.get(0);   // OK
        // Number n = list.get(0); // コンパイルエラー
        // Integer i = list.get(0);// コンパイルエラー
    }
}

実体がList<Object>なら、中にStringが入っている可能性もあります。そのため、IntegerやNumberとして取り出すことは安全ではありません。

メソッド引数の選び方

メソッドを設計する側の視点で見ると、型の幅を広げるためにワイルドカードを使います。

やりたいこと推奨形理由
Listの中身を読むだけvoid print(List<? extends Number> list)List<Integer>やList<Double>を受け取れる
Integerを追加するvoid add(List<? super Integer> list)List<Integer>, List<Number>, List<Object>を受け取れる
String専用で読み書きするvoid edit(List<String> list)読み書き両方を強く型チェックできる
型を問わずサイズだけ見るvoid size(List<?> list)要素型を使わないため最も広い

ワイルドカードと継承の代入パターン

代入可否は、右辺の型が左辺の条件を満たすかで判定します。

import java.util.*;

class Main {
    public static void main(String[] args) {
        List<Integer> ints = new ArrayList<>();
        List<Number> nums = new ArrayList<>();
        List<Object> objs = new ArrayList<>();

        List<? extends Number> a;
        a = ints; // OK
        a = nums; // OK
        // a = objs; // コンパイルエラー

        List<? super Integer> b;
        b = ints; // OK
        b = nums; // OK
        b = objs; // OK
    }
}

? extends Numberは上限がNumberなので、List<Object>は範囲外です。? super IntegerはInteger以上の型なので、Integer, Number, Objectが入ります。

戻り値にワイルドカードを使うと、呼び出し側が扱いにくい

戻り値をList<? extends Number>にすると、呼び出し側は読み取りには使えますが、追加できません。

import java.util.*;

class Main {
    static List<? extends Number> create() {
        return List.of(1, 2, 3);
    }

    public static void main(String[] args) {
        List<? extends Number> list = create();
        Number n = list.get(0);
        // list.add(4); // コンパイルエラー
        System.out.println(n);
    }
}

API設計としては、戻り値にワイルドカードを多用すると使いにくくなります。コードを読むときは「コンパイルはできるが、呼び出し側で操作が制限される」と読めば十分です。

キャプチャ：List<?>の要素を同じListに戻せないことがある

List<?>から取り出した値はObjectとして見えます。しかし、そのObjectを同じListに戻すことはできません。

import java.util.*;

class Main {
    static void test(List<?> list) {
        Object value = list.get(0);
        // list.set(0, value); // コンパイルエラー
    }
}

人間から見ると「同じListから取り出した値を戻すだけ」に見えます。しかしコンパイラは、Objectを未知の要素型のListに安全に戻せるとは判断しません。

helperメソッドで型を捕まえる

ワイルドカードの型をメソッド型パラメータとして捕まえると、同じ型として扱えることがあります。

import java.util.*;

class Main {
    static void duplicateFirst(List<?> list) {
        duplicateFirstHelper(list);
    }

    private static <T> void duplicateFirstHelper(List<T> list) {
        T value = list.get(0);
        list.set(0, value); // OK
    }
}

これはGoldで深く確認する頻度は高くありませんが、?は「完全に何でもあり」ではなく「未知の具体型」と読むと理解しやすくなります。

ジェネリックvarargsとheap pollution

可変長引数は内部的に配列として扱われます。ジェネリック型と配列は相性が悪いため、警告やheap pollutionの話が出ます。

import java.util.*;

class Main {
    @SafeVarargs
    static void printAll(List<String>... lists) {
        for (List<String> list : lists) {
            for (String s : list) {
                System.out.println(s);
            }
        }
    }

    public static void main(String[] args) {
        printAll(List.of("A"), List.of("B"));
    }
}

@SafeVarargsは、ジェネリックな可変長引数に対してヒープ汚染につながる操作を行わないことを表明するアノテーションです。使える場所は制限されます。staticメソッド、finalメソッド、privateメソッド、コンストラクタなどが代表です。

不可能パターン一覧

コード結果理由
List<int>不可型引数は参照型のみ
new T()不可型消去により実行時に具体型がない
new T[10]不可配列は実行時要素型が必要
new List<String>[10]不可ジェネリック型の配列生成は不可
obj instanceof List<String>不可実行時に型引数を判定できない
List<String>.class不可型引数付きclassリテラルは使えない
static T value不可クラス型パラメータはstatic文脈で使えない
class E<T> extends Exception不可ジェネリックなThrowableサブクラスは禁止
void m(List<String>) と void m(List<Integer>)不可型消去後に同じシグネチャ

ジェネリクスの読み分け

確認する内容判定理由
List<String>はList<Object>に代入できる誤りジェネリクスは不変
List<? extends Number>にはIntegerを追加できる誤り実体がList<Double>かもしれない
List<? super Integer>からIntegerとして取り出せる誤り安全に読めるのはObject
List<?>には任意の型を追加できる誤り追加できるのは基本的にnullだけ
raw typeを使うと必ずコンパイルエラー誤り多くは警告。実行時例外の原因になる
obj instanceof List<?>は使える正しい非境界ワイルドカードは判定可能

ミニ演習1：これはコンパイルできるか

import java.util.*;

class Main {
    static void test(List<? extends Number> list) {
        Number n = list.get(0);
        list.add(null);
        // list.add(1);
    }
}

コメントを外さなければコンパイルできます。getの戻り値はNumberとして扱えます。nullの追加はコンパイル上はできます。ただしlist.add(1)は不可です。

ミニ演習2：これは何が出るか

import java.util.*;

public class Main {
    public static void main(String[] args) {
        List<String> list = new ArrayList<>();
        List raw = list;
        raw.add(10);
        String s = list.get(0);
        System.out.println(s);
    }
}

コンパイル時にはunchecked warningが出ますが、コンパイル自体は通る可能性があります。実行すると、IntegerをStringとして取り出すところでClassCastExceptionになります。

ミニ演習3：extendsとsuperの代入

import java.util.*;

class Main {
    public static void main(String[] args) {
        List<Integer> a = new ArrayList<>();
        List<Number> b = new ArrayList<>();
        List<Object> c = new ArrayList<>();

        List<? extends Number> x = a;
        // x = c;

        List<? super Integer> y = c;
        y.add(10);
    }
}

x = aはOKです。x = cはObjectがNumber以下ではないため不可です。y = cはOKで、Integerを追加できます。

コードを読むときにの解き方

  1. まず、型引数なしのraw typeかどうかを見る。rawなら「警告か、実行時例外か」を疑う。
  2. List<A>とList<B>の代入では、AとBの継承関係をそのまま使わない。ジェネリクスは不変と読みます。
  3. extendsなら「読み出せますが、追加できません」。取り出し型は上限。
  4. superなら「追加できますが、読み出し型は制限されます」。取り出し型は基本Object。
  5. instanceof、配列生成、オーバーロード、static文脈では型消去を疑う。
  6. コンパイル可でも、List.of()の変更不可やraw typeの混入で実行時例外が起きないかを見る。
ジェネリクスは、各変数について「何型として取り出せるか」と「何型を追加できるか」を分けて判定します。ワイルドカードの記号だけを暗記すると、読み取りと追加の可否を逆にしやすくなります。

Genericsの確認

Genericsは「入れ物の中身の型」を読む単元です。まず、代入先と代入元の型引数が一致するかを見ます。

List<String> は List<Object> のサブタイプではありません。StringがObjectのサブタイプでも、List同士の代入互換は別です。

  • ? extends T は、Tとして読み取る向きに強い指定です。
  • ? super T は、Tを追加する向きに強い指定です。
  • raw型はコンパイル警告の原因になり、実行時のClassCastExceptionにもつながります。
