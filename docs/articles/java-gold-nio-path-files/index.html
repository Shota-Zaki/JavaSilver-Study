PathとFilesの読み方

NIO.2では、Pathを「場所の表現」、Filesを「ファイルシステム上の実体への操作」として分けます。例外が発生するタイミングも、Pathの操作とFilesの操作で異なります。

まず分ける: Pathはパス、Filesは操作

Pathはファイルやディレクトリの場所を表すオブジェクトです。Pathを作っただけでは、ファイルやディレクトリの存在確認、作成、読み込みは行われません。存在判定やI/OはFilesの操作で発生します。

import java.nio.file.*;

public class Main {
  public static void main(String[] args) {
    Path p = Path.of("data", "sample.txt");
    System.out.println(p);              // data/sample.txt など
    System.out.println(p.getFileName()); // sample.txt
  }
}

上のコードは、data/sample.txt が実在しなくてもコンパイル・実行できます。存在判定や読み書きは Files の仕事です。

悪い判断

Path.of() が成功したので、ファイルが存在すると読みます。

良い判断

Path作成、存在判定、読み書き、削除を別の段階として判定する。

Path.of と Paths.get

Path.of はJava 11以降で使える書き方です。Paths.get も同じ目的で使えます。コードを読むときは両方出る可能性があります。

Path a = Path.of("data", "sample.txt");
Path b = Paths.get("data", "sample.txt");

System.out.println(a.getFileName()); // sample.txt
System.out.println(b.getParent());   // data

引数を複数に分けると、環境に応じた区切り文字でPathが構成されます。Windowsなら \、macOS/Linuxなら / が使われます。ただし、表示は環境依存です。コードを読むときは「完全に同じ文字列が出る」と大まかに決めつけない方が安全です。

Pathの基本メソッド

Pathの読解では、getRoot、getParent、getFileName、getNameCount、getName、subpath がよく違いを確認します。

Path p = Path.of("/home/user/work/sample.txt");

System.out.println(p.getRoot());      // / など
System.out.println(p.getParent());    // /home/user/work
System.out.println(p.getFileName());  // sample.txt
System.out.println(p.getNameCount()); // 4
System.out.println(p.getName(0));     // home
System.out.println(p.subpath(1, 3));  // user/work

getName(0) はルートではありません。ルート / は getRoot() で扱います。getRoot()と名前要素を分けないと、getNameCount()やgetName(index)の結果を一つ多く数えてしまいます。

メソッド意味注意
getRoot()ルート部分相対パスなら null になりうる。
getParent()親パス親がなければ null。
getFileName()末尾の名前ファイルとは限らない。ディレクトリ名でもよい。
getNameCount()名前要素数ルートは数えない。
getName(i)i番目の名前要素0始まり。範囲外なら例外。
subpath(a, b)a以上b未満の部分パス終端indexは含まない。

絶対パスと相対パス

isAbsolute() は、そのPathが絶対パスかどうかを返します。toAbsolutePath() は現在の作業ディレクトリを基準に絶対パスを作りますが、これも実体の存在判定ではありません。

Path p = Path.of("data/sample.txt");

System.out.println(p.isAbsolute());       // false
System.out.println(p.toAbsolutePath());   // カレントディレクトリ基準の絶対パス

toAbsolutePath() と toRealPath() は別物です。toRealPath() は実ファイルシステムに問い合わせ、存在しなければ IOException になります。

normalize と toRealPath を混同しない

normalize() は、パス文字列上の . や .. を整理するだけです。ファイルの存在判定はしません。

Path p = Path.of("a", ".", "b", "..", "c");
System.out.println(p);             // a/./b/../c など
System.out.println(p.normalize()); // a/c

toRealPath() は実体にアクセスします。存在しないパスに対して呼ぶと IOException が発生します。

Path p = Path.of("missing.txt");
// p.toRealPath(); // 実体がなければ IOException
API実体判定例外
normalize()しない通常はファイル未存在で失敗しない。
toAbsolutePath()しない通常はファイル未存在で失敗しない。
toRealPath()するIOException の可能性。

resolve: パスを後ろにつなげる

resolve は、基準パスに別のパスをつなげます。

Path base = Path.of("data");
Path p = base.resolve("sample.txt");
System.out.println(p); // data/sample.txt など

注意点は、右側に絶対パスを渡した場合です。この場合、基準パスは無視され、右側の絶対パスが結果になります。

Path base = Path.of("data");
Path absolute = Path.of("/tmp/sample.txt");

System.out.println(base.resolve(absolute)); // /tmp/sample.txt など

resolveは単純な文字列連結ではなく、基準Pathに別のPathを解決する操作です。data/tmp/sample.txt のようにつながる、ではありません。

resolveSibling: 同じ親にある別名へ差し替える

resolveSibling は、末尾の名前を同じ階層の別名に差し替えるイメージです。

Path p = Path.of("data", "old.txt");
System.out.println(p.resolveSibling("new.txt")); // data/new.txt

ファイル名だけ変えたいコードでは resolve ではなく resolveSibling が自然です。

relativize: 片方から片方への相対パス

relativize は、基準Pathから対象Pathへ到達するための相対パスを作ります。

Path a = Path.of("work", "src");
Path b = Path.of("work", "out", "Main.class");

System.out.println(a.relativize(b)); // ../out/Main.class など

相対パスと絶対パスを混ぜて relativize すると、IllegalArgumentException になることがあります。

Path relative = Path.of("data", "a.txt");
Path absolute = Path.of("/tmp/a.txt");

// relative.relativize(absolute); // IllegalArgumentException になりうる

この例外は IOException ではありません。Pathの組み合わせが不適切なため、実ファイルの読み書き前に失敗するイメージです。

Files.exists と notExists の注意

Files.exists(path) は存在すれば true、存在しなければ通常 false です。ただし権限などで判断できない場合があります。Files.notExists(path) は単純に !exists(path) と同じではありません。

状態existsnotExists
存在するtruefalse
存在しないfalsetrue
判定不能false になりうるfalse になりうる

存在判定APIの戻り値はbooleanであり、読み込みAPIのように必ず例外処理が必要な形ではない、という差も判定の軸です。

Files.isDirectory / isRegularFile / isSameFile

属性判定系は、Pathが指す実体の種類を判定します。

Path p = Path.of("data");

System.out.println(Files.isDirectory(p));
System.out.println(Files.isRegularFile(p));
System.out.println(Files.isReadable(p));

Files.isSameFile(a, b) は、2つのPathが同じ実体を指すかを調べます。単なる文字列比較ではありません。実ファイルシステムに問い合わせるため、IOException が発生しえます。

Path a = Path.of("data", "sample.txt");
Path b = Path.of("data", ".", "sample.txt");

// 実体があれば true になりうる
// System.out.println(Files.isSameFile(a, b));

readString / readAllLines / lines

読み込みAPIは似ていますが、戻り値とリソース管理が違います。

API戻り値特徴
Files.readString(path)Stringファイル全体を文字列として読む。
Files.readAllLines(path)List<String>全行をListに読み込む。
Files.lines(path)Stream<String>遅延処理。Streamを閉じる意識が必要。
Path p = Path.of("data", "sample.txt");

String text = Files.readString(p);
List<String> lines = Files.readAllLines(p);

try (Stream<String> stream = Files.lines(p)) {
  stream.filter(s -> !s.isBlank())
        .forEach(System.out::println);
}

Files.lines はStreamを返します。Streamは再利用できません。さらに、ファイルを開いたStreamなので、try-with-resourcesで閉じる形が安全です。

Stream<String> s = Files.lines(Path.of("data.txt"));
s.count();
// s.count(); // IllegalStateException: stream has already been operated upon or closed

writeString / write / OpenOption

Files.writeString は文字列を書き込みます。標準では既存ファイルを上書きする挙動を取ります。追記したい場合は StandardOpenOption.APPEND を指定します。

Path p = Path.of("out", "memo.txt");

Files.createDirectories(p.getParent());
Files.writeString(p, "A
");
Files.writeString(p, "B
", StandardOpenOption.APPEND);
Option意味注意
CREATEなければ作るよく使う。
CREATE_NEW新規作成のみ既にあれば FileAlreadyExistsException。
APPEND追記先頭ではなく末尾に書く。
TRUNCATE_EXISTING既存内容を空にする上書き系の挙動。

createFile / createDirectory / createDirectories

作成系APIは、既存有無と親ディレクトリの有無が判定の軸です。

API親がない場合対象が既にある場合
Files.createFile(path)NoSuchFileException になりうるFileAlreadyExistsException
Files.createDirectory(path)NoSuchFileException になりうるFileAlreadyExistsException
Files.createDirectories(path)親もまとめて作る既にあっても通常成功
Files.createDirectory(Path.of("a"));        // aだけ作る。親が必要
Files.createDirectories(Path.of("a/b/c"));  // 親もまとめて作る

createDirectory と createDirectories の違いは実務でも確認が必要です。親も作るのは複数形の createDirectories です。

copy / move と上書き指定

Files.copy と Files.move は、コピー先・移動先が既に存在する場合に注意します。上書きしたいなら StandardCopyOption.REPLACE_EXISTING を指定します。

Path src = Path.of("data", "a.txt");
Path dst = Path.of("backup", "a.txt");

Files.createDirectories(dst.getParent());
Files.copy(src, dst, StandardCopyOption.REPLACE_EXISTING);
Files.move(dst, Path.of("backup", "b.txt"), StandardCopyOption.REPLACE_EXISTING);

ATOMIC_MOVE は、可能なら原子的に移動する指定です。環境やファイルシステムによりサポートされない場合があります。

Option用途注意
REPLACE_EXISTING既存先を置き換える指定しないと既存先で失敗しやすい。
COPY_ATTRIBUTES属性もコピーコピー系で使う。
ATOMIC_MOVE原子的移動サポートされない可能性。

delete と deleteIfExists

Files.delete は対象が存在しないと例外です。Files.deleteIfExists は、存在しなければ false を返します。

Path p = Path.of("out", "memo.txt");

Files.deleteIfExists(p); // なければfalse
// Files.delete(p);      // なければNoSuchFileException

ディレクトリ削除では、空でないディレクトリを delete しようとすると DirectoryNotEmptyException になります。delete は中身を再帰的に消してくれるAPIではありません。

list / walk / find

ディレクトリをStreamとして扱うAPIもGoldで判定の軸です。これらはStreamを返すため、try-with-resourcesで閉じるのが安全です。

API対象特徴
Files.list(dir)直下のみ再帰しない。
Files.walk(dir)再帰起点自身もStreamに含まれる。
Files.find(dir, depth, predicate)条件付き再帰属性も使って判定できる。
Path dir = Path.of("src");

try (Stream<Path> s = Files.walk(dir)) {
  s.filter(Files::isRegularFile)
   .forEach(System.out::println);
}

walk は起点ディレクトリ自身も含みます。ファイルだけを対象にしたい場合は Files.isRegularFile で絞ります。

walkで再帰削除するときの順序

ディレクトリを再帰削除する場合、親を先に除外すると中身が残っていて失敗します。深い要素から除外する必要があります。

Path dir = Path.of("work");

try (Stream<Path> s = Files.walk(dir)) {
  s.sorted(Comparator.reverseOrder())
   .forEach(path -> {
     try {
       Files.delete(path);
     } catch (IOException e) {
       throw new UncheckedIOException(e);
     }
   });
}

ラムダ式内ではchecked例外をそのまま投げられない場面があります。そのため、UncheckedIOException に包む例がよく使われます。

FileVisitor / walkFileTree

Files.walkFileTree は、ファイルツリーを歩くためのAPIです。SimpleFileVisitor を継承し、必要なメソッドだけオーバーライドする形が基本です。

Path start = Path.of("src");

Files.walkFileTree(start, new SimpleFileVisitor<Path>() {
  @Override
  public FileVisitResult visitFile(Path file, BasicFileAttributes attrs)
      throws IOException {
    System.out.println(file);
    return FileVisitResult.CONTINUE;
  }
});
戻り値意味
CONTINUE探索を続ける。
SKIP_SUBTREE現在のディレクトリ配下をスキップ。
SKIP_SIBLINGS兄弟要素をスキップ。
TERMINATE探索終了。

シンボリックリンクと NOFOLLOW_LINKS

Filesの属性判定や探索では、シンボリックリンクをたどるかどうかで結果が変わります。LinkOption.NOFOLLOW_LINKS を指定すると、リンク先ではなくリンク自体を対象にする方向になります。

Path p = Path.of("link.txt");

System.out.println(Files.exists(p));
System.out.println(Files.exists(p, LinkOption.NOFOLLOW_LINKS));

Goldで深く出る場合は、リンク先を見ているのか、リンクそのものを見ているのかを分けます。

IOExceptionとRuntimeExceptionを分ける

NIO.2の失敗は全部 IOException ではありません。ファイルシステムに触る操作は IOException が多い一方、Pathの形式不整合やStream再利用は実行時例外です。

状況例主な結果
存在しないファイルを読むFiles.readString(p)IOException 系
存在しないファイルをdeleteFiles.delete(p)NoSuchFileException
空でないディレクトリをdeleteFiles.delete(dir)DirectoryNotEmptyException
既存先にcopyFiles.copy(src, dst)FileAlreadyExistsException
相対Pathと絶対Pathをrelativizerelative.relativize(absolute)IllegalArgumentException
Stream再利用stream.count(); stream.count();IllegalStateException

コンパイルエラー・実行時例外になりやすいパターン

Path生成だけでIOExceptionをcatchする

try {
  Path p = Path.of("missing.txt");
} catch (IOException e) { // コンパイルエラー
}

Path.of は通常 IOException を投げません。到達不能catchとして扱われます。

Files.readStringのIOException処理漏れ

String s = Files.readString(Path.of("a.txt")); // コンパイルエラー

readString はchecked例外の IOException を投げうるため、catchまたはthrowsが必要です。

Files.linesのStreamを2回使う

Stream<String> s = Files.lines(Path.of("a.txt"));
s.count();
s.forEach(System.out::println); // 実行時例外

Streamは終端操作後に再利用できません。

deleteでディレクトリを丸ごと除外できると思う

Files.delete(Path.of("dir")); // dirが空でなければ失敗

中身のあるディレクトリはそのままでは削除できません。

copyの上書き指定忘れ

Files.copy(src, dst); // dstが既存なら失敗

上書きしたいなら StandardCopyOption.REPLACE_EXISTING が必要です。

コードの読み方

  1. Path操作かFiles操作かを見る。
    Pathだけなら存在判定は基本的にしない。Filesなら実体アクセスやIOExceptionを疑う。
  2. 絶対パス・相対パスを判定する。
    resolve、relativize、normalize の結果が変わる。
  3. 対象がファイルかディレクトリかを見る。
    delete、walk、list の挙動に影響する。
  4. 既存ファイルの扱いを見る。
    copy、move、createFile、createDirectory は既存有無で結果が変わる。
  5. Streamを返すAPIかを見る。
    Files.lines、Files.list、Files.walk はStream。遅延評価、再利用不可、closeを意識する。

読解ミニ演習

確認例1

Path p = Path.of("a", "b", "..", "c.txt");
System.out.println(p.normalize());

結果: a/c.txt など。b/.. が整理されます。ファイルが実在する必要はありません。

確認例2

Path p = Path.of("missing.txt");
System.out.println(p.getFileName());

結果: missing.txt。Path操作だけなので、ファイルが存在しなくても問題ありません。

確認例3

Path p = Path.of("missing.txt");
System.out.println(Files.readString(p));

結果: 例外処理がなければコンパイルエラーです。Files.readString は IOException を投げうるchecked例外APIです。

確認例4

Path a = Path.of("work", "src");
Path b = Path.of("work", "out", "Main.class");
System.out.println(a.relativize(b));

結果: ../out/Main.class など。work/src から一度上がって out/Main.class に向かいます。

確認例5

try (Stream<Path> s = Files.walk(Path.of("src"))) {
  System.out.println(s.count());
  System.out.println(s.count());
}

結果: 2回目の count() で実行時例外です。Streamは終端操作後に再利用できません。

まとめ

NIO.2は、メソッド名を覚えるだけでは不十分です。特に以下を機械的に判定します。

  • Path は場所の表現。実在判定ではない。
  • Files は実体操作。IOException 処理を疑う。
  • normalize は文字列的整理、toRealPath は実体判定。
  • resolve は絶対パスを渡すと右側優先。
  • relativize は相対・絶対の混在に注意。
  • Files.lines、list、walk はStream。再利用不可。
  • copy、move、createFile、delete は既存有無で結果が変わる。

この単元で曖昧にしてはいけないのは「Pathを作る」と「ファイルに触る」の境界です。ここが混ざっていると、コンパイルエラー・実行時例外・正常出力の判定を外します。

確認ポイントの補足

NIO.2コードでは、Pathの文字列操作とFilesの実ファイル操作を分けて読む必要があります。Path.of、resolve、relativize、normalizeは、基本的にパス表現を組み立てる処理です。実在判定や入出力はFiles側の処理です。

見る対象判定誤読例
Path.resolve右辺が絶対パスなら右辺が優先される常に左辺へ連結されると読んでしまう
Path.relativize同じ種類のパス同士で相対パスを作る絶対パスと相対パスの混在を見落とす
Files.linesStream<String>を返し、閉じる対象になる終端操作前に再利用できると読む
Files.copy / move既存ファイル、オプション、IOExceptionを読む戻り値や例外処理を飛ばして出力だけを見る

出力を追うコードでは、OS依存の区切り文字をそのまま固定しない方が安全です。候補にa/bとa\bの両方が絡む場合は、実行環境が指定されているかを先に見ます。
