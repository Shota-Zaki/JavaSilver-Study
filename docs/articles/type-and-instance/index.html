「型」と「実体（インスタンス）」とは

Javaを理解するうえで、次の5つは分けて考える必要があります。

広告スポンサーリンク表示エリア
広告スポンサーリンク表示エリア
広告スポンサーリンク表示エリア
用語 意味
クラス オブジェクトの構造や処理を定義したもの
型 その値を「何として扱うか」を表す分類
変数 値や参照を保持するための名前付き領域
インスタンス クラスをもとに生成された具体的なオブジェクト
参照 インスタンスを指し示すための情報

「実体」という言葉は説明上よく使われますが、Javaでは通常、オブジェクトまたはインスタンスと呼びます。

1. クラスは設計、インスタンスは具体物

次のクラスを考えます。

class Dog {
    String name;

    void bark() {
        System.out.println(name + "が鳴きました");
    }
}

この時点では、まだ具体的な犬のオブジェクトは作られていません。

Dogクラスは、次のような内容を定義しています。

  • nameという情報を持つ
  • bark()という処理を持つ

実際のオブジェクトは、newを使って生成します。

Dog dog = new Dog();

この一文には、複数の要素があります。

Dog dog = new Dog();
│   │       └─ Dogクラスのインスタンスを生成
│   └──────── 変数名
└──────────── 変数dogの型

より正確には、次の処理が行われています。

  1. new Dog()でDogのインスタンスを生成する
  2. 生成されたインスタンスを指す参照を取得する
  3. その参照をdog変数へ代入する

概念的には次の状態です。

変数dog
   │
   │ 参照
   ▼
Dogインスタンス
┌──────────────┐
│ name = null  │
│ bark()       │
└──────────────┘

dog変数そのものがインスタンスなのではありません。

dog変数には、インスタンスを指し示す参照が入っています。

2. 「型」とは何か

型は、値や変数をどの種類として扱うかを決める情報です。

int number = 10;
String text = "Java";
Dog dog = new Dog();

それぞれの型は次のとおりです。

変数 型
number int
text String
dog Dog

型によって、次のことが決まります。

  • 代入できる値
  • 呼び出せるメソッド
  • 参照できるフィールド
  • 使用できる演算
  • コンパイル時に許可される処理

たとえば、Dog型に宣言されていないメソッドは呼び出せません。

Dog dog = new Dog();

dog.bark();   // Dogにbark()があるのでOK
// dog.fly(); // Dogにfly()がないのでコンパイルエラー

つまり型は、コンパイラに対して、

この変数は、この種類の値として扱います

と伝える役割を持っています。

3. プリミティブ型と参照型

Javaの型は、大きく2種類に分けられます。

プリミティブ型

int number = 10;
double rate = 1.5;
boolean result = true;
char letter = 'A';

プリミティブ型の変数は、数値や真偽値などの値を直接扱います。

代表的なプリミティブ型は次の8種類です。

byte
short
int
long
float
double
char
boolean

参照型

String text = "Java";
Dog dog = new Dog();
int[] numbers = new int[3];

参照型には次のようなものがあります。

  • クラス型
  • インターフェース型
  • 配列型
  • 列挙型
  • レコード型

参照型の変数には、オブジェクトそのものではなく、オブジェクトを指す参照が入ります。

4. 変数の型とインスタンスの型

継承では、ここが最も重要です。

class Animal {
}

class Dog extends Animal {
}

DogはAnimalを継承しています。

そのため、次の代入ができます。

Animal animal = new Dog();

このコードには2種類の型が登場します。

Animal animal = new Dog();
^^^^^^              ^^^
変数の型             実際に生成するクラス

整理すると次のとおりです。

項目 内容
変数名 animal
変数の型 Animal
生成されたインスタンスのクラス Dog
実行時の実体 Dogインスタンス

animal変数はAnimal型ですが、指している実体はDogです。

この2つを区別するために、次の用語が使われます。

  • コンパイル時型
  • 宣言型
  • 参照型
  • 実行時型
  • 実体型

本来は文脈によって意味が少し異なりますが、初学者段階では次のように整理できます。

Animal animal = new Dog();

Animal：変数の型、コンパイル時型
Dog   ：実体の型、実行時型

5. なぜ親クラス型に子クラスを代入できるのか

DogはAnimalを継承しています。

class Dog extends Animal {
}

これは、DogがAnimalの一種であることを表します。

Dog is an Animal
犬は動物である

したがって、DogインスタンスをAnimalとして扱えます。

Animal animal = new Dog();

これはアップキャストと呼ばれます。

明示的なキャストは不要です。

Dog dog = new Dog();
Animal animal = dog;

親クラス型の変数には、次のものを代入できます。

  • 親クラス自身のインスタンス
  • その子クラスのインスタンス
  • さらに下位の子クラスのインスタンス
class Animal {
}

class Dog extends Animal {
}

class Shiba extends Dog {
}

Animal a1 = new Animal();
Animal a2 = new Dog();
Animal a3 = new Shiba();

すべて有効です。

一方、逆方向はそのままでは代入できません。

Animal animal = new Animal();

// Dog dog = animal;
// コンパイルエラー

すべてのAnimalがDogとは限らないためです。

犬は動物である    → 正しい
動物はすべて犬である → 正しくない

6. 変数の型は「呼べるメンバー」を決める

次のクラスを考えます。

class Animal {
    void eat() {
        System.out.println("食べます");
    }
}

class Dog extends Animal {
    void bark() {
        System.out.println("鳴きます");
    }
}

次のように変数を宣言します。

Animal animal = new Dog();

animal変数の型はAnimalです。

そのため、コンパイラから見えるのは、基本的にAnimal型に宣言されているメンバーです。

animal.eat(); // OK

// animal.bark();
// コンパイルエラー

実体はDogなので、実際にはbark()を持っています。

しかし、コンパイラは変数の型であるAnimalを基準に、呼び出せるメソッドを判定します。

変数の型：Animal
          ↓
Animalにbark()は宣言されていない
          ↓
コンパイルエラー

ここで重要なのは、

実体が持っているすべてのメソッドを、必ず呼べるわけではない

という点です。

7. オーバーライドされたメソッドは実体で決まる

次のように、親と子に同じメソッドがある場合を考えます。

class Animal {
    void speak() {
        System.out.println("Animal");
    }
}

class Dog extends Animal {
    @Override
    void speak() {
        System.out.println("Dog");
    }
}

次のコードを実行します。

Animal animal = new Dog();
animal.speak();

出力は次です。

Dog

変数の型はAnimalですが、実体はDogです。

インスタンスメソッドがオーバーライドされている場合、最終的に実行される処理は実体の型によって決まります。

判定は2段階です。

第1段階：コンパイル時

animal.speak();

コンパイラはanimalの型であるAnimalを確認します。

Animalにspeak()があるか

Animalにspeak()があるため、呼び出しは許可されます。

第2段階：実行時

実際のインスタンスを確認します。

animalが指している実体はDog

そのため、Dogでオーバーライドされたspeak()が実行されます。

まとめると次のとおりです。

呼び出せるかどうか   → 変数の型で決まる
どの実装が動くか     → 実体の型で決まる

8. 最重要の判定ルール

次のコードを見たときは、2つに分けて考えます。

Animal animal = new Dog();
animal.speak();

ルール1：そのメソッドを呼べるか

左側の型を確認します。

Animal animal
^^^^^^

Animalにspeak()が宣言されていれば、コンパイルできます。

ルール2：どのメソッド本体が動くか

右側で生成されたインスタンスを確認します。

new Dog()
    ^^^

speak()がDogでオーバーライドされていれば、Dog側が実行されます。

9. 具体例

class Animal {
    void eat() {
        System.out.println("Animal.eat");
    }

    void speak() {
        System.out.println("Animal.speak");
    }
}

class Dog extends Animal {
    @Override
    void speak() {
        System.out.println("Dog.speak");
    }

    void fetch() {
        System.out.println("Dog.fetch");
    }
}

次の変数を作ります。

Animal animal = new Dog();

eat()を呼ぶ

animal.eat();

Animalにeat()があります。

Dogではオーバーライドされていません。

出力：

Animal.eat

speak()を呼ぶ

animal.speak();

Animalにspeak()があります。

Dogでオーバーライドされています。

出力：

Dog.speak

fetch()を呼ぶ

animal.fetch();

これはコンパイルエラーです。

fetch()は実体であるDogにはありますが、変数の型であるAnimalにはありません。

10. ダウンキャスト

Animal型の変数からDog固有のメソッドを呼びたい場合、Dog型として扱う必要があります。

Animal animal = new Dog();

Dog dog = (Dog) animal;
dog.fetch();

または次のようにも書けます。

((Dog) animal).fetch();

この処理をダウンキャストと呼びます。

ただし、キャストによってオブジェクトそのものが変化するわけではありません。

Animal animal = new Dog();
Dog dog = (Dog) animal;

実体は最初から最後まで同じDogインスタンスです。

変化したのは、コンパイラにどの型として扱わせるかです。

キャスト前：Animalとして扱う
キャスト後：Dogとして扱う

実体：どちらも同じDogインスタンス

11. 不正なダウンキャスト

次のコードはコンパイルできます。

Animal animal = new Animal();
Dog dog = (Dog) animal;

しかし、実行するとClassCastExceptionが発生します。

java.lang.ClassCastException

animalが指している実体はAnimalであり、Dogではないためです。

変数の型：Animal
実体の型：Animal

Dogとして扱おうとする
        ↓
実体がDogではない
        ↓
ClassCastException

安全に確認するにはinstanceofを使います。

if (animal instanceof Dog) {
    Dog dog = (Dog) animal;
    dog.fetch();
}

Java 17では、パターン変数を使えます。

if (animal instanceof Dog dog) {
    dog.fetch();
}

この場合、実体がDogであるときだけdog変数を使用できます。

12. キャストは実体を変えない

次のコードを考えます。

Animal animal = new Dog();

ここでキャストしても、Dogインスタンスが別のインスタンスに変化するわけではありません。

Dog dog = (Dog) animal;

概念的には次の状態です。

animal ─┐
        ├──▶ 同じDogインスタンス
dog ────┘

2つの変数が、同じインスタンスを参照しています。

13. 複数の変数が同じインスタンスを参照する

class Dog {
    String name;
}

Dog dog1 = new Dog();
Dog dog2 = dog1;

この場合、Dogインスタンスが2つ作られたわけではありません。

作られたインスタンスは1つです。

dog1 ─┐
      ├──▶ Dogインスタンス
dog2 ─┘

そのため、片方から値を変更すると、もう片方から見ても変化しています。

dog1.name = "ポチ";

System.out.println(dog2.name);

出力：

ポチ

dog1とdog2が同じインスタンスを指しているためです。

14. 新しくnewすると別のインスタンス

Dog dog1 = new Dog();
Dog dog2 = new Dog();

今回はnew Dog()を2回実行しています。

そのため、別々のインスタンスが作られます。

dog1 ───▶ DogインスタンスA
dog2 ───▶ DogインスタンスB

dog1.name = "ポチ";
dog2.name = "タロウ";

それぞれ別の状態を持てます。

15. ==は参照先が同じかを調べる

Dog dog1 = new Dog();
Dog dog2 = dog1;

System.out.println(dog1 == dog2);

出力：

true

同じインスタンスを参照しているためです。

一方、次は別のインスタンスです。

Dog dog1 = new Dog();
Dog dog2 = new Dog();

System.out.println(dog1 == dog2);

出力：

false

内容が同じでも、別々に生成されたインスタンスなら==はfalseになります。

オブジェクトの内容を比較する場合は、通常equals()を使用します。ただし、クラス側で適切にequals()が実装されている必要があります。

16. nullは「インスタンスがない」

参照型の変数にはnullを代入できます。

Dog dog = null;

この時点でdog変数の型はDogです。

しかし、指しているインスタンスはありません。

dog ───▶ 参照先なし

したがって、次はコンパイルできます。

Dog dog = null;

しかし、実行時にメソッドを呼ぶと例外になります。

dog.bark();

実行結果：

NullPointerException

コンパイル時にはdogの型がDogなので、bark()を呼べると判断されます。

しかし実行時にはインスタンスが存在しないため、処理を実行できません。

ここでも、コンパイル時型と実体の有無を分けて考える必要があります。

17. 変数の型は固定、参照先は変更可能

Animal animal;

この変数の型は、宣言した時点でAnimalに固定されます。

後からDog型の変数自体に変化するわけではありません。

ただし、参照先は変更できます。

animal = new Animal();
animal = new Dog();
animal = new Cat();

DogとCatがAnimalの子クラスなら代入できます。

変数の型：常にAnimal
参照先    ：Animal、Dog、Catなどに変わる

たとえば次のコードでは、実行されるメソッドも変わります。

class Animal {
    void speak() {
        System.out.println("Animal");
    }
}

class Dog extends Animal {
    @Override
    void speak() {
        System.out.println("Dog");
    }
}

class Cat extends Animal {
    @Override
    void speak() {
        System.out.println("Cat");
    }
}

Animal animal;

animal = new Dog();
animal.speak();

animal = new Cat();
animal.speak();

出力：

Dog
Cat

同じAnimal型変数でも、参照している実体によって動作が変化します。

これがポリモーフィズムの基本です。

18. インターフェース型と実体

「型」はクラスだけではありません。

インターフェースも型として利用できます。

interface Flyable {
    void fly();
}

class Bird implements Flyable {
    @Override
    public void fly() {
        System.out.println("鳥が飛びます");
    }
}

Flyable flyable = new Bird();
flyable.fly();

整理すると次のとおりです。

項目 内容
変数の型 Flyable
実体の型 Bird
呼び出せるメソッド Flyableに宣言されたメソッド
実行される実装 Birdの実装

インターフェース自体は通常、直接インスタンス化できません。

// Flyable flyable = new Flyable();
// コンパイルエラー

しかし、インターフェース型の変数は作れます。

Flyable flyable = new Bird();

「型として使用できること」と「直接インスタンス化できること」は別です。

19. 抽象クラス型と実体

抽象クラスも直接インスタンス化できません。

abstract class Animal {
    abstract void speak();
}

// Animal animal = new Animal();
// コンパイルエラー

しかし、変数の型としては使用できます。

class Dog extends Animal {
    @Override
    void speak() {
        System.out.println("Dog");
    }
}

Animal animal = new Dog();

整理すると次のようになります。

Animal型の変数は作れる
Animalの直接のインスタンスは作れない
DogのインスタンスをAnimal型変数に代入できる

20. 実際の開発でよく使う例

Javaでは、具体的なクラスよりもインターフェース型で変数を宣言することがよくあります。

List<String> list = new ArrayList<>();

このコードでは次のようになっています。

部分 意味
List<String> 変数の型
list 変数名
new ArrayList<>() 実際に生成するインスタンス 
変数の型：List
実体の型：ArrayList

ArrayListはListインターフェースを実装しています。

そのため、ArrayListインスタンスをList型として扱えます。

List<String> list = new ArrayList<>();

実装を変更する場合も、変数の型を変えずに済みます。

List<String> list = new LinkedList<>();

変数の型はListのままで、実体だけを変更しています。

この設計には次の利点があります。

  • 実装を交換しやすい
  • 呼び出し側が具体的な実装に依存しにくい
  • テストしやすい
  • 保守しやすい

21. メソッドの引数でも型と実体を分ける

void makeSpeak(Animal animal) {
    animal.speak();
}

このメソッドの引数型はAnimalです。

そのため、Animalの子クラスを渡せます。

makeSpeak(new Dog());
makeSpeak(new Cat());

実行されるspeak()は、渡された実体によって決まります。

class Dog extends Animal {
    @Override
    void speak() {
        System.out.println("Dog");
    }
}

class Cat extends Animal {
    @Override
    void speak() {
        System.out.println("Cat");
    }
}

makeSpeak(new Dog()); // Dog
makeSpeak(new Cat()); // Cat

makeSpeak()側は、具体的にDogかCatかを意識する必要がありません。

Animalとして扱いながら、実際の処理は各実体に任せられます。

22. 戻り値でも型と実体が異なる

Animal createAnimal() {
    return new Dog();
}

戻り値型はAnimalですが、返している実体はDogです。

Animal animal = createAnimal();

呼び出し側からはAnimalとして扱います。

しかし、オーバーライドされたメソッドはDog側が実行されます。

animal.speak();

23. フィールドは実体ではなく変数の型で決まる

メソッドとフィールドは規則が異なります。

class Animal {
    String name = "Animal";
}

class Dog extends Animal {
    String name = "Dog";
}

Animal animal = new Dog();

System.out.println(animal.name);

出力：

Animal

フィールドはオーバーライドされません。

同名フィールドがある場合は隠蔽になり、参照変数の型で選ばれます。

Animal animal = new Dog();
Dog dog = new Dog();

System.out.println(animal.name); // Animal
System.out.println(dog.name);    // Dog

ここはインスタンスメソッドとの重要な違いです。

対象 選択基準
オーバーライドされたインスタンスメソッド 実体の型
フィールド 変数の型
staticメソッド 変数や型名のコンパイル時型

24. 型が変わると見える範囲が変わる

同じインスタンスでも、どの型の変数から見るかによって、利用できるメンバーが変わります。

class Animal {
    void eat() {
    }
}

class Dog extends Animal {
    void bark() {
    }
}

Dog dog = new Dog();
Animal animal = dog;

dogとanimalは同じインスタンスを指しています。

dog ─────┐
         ├──▶ 同じDogインスタンス
animal ──┘

しかし、呼べるメソッドは異なります。

dog.eat();    // OK
dog.bark();   // OK

animal.eat(); // OK
// animal.bark(); // コンパイルエラー

animal側から見ると、Animal型として公開されている範囲しか使用できません。

これは、同じ人を異なる立場から見るようなものです。

実体：同じ人物

社員として扱う → 社員として使える操作
管理者として扱う → 管理者として使える操作

ただし、この比喩は「実体そのものが変わる」という意味ではありません。見せる機能の範囲が変わるだけです。

25. オーバーロードは変数の型で選ばれる

オーバーライドとオーバーロードは判定方法が異なります。

class Printer {
    void print(Animal animal) {
        System.out.println("Animal");
    }

    void print(Dog dog) {
        System.out.println("Dog");
    }
}

Animal animal = new Dog();

Printer printer = new Printer();
printer.print(animal);

出力：

Animal

実体はDogですが、変数animalのコンパイル時型はAnimalです。

オーバーロードはコンパイル時の型で選択されるため、print(Animal)が呼ばれます。

Dog dog = new Dog();
printer.print(dog);

こちらは出力が次になります。

Dog

整理すると次のとおりです。

処理 選択時期 基準
オーバーロード コンパイル時 変数・引数の型
オーバーライド 実行時 実体の型

26. getClass()で実体のクラスを確認する

Animal animal = new Dog();

System.out.println(animal.getClass());

出力例：

class Dog

getClass()は実際のオブジェクトのクラスを返します。

System.out.println(animal.getClass() == Dog.class);

結果：

true

ただし、通常の継承判定ではinstanceofの方が適しています。

System.out.println(animal instanceof Dog);    // true
System.out.println(animal instanceof Animal); // true

DogインスタンスはDogであると同時に、継承関係上Animalとしても扱えます。

27. インスタンスは複数の型として扱える

interface Runnable {
    void run();
}

interface Swimmable {
    void swim();
}

class Dog extends Animal implements Runnable, Swimmable {
    @Override
    public void run() {
    }

    @Override
    public void swim() {
    }
}

1つのDogインスタンスを、複数の型の変数から参照できます。

Dog dog = new Dog();

Animal animal = dog;
Runnable runnable = dog;
Swimmable swimmable = dog;

すべて同じインスタンスを参照しています。

Dog変数 ───────┐
Animal変数 ────┤
Runnable変数 ──┼──▶ 同じDogインスタンス
Swimmable変数 ─┘

各変数から利用できるメソッドは、それぞれの変数の型によって決まります。

28. 初学者が混同しやすい点

誤解1：変数がオブジェクトそのものである

Dog dog = new Dog();

dogはオブジェクトそのものではなく、オブジェクトを参照する変数です。

誤解2：左辺と右辺の型は必ず同じ

継承関係やインターフェース実装関係があれば異なっていても構いません。

Animal animal = new Dog();
List<String> list = new ArrayList<>();

誤解3：実体が持つメソッドはすべて呼べる

呼び出せるメソッドは、まず変数の型で判定されます。

Animal animal = new Dog();

// animal.fetch(); // Animalにないためコンパイルエラー

誤解4：キャストすると実体が変化する

キャストは実体を作り替える処理ではありません。

Dog dog = (Dog) animal;

同じインスタンスをDog型として扱い直しているだけです。

誤解5：親型に代入すると子クラス部分が消える

Animal animal = new Dog();

Dog固有の情報が消えるわけではありません。

Animal型の変数から直接見えなくなるだけです。

ダウンキャストすれば、実体が本当にDogである場合は再び利用できます。

誤解6：変数の型で実行されるメソッドが決まる

オーバーライドされたインスタンスメソッドは、実体の型で決まります。

Animal animal = new Dog();
animal.speak(); // Dog側

29. 型と実体を判断する手順

次のコードを見たとします。

Animal animal = new Dog();
animal.speak();

次の順番で確認します。

1. 変数の型を確認する

Animal animal

変数の型はAnimalです。

2. 実体の型を確認する

new Dog()

実体の型はDogです。

3. 呼び出すメソッドが変数の型に存在するか確認する

Animalにspeak()があるか

なければコンパイルエラーです。

4. オーバーライドされているか確認する

Dogがspeak()をオーバーライドしていれば、Dog側が動きます。

5. フィールドやstaticの場合は別規則

フィールドやstaticメソッドは、実体ではなくコンパイル時型を基準に判断します。

30. まとめ

Animal animal = new Dog();

この一文を正確に説明すると、次のようになります。

Animal型の参照変数animalに、Dogクラスから生成したインスタンスへの参照を代入している。

重要な規則は次のとおりです。

観点 基準
変数へ代入できるか 型の継承・実装関係
どのメソッドを記述できるか 変数のコンパイル時型
オーバーライドされたどの実装が動くか 実体の実行時型
どのフィールドが選ばれるか 変数のコンパイル時型
どのstaticメソッドが選ばれるか コンパイル時型
キャストが成功するか 実体が指定型として扱えるか
nullの場合 型はあるが参照先インスタンスはない

最も重要な形はこれです。

Animal animal = new Dog();

Animal：この変数を何として扱うか
Dog   ：実際に存在するオブジェクトは何か

そして、メソッド呼び出しでは次のように判断します。

呼べるか         → 変数の型
どの実装が動くか → 実体の型

この区別ができると、継承、オーバーライド、ポリモーフィズム、キャスト、インターフェースの理解が大幅に安定します。
