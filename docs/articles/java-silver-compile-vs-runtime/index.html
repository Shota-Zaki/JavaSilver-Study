コンパイルエラー・実行時例外・正常終了の分け方

Java 11とJava 17の共通の判定方法です。プログラムが止まる時点を3分類し、出力やcatchを追う前に実行段階へ到達するかを確定します。

3分類は発生時点が違う

分類発生時点代表例出力追跡
コンパイルエラーjavacがソースを検査するとき型不一致、未初期化、到達不能、checked例外未処理実行されないため不要。
実行時例外JVMが該当式を実行したときnull使用、範囲外、キャスト失敗、整数0除算発生式より前だけ追う。
正常終了最後まで例外なく実行型・範囲・参照がすべて成立全経路を追う。

同じ「止まる」でも原因は異なる

// 1. コンパイルエラー
int a;
// System.out.println(a); // ローカル変数が未初期化

// 2. 実行時例外
String b = null;
// System.out.println(b.length()); // NullPointerException

// 3. 正常実行
String c = "Java";
System.out.println(c.length()); // 4

1は実行前に拒否されます。2はコンパイル後、メソッド呼び出しの式を実行した時点で停止します。3は正常に4を表示します。

catchで扱えるのは、コンパイル後に投げられた例外

public class CatchTiming {
    public static void main(String[] args) {
        try {
            System.out.print("A");
            String text = null;
            System.out.print(text.length());
        } catch (NullPointerException e) {
            System.out.print("B");
        }
    }
}

このコードはコンパイルでき、実行時のNullPointerExceptionをcatchします。出力はABです。構文エラーや型不一致をtry-catchで回避することはできません。

checked例外は「例外」でもコンパイル時に処理を要求される

import java.io.IOException;

class CheckedExample {
    static void read() throws IOException {
        throw new IOException();
    }

    public static void main(String[] args) throws IOException {
        read();
    }
}

IOExceptionが実際に投げられるのは実行時ですが、呼び出し側が処理または宣言しているかはコンパイル時に検査されます。throws IOExceptionを削除し、catchも追加しなければコンパイルエラーです。

判定手順

  1. 対象Javaバージョンで構文とAPIが使えるか。
  2. 宣言、型、アクセス、戻り値、例外処理が成立するか。
  3. 成立する場合だけ、実行時に最初に危険な式を探す。
  4. 例外がcatchされるか、外へ伝播するかを追う。
  5. 正常に通過した文の出力を順に並べる。

記事末尾の確認

  • コンパイルエラーでは実行が始まらないことを説明できる。
  • 実行時例外が起きる正確な式を示せる。
  • checked例外の処理義務と、例外が投げられる時点を混同していない。
