javac・javaコマンドで詰まる原因

コンパイルと実行は別工程です。javac は .java を検査して .class を作るコマンド、java はクラス名を指定してJVMを起動するコマンドです。ここを曖昧にすると、コマンドコードはほぼ読み違えます。

最初に固定する判定順

コマンドコードは、出力結果を追う前に次の順番で切ります。

  1. ソースファイル名と publicトップレベルクラス名 が一致しているか。
  2. javac に渡しているものが ソースファイル か。クラス名や .class を渡していないか。
  3. package がある場合、-d で正しい出力先に .class を作っているか。
  4. java に渡しているものが 完全修飾クラス名 か。.class やパスを渡していないか。
  5. 起動対象クラスに、起動できる形の main メソッドがあるか。
重要: javac はファイルを指定します。java はクラス名を指定します。この1文を基準にすると、かなりの誤りを切れます。

javac と java の役割

コマンド入力出力・動作よくある誤り
javacSample.java のようなソースファイルSample.class を作るjavac Sample、javac Sample.class
javaSample や ex.Sample のようなクラス名JVMを起動して main を呼ぶjava Sample.class、java ex/Sample
 
// Sample.java
public class Sample {
    public static void main(String[] args) {
        System.out.println("OK");
    }
}
# コンパイル
javac Sample.java

# 実行
java Sample

java Sample.class は、クラス名に .class を付けているので誤りです。java コマンドはファイル名ではなく、JVMに読み込ませるクラス名を指定します。

publicクラス名とファイル名

トップレベルの public クラスがある場合、ファイル名はそのクラス名と一致しなければなりません。大文字小文字も区別します。

ファイル名コード結果理由
Sample.javapublic class Sample {}OKpublicクラス名とファイル名が一致
Main.javapublic class Sample {}NGSample は Sample.java に置く必要がある
Memo.javaclass Sample {}OKpublicでなければファイル名一致は必須ではない
Sample.javapublic class A {} public class B {}NG1ファイルにpublicトップレベルクラスは複数書けない
切り分け: ファイル名の制約を受けるのは public なトップレベルクラスです。public ではないトップレベルクラスは、ファイル名と一致しなくてもコンパイルできます。

1つのファイルに複数のclassを書いた場合

Javaでは、1つの .java ファイルに複数のトップレベルクラスを書けます。ただし、publicトップレベルクラスは最大1つです。javac はファイル内の各トップレベルクラスごとに .class ファイルを作ります。

// A.java
public class A {
    public static void main(String[] args) {
        System.out.println("A");
    }
}

class B {
    public static void main(String[] args) {
        System.out.println("B");
    }
}

class C {
}
javac A.java

コンパイル後は、次のようにクラスごとの .class が生成されます。

A.java
A.class
B.class
C.class
コマンド判定理由
java AOKA に正しい main がある
java BOKB はpublicでないが、トップレベルクラスであり、正しい main がある
java C起動失敗C.class は存在するが、main がない
java A.classNGjava に渡すのはファイル名ではなくクラス名
重要: 実行時に指定するのは「ファイル名」ではなく「クラス名」です。A.java から生成された B.class でも、B に正しい main があれば java B で起動できます。

publicクラスがない場合

トップレベルクラスがすべて public ではないなら、ファイル名は任意です。

// Memo.java
class A {
    public static void main(String[] args) {
        System.out.println("A");
    }
}

class B {
    public static void main(String[] args) {
        System.out.println("B");
    }
}
javac Memo.java
java A
java B

Memo.java というファイル名でも、生成されるのは Memo.class ではなく、A.class と B.class です。class Memo が存在しないなら、java Memo は起動できません。

packageがある場合

package 宣言は、そのファイル内のすべてのトップレベルクラスに適用されます。

// src/ex/A.java
package ex;

public class A {
    public static void main(String[] args) {
        System.out.println("A");
    }
}

class B {
    public static void main(String[] args) {
        System.out.println("B");
    }
}
javac -d out src/ex/A.java
java -cp out ex.A
java -cp out ex.B

B も ex パッケージ所属なので、java -cp out B ではなく java -cp out ex.B です。-d out を使った場合、生成物は out/ex/A.class と out/ex/B.class になります。

不可能パターン

コード・コマンド判定理由
public class A {} public class B {}NG1つのソースファイルにpublicトップレベルクラスは複数書けない
B.java に public class A {}NGpublicクラス A は A.java に置く必要がある
javac A.java 後に java A.java通常はNGコンパイル済みclassの起動ではクラス名を指定する。source-file modeとは別
java MemoNGになり得るMemo.java というファイル名でも、class Memo がなければ Memo.class は生成されない
java B を package ex; 付きの B に対して実行NG完全修飾クラス名 ex.B を指定する必要がある

mainメソッドのOK・NG

java コマンドで通常起動するには、起動対象クラスに正しい main が必要です。

宣言判定理由
public static void main(String[] args)OK標準形
public static void main(String... args)OK可変長引数は配列として扱える
static public void main(String[] args)OKpublicとstaticは、どちらを先に書いてもよい
public static void main(String args[])OK配列記法の位置が違うだけ
public void main(String[] args)NGstatic ではない
static void main(String[] args)NGpublic ではない
public static int main(String[] args)NG戻り値が void ではない
public static void main(String args)NG引数が String 配列または可変長引数ではない
注意: main が間違っていても、クラス自体はコンパイルできる場合があります。その場合はコンパイルエラーではなく、起動時にmainメソッドが見つからない状態です。

packageなしの最小パターン

まずは package がない単純なケースを固定します。

work/
  Sample.java
# work ディレクトリで実行
javac Sample.java
java Sample

コンパイル後は次のようになります。

work/
  Sample.java
  Sample.class
コマンド判定理由
javac Sample.javaOKソースファイルをコンパイルしている
java SampleOKクラス名を指定している
javac SampleNGjavac にソースファイル名を渡していない
java Sample.java原則NGとして扱う通常のコンパイル済みクラス実行ではクラス名を指定する。Java 11以降の単一ソースファイル実行モードは別扱い
java Sample.classNGjava は .class ファイル名ではなくクラス名を受け取る

-d でclassファイルの出力先を分ける

-d は、コンパイル後の .class ファイルをどこに出力するかを指定するオプションです。ソースとclassを混ぜないために使います。

work/
  src/
    Sample.java
  out/
# work ディレクトリで実行
javac -d out src/Sample.java
java -cp out Sample

コンパイル後は次の形になります。

work/
  src/
    Sample.java
  out/
    Sample.class

-d out を使うと、Sample.class は src ではなく out に作られます。そのため、実行時には -cp out でクラス検索場所を指定します。

コマンド判定理由
javac -d out src/Sample.javaOKsrc/Sample.java をコンパイルし、out にclassを出力
java -cp out SampleOKout から Sample.class を探せる
java SampleNGになりやすいカレントディレクトリに Sample.class がないなら見つからない
javac -d src/Sample.java outNG-d の直後に出力先ディレクトリを置く

packageありのコンパイルと実行

package がある場合、実行時はパッケージ名を含めた完全修飾クラス名を指定します。

// src/ex/Sample.java
package ex;

public class Sample {
    public static void main(String[] args) {
        System.out.println("ex.Sample");
    }
}
javac -d out src/ex/Sample.java
java -cp out ex.Sample

-d out によって、package ex; に対応する ex ディレクトリが out の下に作られます。

work/
  src/
    ex/
      Sample.java
  out/
    ex/
      Sample.class
コマンド判定理由
java -cp out ex.SampleOKクラスパスのルートが out、クラス名が ex.Sample
java -cp out SampleNGSample ではなく ex.Sample というクラス
java -cp out ex/SampleNGパスではなくドット区切りのクラス名を指定する
java -cp out ex.Sample.classNG.class は付けない
java -cp out/ex SampleNGクラスパスはパッケージの親ディレクトリにする。out/ex ではなく out

複数ファイルのコンパイル

別クラスを参照している場合、依存するソースも一緒にコンパイルします。

src/
  app/
    Main.java
    Message.java
// src/app/Main.java
package app;

public class Main {
    public static void main(String[] args) {
        System.out.println(Message.text());
    }
}
// src/app/Message.java
package app;

class Message {
    static String text() {
        return "hello";
    }
}
javac -d out src/app/Main.java src/app/Message.java
java -cp out app.Main

同じディレクトリ内のソースをまとめてコンパイルするなら、次のようにも書けます。

javac -d out src/app/*.java
Windows PowerShellでも *.java は使えます。ただし、サブディレクトリまで自動で再帰的に拾うわけではありません。

-cp / -classpath の意味

-cp は -classpath の短縮形です。実行時にJVMがクラスを探す場所を指定します。

java -cp out app.Main
指定意味
-cp outout をクラス検索の起点にする
-cp out app.Mainout/app/Main.class を探して起動する
-cp src app.Mainsrc に .class がなければ失敗する。.java だけでは通常実行できない

複数の場所を指定する場合、区切り文字はOSで異なります。

# Windows
java -cp out;lib/sample.jar app.Main

# macOS / Linux
java -cp out:lib/sample.jar app.Main

不可能・誤りパターン一覧

正しいコマンドよりも「なぜそれが無理か」を説明できる方が判定の軸です。

パターン判定理由
javac SampleNGjavac は基本的に .java ソースファイルを指定する
javac Sample.classNG.class はコンパイル結果であり、コンパイル対象ではない
java Sample.classNGjava にはファイル名ではなくクラス名を渡す
java out/SampleNGパスではなく、-cp out Sample のように分ける
java ex.Sample だけNGになりやすいex/Sample.class がカレントディレクトリ配下にないなら見つからない。必要なら -cp を使う
java -cp out/ex SampleNGパッケージ付きクラスは、パッケージの親をクラスパスに指定する
javac -d out ex.SampleNGjavac に渡すのはクラス名ではなくソースファイル
java -d out SampleNG-d は javac の出力先指定。java の実行時指定ではない

Java 11以降の source-file mode との区別

Java 11以降では、単一ソースファイルを直接実行する source-file mode があります。

java Sample.java

ただし、これは「javac でコンパイルしてから java でclassを起動する」通常パターンとは別です。コード例で javac Sample.java の後に実行する実行方法を比較する場合は、基本は java Sample を選びます。

source-file modeで1つのファイルに複数のトップレベルクラスがある場合、起動対象はファイル内で最初に宣言されたトップレベルクラスです。ファイル名と同じクラスではありません。最初のクラスに main がなければ、後続のクラスに main があっても起動できません。

観点通常の実行source-file mode
コマンドjavac Sample.java → java Samplejava Sample.java
指定するものクラス名ソースファイル名
扱い基本パターン明示されている場合だけ考える
複数classがある場合生成された各classをクラス名で起動最初のトップレベルクラスが起動対象
注意: java Sample.javaはJava 11以降の単一ソースファイル実行で利用できます。「常に使えない」という判断は誤りです。ただし、コンパイル済みclassの起動コードでは java Sample が基本です。

パッケージ宣言とディレクトリの注意

package 宣言はソースファイルの先頭に書きます。コメント以外を package より前に置くとコンパイルエラーです。

// OK
package ex;
import java.util.List;
public class Sample { }
// NG
import java.util.List;
package ex;
public class Sample { }

また、package ex; のクラスを実行するときは、JVMには ex.Sample として見えます。ソースがどこに置かれていたかではなく、コンパイル後の .class がクラスパス上でどう配置されているかが判定の軸です。

起動引数 args の基本

java Sample A B C のように実行すると、args は長さ3の配列になります。

public class Sample {
    public static void main(String[] args) {
        System.out.println(args.length);
        System.out.println(args[0]);
    }
}
javac Sample.java
java Sample A B C

出力は次のとおりです。

3
A
実行args.length注意
java Sample0args は null ではない。長さ0の配列
java Sample A1args[0] は "A"
java Sample A B2args[1] は "B"
引数なしで args[0] を読むと、コンパイルエラーではなく ArrayIndexOutOfBoundsException です。コンパイル時には実行時引数の数は決まりません。

コンパイルと実行の切り分け

出力を追う前に、次のテンプレートで判定します。

1. publicクラス名とファイル名は一致しているか
2. package宣言は先頭にあるか
3. javac に .java を渡しているか
4. -d の出力先と class の配置は合っているか
5. java に完全修飾クラス名を渡しているか
6. -cp はパッケージの親ディレクトリを指しているか
7. 1ファイルに複数classがある場合、どのclassが生成・起動対象か
8. main は public static void main(String[] or String...) か
9. 起動引数を読む箇所で実行時例外が起きないか

この順番で見ると、コンパイルエラー、起動失敗、実行時例外、正常出力を混ぜずに判断できます。

ミニドリル

次のコマンドが成功するか、理由付きで判定します。

// src/ex/Main.java
package ex;
public class Main {
    public static void main(String[] args) {
        System.out.println(args.length);
    }
}
コマンド判定の観点
javac -d out src/ex/Main.java.java を渡しているか。-d の出力先は正しいか。
java -cp out ex.Main A B完全修飾クラス名か。args.length は何か。
java -cp out Mainパッケージ名を省略していないか。
java -cp out/ex Mainクラスパスをパッケージの親にしているか。
java -cp out ex/Mainクラス名をパスで書いていないか。
この単元で必要なのは暗記ではありません。javac はソース、java はクラス名、-d は出力先、-cp は検索場所。この4つに分解します。
