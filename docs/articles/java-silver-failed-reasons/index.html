Java Silverで落ちる理由

Java Silverで落ちる原因は、知識量だけではありません。多くは、コードを読む順番が崩れ、コンパイル可否・実行時例外・出力結果の切り分けに失敗することです。

先に結論

不合格後に最初にやることは、問題集の周回数を増やすことではなく、失点の種類を分けることです。Silverは、出力候補を覚える試験ではなく、型、参照、例外、実行順を順番に切る試験です。

見る基準
  • 出力を見る前にコンパイル可否を見ているか
  • 型変換と数値昇格を式の型で判断しているか
  • 参照型と実体型を分けているか
  • 例外発生後の制御の移動を追っているか
  • 模試後に弱点単元へ戻っているか
Java Practice Labでの使い方模試の誤答をタグ単位で分け、該当する単元別解説と通常演習へ戻します。正答率より、同じ理由で再度間違えない状態を優先します。

このページの内容

  • 出力より前にコンパイル可否を見る
  • 型変換を式の型で見る
  • 参照型と実体型を分ける
  • 例外後の流れを見る
  • 模試後に戻る単元を決める

原因1：出力より前にコンパイル可否を見ていない

出力候補から考えると、コンパイルエラーを見落とします。Silverでは、出力問題でも最初に型、アクセス修飾子、例外処理、override条件を見ます。

byte a = 10;
byte b = 20;
byte c = a + b;
System.out.println(c);

a + b の結果は int です。byte へそのまま代入できないため、出力まで進みません。

原因2：型変換を値の大きさだけで読んでいる

Javaでは、値が小さく見えても式の型が合わなければ代入できません。定数式と変数を使った式も分けます。

コード判定理由
byte c = 10 + 20;コンパイルできる定数式でbyte範囲内
byte c = a + b;コンパイルエラー変数同士の計算結果はint

原因3：参照型と実体型を分けていない

Parent p = new Child(); のようなコードでは、コンパイル時に見る型は Parent、実行時の実体は Child です。呼び出せるかは参照型で判定し、overrideされたインスタンスメソッドだけが実体型で選ばれます。

フィールド、staticメソッド、privateメソッドは同じ扱いではありません。ここをまとめて読むと、継承問題の正答率が不安定になります。

原因4：例外発生後もtryの続きを読む

例外が発生すると、tryブロックの後続行には戻りません。対応するcatchへ移り、finallyがあれば最後に実行します。

try {
    String s = null;
    System.out.println(s.length());
    System.out.println("A");
} catch (NullPointerException e) {
    System.out.println("B");
} finally {
    System.out.println("C");
}

出力は B、C です。"A" は実行されません。

原因5：模試を解くだけで弱点単元に戻っていない

模試は弱点を見つける道具です。採点して終わると、次の模試でも同じ単元で落とします。

  1. 間違えた問題をタグで分ける
  2. 一番多いタグの単元別解説へ戻る
  3. 対応する通常演習を解く
  4. 同じ論点を別問題で解けるか確認する
  5. 固定模試で時間配分も確認する
