Java基礎 Lesson 20

式と文の実行規則

式、文、ブロック、スコープ、演算子の優先順位、オペランドの評価順を定義します。

このレッスンのゴール

コードの構文構造と実行順序を分け、式の値と副作用を順に追跡できること。

基本用語の定義

ソースコードの見た目の並びだけではなく、演算子の結び付きとオペランドの評価順を分けて考えます。

評価順とは式やメソッド呼び出しが処理される順番。
制御フローとはif、for、return、例外などで処理が進む道筋。
スコープとは変数や名前が有効な範囲。
ライフタイムとは変数やオブジェクトが使える期間。
副作用とは値を返すだけでなく、変数変更や出力など状態を変えること。

式・文・ブロック・評価の定義

項目定義・規則具体例
式評価すると値になるコードのまとまり。a + b、method()、x = 10。
文実行の単位となる構文。式文、if文、return文など。
ブロック波かっこで囲まれた文の並び。独自のスコープを作ります。
優先順位どの演算子が強く結び付くかを決める規則。1 + 2 * 3は1 + (2 * 3)。
評価順オペランドを実際に評価する順序。Javaでは演算子のオペランドを原則として左から評価します。
副作用値の計算以外に状態を変える動作。代入、インクリメント、メソッド呼び出しなど。

読解の基本手順

  1. publicクラス名・ファイル名・mainの形を見る。
  2. 宣言されている変数の型を見る。
  3. コンパイルエラーになる箇所がないか判定する。
  4. newで作られるインスタンスや配列を図にする。
  5. 値が変わる行だけを追う。
  6. メソッド呼び出しでどこへ移動するか判定する。
  7. 最後に出力順を判定する。

小さい例で読む

class Item {
    int price;
}

public class Main {
    public static void main(String[] args) {
        Item a = new Item();
        Item b = a;
        b.price = 100;
        System.out.println(a.price);
    }
}

読み方は、aが新しいItemを指す、b = aで同じItemを指す、b.priceを書き換える、同じ実体をaから読んで100が出る、という順。

値の変化だけを表にする

行abオブジェクトのprice
Item a = new Item();Item#1-0
Item b = a;Item#1Item#10
b.price = 100;Item#1Item#1100
println(a.price)Item#1Item#1100を読む

読み違いを防ぐための確認

  • いきなり出力だけを予想する。
  • コンパイルエラー判定を飛ばす。
  • 宣言型と実体型を混同する。
  • Stringや配列の参照を図にせず頭だけで追う。
  • ++や+=の副作用を読み飛ばす。

優先順位と評価順を分けて確認する

`Main.java`として保存する完全なコード

public class Main {
    static int value(String label, int n) {
        System.out.print(label);
        return n;
    }

    public static void main(String[] args) {
        int result = value("A", 1) + value("B", 2) * value("C", 3);
        System.out.println("=" + result);
    }
}

演算の結び付きは1 + (2 * 3)ですが、メソッド呼び出しは左からA、B、Cの順に評価されます。出力はABC=7です。

関連する用語との違い

混同しやすいもの違い見分け方
定義順 / 実行順上に書いてあるから先に実行されるとは限らない。mainから呼び出しを追う。
型 / 実体変数の型とnewされた型が異なる場合がある。A a = new B()を分解する。
宣言 / 代入宣言だけでは値が入らない場合がある。初期化済みか見る。

式と文を処理する規則

  1. 構文として成立するか、型が適合するかを先に確認する。
  2. 演算子の優先順位と結合規則で式の構造を決める。
  3. オペランドを左から評価し、各副作用をその時点で反映する。
  4. if・ループ・returnなどの制御文で次に実行する文を決める。
  5. ブロックを出た後は、そのブロック内で宣言したローカル変数を参照できない。

確認用コード

class Item {
    int price;
    Item(int price) { this.price = price; }
    void change(int price) { this.price = price; }
}
Item item = new Item(100);
item.change(200);
System.out.println(item.price);
見る点このコードでの判断結論
new Item(100)コンストラクタ実行price=100
change(200)フィールドを200に更新price=200
出力item.price200

読む順番を固定する

コード読解では、いきなり出力を計算しません。まずコンパイルできるか、次にどこから実行が始まるか、その後に変数や参照がどう変わるかを追います。

順番判定内容飛ばすと起きる失敗
1クラス・メソッド・型が成立しているかコンパイル不可なのに出力を考える
2mainからどこへ進むか呼ばれないメソッドの処理を読んでしまう
3変数・参照・条件式の変化古い値のまま計算する
4出力、return、例外、finally途中で制御が移る箇所を見失う
int x = 1;
int y = x++ + ++x;
System.out.println(x + "," + y);

このようなコードを見たとき、いきなり暗算せず、式を左から分けて追います。x++で「式には元の値を使い、その後増える」、++xで「先に増えて、その値を使う」と順番に処理します。表を書けばミスは減ります。

ここで整理する誤解

コード読解は、決まった順番で値や参照を表にして追うほど正確になります。頭の中だけで一度に計算せず、変化する行を分けて記録します。

定義と規則の整理

コード読解で順番を間違える点

つまずき正しい考え方
上から全部読もうとするまずコンパイル可否を確認し、コンパイルできる場合に実行時例外の有無と出力を追う。
予想した出力に合わせて読み方を変える出力を予想する前に、コンパイルできない行と例外が発生する式を確認する。
状態変化を書かない変数、参照、配列、フィールド値は表にしないと高確率で読み違える。

確認できれば次へ

  • コンパイル可否→実行可否→出力の順で説明できる
  • 変数表が必要なコードを判断できる
  • 参照の変化を図で説明できる
理解の基準

成立するコードと成立しないコード

成立する例：コンパイル可否から読む

int a = 1;
int b = ++a + a;
System.out.println(b);

成立しない例：出力以前に型が合っていない

int a = "1";
int b = ++a + a;
System.out.println(b);

成立する形：型を合わせてから評価順を読む

int a = 1;
int b = ++a + a;
System.out.println(b); // 4
確認ポイント

Javaコードは出力から考え始めず、最初にコンパイル可否を確認します。

定義と規則を確認する

用語の確認

コード読解で見る順番を説明できますか。

説明を見る

まずコンパイル可否、次に実行可否、例外発生、最後に出力結果です。出力から考え始めるとコンパイルエラーを見落とします。

コードの確認

次のコードは出力を考える前に何を判定しますか。

int n = "10";
System.out.println(n);
説明を見る

コンパイル可否です。Stringをintへ代入しているためコンパイルエラーです。出力はありません。

仕組みを説明する

同じコードについて、コンパイルエラーと実行時例外が通常は同時に成立しない理由を説明してください。

説明を見る

コンパイルエラーなら実行段階に進みません。実行時例外はコンパイルできた後に起きるため、同じコードの結果としては通常同時に成立しません。

基本ルールのまとめ

  • コンパイル可否を最初に確認できる。
  • mainや呼び出し元から実行順を追える。
  • 値と参照の変化を表やメモにして確認できる。

コードに規則を当てはめる

コードを読むとき、出力を考える前に判定すべき最初の3点を答える。

説明を見る

1つ目は、コンパイルできる構造か。ファイル名、クラス名、main、型の整合性を見る。

2つ目は、実行入口がどこか。通常はmainから追う。

3つ目は、変数の型と初期値。特にローカル変数、フィールド、参照型を分ける。
