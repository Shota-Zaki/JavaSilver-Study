Java基礎 Lesson 23

メモリ・スタック・ヒープ

JVMのヒープ、Java仮想マシンスタック、スタックフレーム、参照、到達可能性、ガベージコレクションを説明します。

このレッスンのゴール

メソッド呼び出しごとのローカル変数と、共有されるオブジェクトの関係を学習用モデルとして説明できること。

基本用語の定義

スタックとヒープはJVMの実行時データ領域を理解するためのモデルです。特定JVMの物理配置や最適化を断定せず、仕様上の役割を中心に扱います。

メモリとは実行中の値やオブジェクトを保持する領域。
スタックとは学習上は、主にメソッド呼び出しやローカル変数を追うための領域として扱う。
ヒープとは学習上は、主にnewされたオブジェクトを追うための領域として扱う。
nullとは参照型変数がどのオブジェクトも指していない状態。
ガベージコレクションとは参照されなくなったオブジェクトをJVMが回収する仕組み。

JVMの実行時データ領域と参照の規則

項目定義・規則具体例
ヒープすべてのスレッドで共有され、オブジェクトや配列が割り当てられる実行時領域。ガベージコレクションの対象です。
Java仮想マシンスタックスレッドごとに作られ、メソッド呼び出しごとのフレームを保持します。別スレッドとは共有しません。
スタックフレーム1回のメソッド呼び出しに対応するデータ領域。ローカル変数や演算途中の値を扱います。
参照オブジェクトへ到達するための値。複数の変数が同じオブジェクトを参照できます。
到達可能性実行中のルートから参照をたどってオブジェクトへ到達できる状態。到達不能になったオブジェクトは回収候補です。
ガベージコレクション到達不能なオブジェクトが使う領域を自動的に回収する仕組み。実行時期は保証されず、System.gc()も要求にすぎません。

スタックとヒープの役割

領域何を置くと読むか学習上の使い道
スタック学習上は、メソッド呼び出しやローカル変数など今どのメソッドを実行中か、ローカル変数が何を持つかを見る
ヒープ学習上は、newされたオブジェクトや配列参照型変数が何を指しているかを見る

参照型の図解を言葉で書く

Item a = new Item();
Item b = a;

このとき、ヒープ上にItemオブジェクトが1つ作られる。スタック上のaとbは、その同じオブジェクトを指す参照を持つ。

b = new Item();

ここでbだけが新しいItemを指すようになる。aが指す先は変わらない。

プリミティブ型は値のコピー

int a = 10;
int b = a;
b = 20;
System.out.println(a); // 10

intでは値がコピーされる。bを変えてもaは変わらない。参照型の「同じオブジェクトを見ている」動きとは別です。

ガベージコレクションの入口

どこからも参照されなくなったオブジェクトは、将来的にガベージコレクションの対象になる。いつ回収されるかをプログラマが正確に決めるわけではない。

Object a = new Object();
a = null; // さきほどのObjectは参照されなくなる

参照の共有と到達可能性

`Main.java`として保存する完全なコード

class Box {
    int value;
}

public class Main {
    public static void main(String[] args) {
        Box first = new Box();
        first.value = 10;

        Box second = first;
        first = null;

        System.out.println(second.value);
    }
}

first = null後もsecondからBoxへ到達できるため、そのオブジェクトは回収候補になりません。出力は10です。

関連する用語との違い

混同しやすいもの違い見分け方
変数 / オブジェクト参照型変数は参照を持つ。オブジェクト本体は別にある。newされたものと変数を分ける。
参照あり / 参照なしどこかの変数などから到達できるかどうか。代入やnullで参照が外れたか見る。
GC対象 / 即回収対象になっても即座に回収されるとは限らない。「対象」と「実行タイミング」を分ける。

参照とオブジェクトを追う規則

  1. メソッド呼び出しごとにローカル変数の範囲を分ける。
  2. newで生成されるオブジェクトを別に記録する。
  3. 各変数がどのオブジェクトを参照しているか線で結ぶ。
  4. 代入やnull代入で参照の線がどう変わるか追う。
  5. どのルートからも到達できないオブジェクトだけを回収候補として考える。

確認用コード

Object a = new Object();
Object b = new Object();
Object c = a;
a = null;
b = null;
見る点このコードでの判断結論
1つ目のObjectcがまだ参照しているGC対象ではない
2つ目のObjectbがnullになり参照なしGC対象
結論対象は2つ目だけただし回収タイミングは未定

学習用のメモリ図として使う

スタックとヒープの説明は、値と参照を追うための学習用モデルです。実際のJVMは最適化を行いますが、基礎ではローカル変数や呼び出し情報と、newされたオブジェクトを分けて考えるとコードを追いやすくなります。

領域置かれるもののイメージ読解での使い方
スタックメソッド呼び出しごとのローカル変数今どのメソッドの変数を見ているかを追う
ヒープnewされたオブジェクト複数の変数が同じオブジェクトを参照する可能性を見る
参照オブジェクトへのリンク情報代入で参照がコピーされることを見る
int[] a = {1, 2, 3};
int[] b = a;
b[0] = 9;
System.out.println(a[0]); // 9

b = aで配列が複製されたわけではありません。aとbが同じ配列オブジェクトを参照するようになっただけです。だからb[0]を書き換えると、a[0]を見ても同じ変更が見えます。

ここで整理する誤解

参照型の代入を「オブジェクトのコピー」だけでは判断できません。コピーされるのは参照であり、参照先が同じなら変更の影響も共有されます。

定義と規則の整理

メモリ理解で混ざる点

つまずき正しい考え方
変数にオブジェクト本体が入ると思う参照型変数には参照値が入る。学習上は、参照先のオブジェクトをヒープ側として追う。
null代入で即メモリ解放されると思う参照が切れるとGC対象になり得るが、回収タイミングはJVMが決める。
スタックとヒープを暗記語にするローカル変数や呼び出し情報は主にスタック、オブジェクトは主にヒープで考える。

確認できれば次へ

  • スタックとヒープの役割をざっくり説明できる
  • 参照が切れるとは何か説明できる
  • ガベージコレクション対象の条件を説明できる
理解の基準

成立するコードと成立しないコード

成立する例：参照先のある変数を使う

Box first = new Box();
Box second = first;
second.value = 10;

成立しない例：null参照のままフィールドへアクセスしている

Box b = null;
b.value = 10;

成立する形：インスタンスを生成してからアクセスする

Box b = new Box();
b.value = 10;
確認ポイント

参照がnullのままフィールドやメソッドへアクセスするとNullPointerExceptionです。

定義と規則を確認する

用語の確認

スタック、ヒープ、ガベージコレクションを説明できますか。

説明を見る

スタックはメソッド呼び出しやローカル変数の管理、ヒープはnewされたオブジェクトの置き場所、ガベージコレクションは参照されなくなったオブジェクトの回収です。

コードの確認

次のあと、最初に作ったObjectは参照されていますか。

Object a = new Object();
a = new Object();
説明を見る

参照されていません。変数aは新しいObjectを参照するため、最初のObjectへの参照は外れます。

仕組みを説明する

System.gc() を呼んでも即時回収が保証されない理由を説明してください。

説明を見る

System.gc() はJVMへガベージコレクションを促すだけです。実際にいつ回収するかはJVMが決めます。

基本ルールのまとめ

  • 変数とオブジェクト本体を分けて考えられる。
  • 参照型変数の代入で参照がコピーされると説明できる。
  • GC対象になることと、直ちに回収されることを区別できる。

コードに規則を当てはめる

次のコードで、最後にGC対象になる可能性があるのはどちらのオブジェクトか。

確認用コード

Object x = new Object(); // 1つ目
Object y = new Object(); // 2つ目
x = y;
y = null;
説明を見る

1つ目のオブジェクトがGC対象になる。

x = yでxは2つ目のオブジェクトを参照するようになる。この時点で1つ目への参照がなくなる。y = null後もxが2つ目を参照しているため、2つ目は対象ではない。
