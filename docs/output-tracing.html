出力追跡トレーニング

出力問題を感覚で解かず、変数表・参照表・制御フロー表で正確に追うための練習ページです。

出力問題は状態表を作って追う

出力追跡は「なんとなく頭で読む」単元ではありません。試験では、1文字違い・1行違い・例外発生位置の違いで選択肢が分かれます。紙やメモに 変数表・参照表・制御フロー表 を作って、実行順に潰してください。

  1. まずコンパイル可否: 文法、型、アクセス、例外処理、到達不能文を先に見る。
  2. 次に実行時例外: null、配列/List範囲外、キャスト、parse、算術例外を探す。
  3. 最後に出力: 評価順、参照の共有、メソッド呼び出し、finallyを表にする。
重要: コンパイルエラーのコードは実行されません。例外が発生するコードは、例外発生前の出力だけ残ります。この2つを混ぜると、出力問題でほぼ確実に落とします。

出力追跡の基本手順

順番見るもの切れる選択肢
1クラス・メソッド・変数宣言・型コンパイルエラー
2実行開始位置、起動引数、初期値mainに入れるか、argsで落ちないか
3式の評価順数値結果、文字列連結結果
4参照の向きとオブジェクトの中身同じ実体を見ているか、別物か
5分岐・ループ・例外処理表示される/されない行
6println / print / return / throw改行、途中終了、finally実行

選択肢を見ながら解くと引っ張られます。まず自分で「コンパイル可否」「例外発生位置」「出力文字列」を確定し、その後で選択肢に当てはめる方が安定します。

出力追跡を表にする

複雑な式を頭の中だけで処理せず、「実行行」「式が返す値」「変数の更新後値」「出力」の4列を作ります。特にインクリメント、メソッド呼び出し、可変オブジェクト、例外を含む問題で有効です。

実行行式の値状態変化出力
x++更新前の値xを1増やすまだなし
++x更新後の値先にxを1増やすまだなし
println(...)戻り値なし引数式の副作用は完了文字列と改行
代入と後置++

出力を判定してください。

int x = 1;
      x = x++;
      System.out.println(x);
代入後の値：1です。後置++で一度2になりますが、式の値1を同じ変数へ代入して上書きします。

変数表の作り方

数値・boolean・String結合の問題は、行番号ごとに変数の値を書きます。式の途中で値が変わる ++/-- は、1つの式をさらに分解してください。

int a = 10;
int b = a++ + ++a + a--;
System.out.println(a + ":" + b);
処理aの状態式に使う値説明
開始10-初期値
a++1110後置。先に10を使い、その後11になる
++a1212前置。先に12にしてから使う
a--1112後置。12を使った後、11になる
代入1134b = 10 + 12 + 12

出力は 11:34 です。式を左から読むだけでは不十分です。前置/後置で「値を使うタイミング」が違います。

String結合の追跡

+ は左から評価されます。片方がStringになると、以降は文字列結合として進みます。

int a = 10;
int b = 20;
System.out.println(a + b + "A" + a + b);
System.out.println("A" + a + b + (a + b));
式評価出力
a + b + "A" + a + b30 + "A" + 10 + 2030A1020
"A" + a + b + (a + b)"A10" + 20 + 30A102030
判断: 文字列結合で迷ったら、左から順に「今の結果の型は数値かStringか」を書いてください。括弧の中だけは先に評価します。

参照表の作り方

参照型の問題では、変数の中にオブジェクトそのものが入るわけではありません。変数はオブジェクトへの参照を持ちます。代入はオブジェクトのコピーではなく、参照のコピーです。

class Item { String name; }

Item a = new Item();
Item b = new Item();
a.name = "apple";
b.name = "banana";
a = b;
a.name = "cherry";
System.out.println(b.name);
行abItem#1Item#2
a = new Item()Item#1-name=null-
b = new Item()Item#1Item#2name=nullname=null
a.name = "apple"Item#1Item#2applenull
b.name = "banana"Item#1Item#2applebanana
a = bItem#2Item#2applebanana
a.name = "cherry"Item#2Item#2applecherry

出力は cherry です。a = b 後は、aとbが同じItem#2を見ています。

参照表で可変・不変を分ける

String s1 = "A";
      String s2 = s1;
      s1 += "B"; // s1だけ別Stringへ
      
      StringBuilder b1 = new StringBuilder("A");
      StringBuilder b2 = b1;
      b1.append("B"); // 同じオブジェクトがAB

変数ごとに「どのオブジェクト番号を指すか」を記録します。Stringメソッドは新しい参照を返し、StringBuilderメソッドは同じオブジェクトを変更することが多い点を反映します。

StringBuilderは「中身が変わる」

String は不変ですが、StringBuilder は可変です。同じインスタンスを複数変数で参照していると、片方からの変更がもう片方にも見えます。

StringBuilder a = new StringBuilder("A");
StringBuilder b = a;
a.append("B");
b.append("C");
System.out.println(a);
処理ab中身
生成SB#1-A
b = aSB#1SB#1A
a.append("B")SB#1SB#1AB
b.append("C")SB#1SB#1ABC

出力は ABC です。append は新しいStringBuilderを返しているのではなく、基本的に自分自身を変更します。

メソッド呼び出しの追跡

Javaは常に値渡しです。参照型引数でも、渡されるのは「参照値のコピー」です。メソッド内でオブジェクトの中身を変えることと、仮引数へ別の参照を代入することは別です。

static void test(StringBuilder sb) {
    sb.append("B");
    sb = new StringBuilder("C");
    sb.append("D");
}

StringBuilder x = new StringBuilder("A");
test(x);
System.out.println(x);
処理xsb見える中身
呼び出し直後SB#1SB#1SB#1 = A
sb.append("B")SB#1SB#1SB#1 = AB
sb = new StringBuilder("C")SB#1SB#2SB#1 = AB / SB#2 = C
sb.append("D")SB#1SB#2SB#1 = AB / SB#2 = CD

呼び出し元の x はSB#1のままなので、出力は AB です。

メソッド呼び出しは引数評価から追う

static int f(int x) {
          System.out.print("F" + x);
          return x * 2;
      }
      int a = 1;
      System.out.println(f(a++) + f(a));
      // 左の引数評価から順に追う

メソッド本体へ入る前に、対象式と引数式が評価されます。複数引数は左から評価されるため、副作用が後続引数へ影響します。

引数評価順

m(i++, ++i)をi=1で呼ぶと、引数値は何ですか。

左からの評価結果：左から評価され、第一引数は1、その後i=2、第二引数でi=3となり値3です。

フィールドとメソッド呼び出しの違い

継承問題では、フィールドは宣言型、オーバーライドされたインスタンスメソッドは実体 で決まります。ここは混同しやすいので表で分けます。

class A {
    String name = "A";
    String getName() { return name; }
}
class B extends A {
    String name = "B";
    @Override String getName() { return name; }
}

A obj = new B();
System.out.println(obj.name + ":" + obj.getName());
式決まり方結果
obj.name変数の宣言型 A のフィールドA
obj.getName()実体 B のオーバーライドメソッドB

出力は A:B です。メソッドとフィールドを同じルールで見てはいけません。

例外処理の出力表

例外が発生したら、tryブロックの残りは実行されません。対応するcatchへ移り、その後finallyがあれば実行します。

try {
    System.out.print("A");
    String s = null;
    System.out.print(s.length());
    System.out.print("B");
} catch (NullPointerException e) {
    System.out.print("C");
} finally {
    System.out.print("D");
}
System.out.print("E");
処理出力制御
print("A")A続行
s.length()-NullPointerException 発生
print("B")-実行されない
catchCfinallyへ
finallyDtry文の後へ
後続処理E終了

出力は ACDE です。catchで処理された例外は、通常はその後の処理へ進みます。

初期化順序と例外を出力列へ含める

static初期化、インスタンス初期化、コンストラクタ、finallyは、通常の上から下の流れへ追加される処理です。出力がある場合は、発生時点で表へ記録します。例外発生後の未実行行は出力しません。

finally出力

tryで例外が発生しcatchされなくてもfinally内の出力は行われますか。

finally後の状態：finallyが存在し、VM終了など特殊状況でなければ実行され、その後例外が伝播します。

try-with-resourcesの追跡

リソース付きtryでは、リソースの close() は宣言と逆順に呼ばれます。finallyがある場合、closeの後にfinallyです。

class R implements AutoCloseable {
    private String name;
    R(String name) { this.name = name; }
    public void close() { System.out.print(name); }
}

try (R a = new R("A"); R b = new R("B")) {
    System.out.print("C");
} finally {
    System.out.print("D");
}
順番処理出力
1try本体C
2b.close()B
3a.close()A
4finallyD

出力は CBAD です。宣言順ではなく、逆順です。

ループ追跡の型

ループは「初期化 → 条件 → 本体 → 更新」を表にします。continue がある場合、通常forでは更新式へ進む点に注意します。

int sum = 0;
for (int i = 0; i < 4; i++) {
    if (i == 2) continue;
    sum += i;
}
System.out.println(sum);
i条件本体sum更新
0truesum += 00i=1
1truesum += 11i=2
2truecontinue1i=3
3truesum += 34i=4
4false終了4-

出力は 4 です。

出力順を崩す原因

コンパイルエラーを見落とす

未初期化ローカル変数、戻り値不足、到達不能catch、型不一致があると、出力追跡以前に終了です。

参照代入をコピーだと思う

a = b はオブジェクト複製ではありません。同じ参照先になるだけです。

後置++を先に増やす

a++ は「式には元の値を使い、その後増える」です。値の使用と変数更新を分けます。

例外後のtry残りを実行する

例外発生後、同じtryブロックの残りは実行されません。catch/finallyへ移ります。

値・参照・例外を一行ずつ記録する

Q1. String s = null; System.out.println("A" + s); は例外？

例外ではありません。文字列結合では null が文字列 "null" として扱われ、Anull と表示されます。s.length() のようにメソッドを呼ぶと例外です。

Q2. System.out.print("A"); throw new RuntimeException(); の後に例外が出たらAは消える？

消えません。例外発生前の出力は残ります。ただし、その後にcatchされなければプログラムは異常終了します。

Q3. A obj = new B(); obj.field と obj.method() は同じ決まり方？

違います。フィールドは宣言型、オーバーライドされたインスタンスメソッドは実体で決まります。
