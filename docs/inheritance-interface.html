継承問題をメンバー種別へ分解する

  • abstractクラスとinterfaceの実装義務
  • sealed / non-sealed / permits
  • protectedと別パッケージのアクセス
  • フィールド解決とメソッド解決の違い

継承・interface問題の全体地図

継承の問題は、型関係、メソッド、フィールド、アクセス制御、抽象性、コンストラクタ、例外、初期化順序を混ぜて出題されます。最初に対象を分類し、その後で各規則を適用します。

対象継承先で起きること選択基準
インスタンスメソッドオーバーライド可能呼べるかは宣言型、実装は実行時型
staticメソッド隠蔽コンパイル時型・型名
フィールド隠蔽コンパイル時型

フィールドとメソッド

出力を判定してください。

class Parent {
          String name = "P";
          String getName() { return "P"; }
      }
      class Child extends Parent {
          String name = "C";
          @Override String getName() { return "C"; }
      }
      Parent x = new Child();
      System.out.println(x.name + " " + x.getName());
選択されるメンバー：P Cです。フィールドは宣言型Parent、インスタンスメソッドは実体Childで選ばれます。

interfaceの暗黙修飾子

interfaceに書いたメンバは、クラスと同じ感覚で読むと間違えます。抽象メソッドとフィールドには暗黙の修飾子があります。

書いた形実質的な意味注意点
void test();public abstract void test();実装側は public 必須
int MAX = 10;public static final int MAX = 10;代入し直せない
default void test(){}実装付きインスタンスメソッドpublic 扱い。private/protected不可
static void test(){}interfaceに属するstaticメソッド実装クラスへ継承されない
private void helper(){}interface内部の補助メソッド外部から呼べない。実装クラスからも呼べない
 
interface A {
    int VALUE = 10;
    void run();
}

class B implements A {
    public void run() {          // publicが必要
        // VALUE = 20;           // NG: static final
        System.out.println(VALUE);
    }
}

defaultメソッドの基本

defaultメソッドは、interfaceに実装を持たせる仕組みです。実装クラスがoverrideしなければ、そのまま使えます。

interface A {
    default void test() {
        System.out.println("A");
    }
}

class B implements A { }

new B().test(); // A

ただし、defaultメソッドは「interfaceの通常メソッド」ではありません。default を付けずに本体を書くとコンパイルエラーです。

interface A {
    // void test() { } // NG: default, static, privateのいずれでもない実装付きメソッド
}

defaultメソッドの衝突

複数のinterfaceから同じシグネチャのdefaultメソッドを受け継ぐと、クラス側でどちらを使うか決めなければいけません。

interface A {
    default void test() { System.out.println("A"); }
}
interface B {
    default void test() { System.out.println("B"); }
}

class C implements A, B {
    public void test() {
        A.super.test();
    }
}

A.super.test() は、interfaceのdefault実装を明示的に呼ぶ構文です。super.test() ではありません。

defaultメソッドの優先順位をアルゴリズム化する

  1. クラス階層に同じシグネチャの具体的インスタンスメソッドがあれば、そのクラス側が優先される。
  2. 複数interfaceに候補がある場合、より具体的なサブinterfaceのdefaultが優先される。
  3. 無関係な複数interfaceのdefaultが残る場合、実装クラスで明示的にオーバーライドする。
  4. 必要ならInterfaceName.super.method()で直接のスーパーinterfaceのdefault実装を呼ぶ。
interface A {
          default void run() { System.out.print("A"); }
      }
      interface B extends A {
          @Override default void run() { System.out.print("B"); }
      }
      class C implements A, B { }
      // new C().run() -> B
Objectメソッド：toString、equals、hashCodeなどObjectの公開インスタンスメソッドと同等のシグネチャを、defaultメソッドとして宣言することはできません。
default競合

無関係な2つのinterfaceから同じdefaultメソッドを継承し、実装クラスが空の場合どうなりますか。

override成立性：コンパイルエラーです。実装クラスでオーバーライドし、必要なら各interfaceのdefaultを明示的に呼びます。
InterfaceName.super

直接実装していない祖先interfaceを飛び越えてA.super.m()と書けますか。

明示呼び出しの制約：原則として書けません。明示的なdefault呼び出しは直接のスーパーinterfaceを基準にします。

クラスの具象メソッドがdefaultより優先される

スーパークラスに具象メソッドがあり、interfaceにも同じシグネチャのdefaultメソッドがある場合、クラス側が優先されます。

class A {
    public void test() {
        System.out.println("class");
    }
}

interface B {
    default void test() {
        System.out.println("interface");
    }
}

class C extends A implements B { }

new C().test(); // class

優先順位は、ざっくり「クラスが勝つ」です。defaultメソッドは、クラス階層に実装がないときの補助と考えると判断しやすくなります。

interfaceのstaticメソッドは継承されない

interfaceのstaticメソッドは、そのinterface名で呼び出します。実装クラス名やインスタンスから呼ぶものではありません。

interface A {
    static void test() {
        System.out.println("A");
    }
}

class B implements A { }

A.test();
// B.test();       // NG
// new B().test(); // NG

クラスのstaticメソッドとは感覚が違います。interface staticは「継承されない」と覚えてください。

interfaceのprivateメソッド

privateメソッドは、defaultメソッドやstaticメソッド内の共通処理を切り出すためのものです。実装クラスから呼ぶためのものではありません。

interface A {
    default void test() {
        log();
    }

    private void log() {
        System.out.println("log");
    }
}

class B implements A {
    void sample() {
        // log(); // NG: interfaceのprivateメソッドは呼べない
    }
}

interfaceのstatic・private・フィールド

宣言実質的な性質継承先からの扱い
void m();public abstract実装側はpublic必須
int X = 1;public static final再代入不可。曖昧ならinterface名で限定
static void m(){}interface固有のstaticメソッド実装クラスへ継承されない
private void helper(){}defaultメソッド等の補助実装クラスから利用不可
private static void helper(){}static処理の補助interface内部だけで利用

interface同士の継承

interfaceはinterfaceを extends で継承します。クラスのように implements は使いません。また、interfaceは複数のinterfaceを継承できます。

interface A {
    void a();
}

interface B {
    void b();
}

interface C extends A, B {
    void c();
}

class D implements C {
    public void a() {}
    public void b() {}
    public void c() {}
}

D は C だけでなく、A と B の抽象メソッドも実装する必要があります。

同名メソッド・戻り値・ジェネリクスの競合

interface A { Number value(); }
      interface B { Integer value(); }
      class C implements A, B {
          public Integer value() { return 1; } // 両方を満たす
      }

戻り値型に共通の互換実装がない場合は両立できません。また、ジェネリック型引数だけが違うメソッドは、型消去後に同じシグネチャとなり名前衝突を起こす場合があります。

class Parent {
          void test(java.util.List<String> list) { }
      }
      class Child extends Parent {
          // void test(java.util.List<Integer> list) { }
          // 型消去後にtest(List)で衝突
      }

コンストラクタ・例外・初期化順序

子クラスのコンストラクタは親コンストラクタを呼ぶ必要があります。親コンストラクタがチェック例外を投げる場合、子コンストラクタも宣言するか、例外を投げない別の親コンストラクタを選びます。Java 17ではsuper(...)より前に通常のtry文を置いて囲むことはできません。

class Parent {
          Parent() throws java.io.IOException { }
      }
      class Child extends Parent {
          Child() throws java.io.IOException {
              super();
          }
      }

初期化は、親static→子static→親インスタンス→親コンストラクタ→子インスタンス→子コンストラクタの大枠で進みます。親コンストラクタからオーバーライド可能メソッドを呼ぶと、子フィールド初期化前に子実装が動くため危険です。

抽象クラスはinterfaceメソッドを未実装のままにできる

interfaceを実装したクラスが抽象クラスなら、すべての抽象メソッドを実装しなくても構いません。ただし、最終的な具象クラスでは実装が必要です。

interface A {
    void test();
}

abstract class B implements A {
    // testを実装しなくてもOK。B自身がabstractだから
}

class C extends B {
    public void test() {
        System.out.println("C");
    }
}

abstractクラスの細部

抽象クラスは直接インスタンス化できませんが、フィールド、コンストラクタ、通常メソッド、staticメソッド、finalメソッド、初期化子を持てます。抽象メソッドだけを持つ仕組みではありません。

abstract class Base {
          private final String name;
          Base(String name) { this.name = name; }
          final String name() { return name; }
          abstract void execute();
      }
      
      abstract class Middle extends Base {
          Middle() { super("M"); }
          // executeを未実装のまま持ち越せる
      }

具象メソッドを中間の抽象クラスで再び抽象化し、さらに下位の具象クラスへ実装を要求することもできます。ただし親メソッドがfinalなら不可能です。

protectedの別パッケージアクセス

protected は「同じパッケージならアクセス可」「別パッケージでも継承関係があれば条件付きでアクセス可」です。別パッケージのサブクラスから、親型の別インスタンス経由で自由に触れるわけではありません。

// package p1;
public class A {
    protected void test() {}
}

// package p2;
class B extends A {
    void ok() {
        test();       // OK: 自分自身の継承メンバとして呼ぶ
        this.test();  // OK
    }

    void ng(A a) {
        // a.test();  // NGになり得る: 別パッケージで親型参照経由
    }
}

protectedは「サブクラスなら何でもOK」ではありません。この誤解はかなり危険です。

sealed階層は独立記事で扱う

sealed型は、通常の継承規則に追加される「直接サブタイプの許可関係」を持ちます。permits、final、sealed、non-sealedを階層図から読む方法は、sealedクラスとsealedインターフェースで扱います。

メソッドとフィールドの解決を分ける

継承/interfaceの詳細問題では、出力追跡と組み合わされます。特に、同名フィールドとoverrideメソッドの混在に注意します。

class A {
    String value = "A";
    String getValue() { return value; }
}

class B extends A {
    String value = "B";
    String getValue() { return value; }
}

A a = new B();
System.out.println(a.value);      // A
System.out.println(a.getValue()); // B

フィールドは宣言型、インスタンスメソッドは実体。ここは機械的に分けてください。

オーバーライドの成立条件を完全に確認する

class Parent {
          protected Number create() throws java.io.IOException {
              return 1;
          }
      }
      class Child extends Parent {
          @Override
          public Integer create() throws java.io.FileNotFoundException {
              return 1;
          }
      }
  1. 親子関係があり、対象がインスタンスメソッドである。
  2. メソッド名と引数リストがオーバーライド可能な関係である。自動ボクシングは引数型一致の代わりにならない。
  3. 戻り値は同じか、参照型なら親戻り値のサブタイプである。
  4. アクセス範囲を狭めていない。
  5. チェック例外を広げていない。
  6. 親メソッドがprivateまたはfinalではない。
@Overrideの役割：オーバーライドする意図をコンパイラへ伝え、引数違いによる偶発的なオーバーロードを検出できます。
例外のoverride

親がthrows IOException、子がthrows Exceptionなら有効ですか。

default競合：無効です。子側がより広いチェック例外を追加しています。FileNotFoundExceptionや例外なしなら可能です。
privateメソッド

子に同名メソッドを書き@Overrideを付けるとどうなりますか。

throws互換性：親メソッドがprivateならオーバーライド対象ではないため、@Overrideでコンパイルエラーです。

defaultメソッドとObjectメソッド

interfaceのdefaultメソッドでは、Object の equals、hashCode、toString などをdefault実装として提供することはできません。すべてのクラスがObjectを継承するため、クラス側のルールと衝突するからです。

interface A {
    // default String toString() { return "A"; } // NG
}

これは細かいですが、interface問題で出されると落としやすい論点です。

継承・interface競合の解決順

  1. 宣言を読む：class / abstract class / interface / sealed interface を分類する。
  2. 継承関係を読む：extends / implements / permits を線でつなぐ。
  3. 未実装メソッドを見る：具象クラスが抽象メソッドを残していないか確認する。
  4. メソッド修飾子を見る：public不足、static/instance不一致、戻り値型、例外を確認する。
  5. 衝突を見る：複数default、クラス具象メソッド、同名フィールドを確認する。
  6. 実行を追う：コンパイルできる場合だけ、宣言型と実体を分けて出力を追う。

メンバー種別ごとに選択規則を適用する

Q1. interfaceの void test(); を実装するとき、void test(){} でよい？

ダメです。interfaceの抽象メソッドは暗黙にpublicなので、実装側は public void test(){} にします。

Q2. 2つのinterfaceに同じdefaultメソッドがある場合、実装クラスは空でよい？

ダメです。どちらを使うか曖昧なので、実装クラス側でoverrideして解決します。

Q3. interfaceのstaticメソッドは実装クラス名で呼べる？

呼べません。InterfaceName.method() の形で呼びます。

Q4. sealedクラスを継承したクラスは、何も修飾子を付けなくてよい？

ダメです。final、sealed、non-sealed のいずれかが必要です。
