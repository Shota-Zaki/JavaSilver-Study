sealedクラスとsealedインターフェース

許可関係を直接の親子単位で読み、permits、final、sealed、non-sealedの役割を階層図から判断します。

sealedは「直接の子」を固定する宣言

sealed階層は、すべての子孫クラスを1枚のリストで固定する仕組みではありません。各sealed型が、自分を直接継承・実装できる型だけを管理します。この「直接」という範囲を先に押さえると、permitsとnon-sealedの働きが明確になります。

sealed interface Result permits Success, Failure {}

record Success(String value) implements Result {}
final class Failure implements Result {}

Resultが直接許可しているのはSuccessとFailureです。Successはレコードなので暗黙的にfinal、Failureは明示的にfinalです。

階層図は矢印1本ずつ検査する

sealed class A permits B, C {}
final class B extends A {}
sealed class C extends A permits D {}
non-sealed class D extends C {}
class E extends D {}
矢印検査する宣言結果
B → AAのpermitsにBがあるか許可
C → AAのpermitsにCがあるか許可
D → CCのpermitsにDがあるか許可
E → DDはnon-sealedか通常の継承へ戻るため許可

AのpermitsにDやEを書く必要はありません。Aが管理するのはBとCへの直接の矢印だけです。

直接サブタイプは次の継承方針を必ず決める

sealed型から許可された直接サブタイプは、階層をそこで閉じるか、さらに限定して続けるか、制限を解除するかを宣言します。

final

その型で継承を終了します。

sealed class Shape permits Circle {}
final class Circle extends Shape {}

sealed

次の直接サブタイプを再び限定します。

sealed class Shape permits Polygon {}
sealed class Polygon extends Shape
        permits Triangle {}

non-sealed

その位置から先を通常の継承へ戻します。

sealed class Shape permits OpenShape {}
non-sealed class OpenShape extends Shape {}
class CustomShape extends OpenShape {}

修飾子なし

通常クラスでは不正です。直接サブタイプは方針を明示します。

sealed class Shape permits Bad {}
class Bad extends Shape {} // エラー

レコードは暗黙的にfinalなので、sealedインターフェースの許可された実装として使用した場合、追加のfinalは書きません。

permitsを省略できる配置と、省略できない配置

許可する直接サブタイプが同じコンパイル単位に宣言されている場合、コンパイラが候補を推論できるため、permitsを省略できます。

sealed class Token {}
final class NumberToken extends Token {}
final class WordToken extends Token {}

別ファイルに分かれるなら、親側でpermitsを明示します。コード断片を読むときは、ファイル境界をコメントやクラス配置から確認します。

さらに、名前付きモジュール内では許可型は同じモジュールに所属する必要があります。名前のないモジュールでは同じパッケージに属する必要があります。単にimportできることと、sealed階層へ参加できることは別です。

sealed interfaceではextendsとimplementsを混同しない

sealed interface Service permits BaseService, LocalService {}

non-sealed interface BaseService extends Service {}
final class LocalService implements Service {}

インターフェースがインターフェースを継承するときはextends、クラスがインターフェースを実装するときはimplementsです。permitsにはクラスとインターフェースの両方を列挙できますが、実際の関係を表すキーワードは通常の型関係と同じです。

コンパイルエラーを親側・子側・配置の3方向から切る

コード上の異常確認場所理由
permitsにない型が直接継承親側許可リスト外
許可型だが親を直接継承していない親子関係permitsは直接サブタイプを列挙する
直接サブタイプに方針修飾子がない子側final / sealed / non-sealedが必要
通常の親に対してnon-sealed子側解除対象となるsealed制限がない
異なるモジュール・不正なパッケージ配置配置許可型の所在制約に反する
class Normal {}
// non-sealed class Child extends Normal {} // コンパイルエラー

sealedはポリモーフィズムを禁止しない

sealedは利用できる実体を限定しますが、親型変数によるポリモーフィズムは通常どおり働きます。

sealed interface Command permits Start, Stop {
    void execute();
}
record Start() implements Command {
    public void execute() { System.out.println("start"); }
}
record Stop() implements Command {
    public void execute() { System.out.println("stop"); }
}

Command command = new Start();
command.execute(); // start

宣言型はCommand、実行時型はStartです。sealedで変わるのは、どの型が階層へ入れるかであり、オーバーライドされたメソッドの実行規則ではありません。

設問では階層全体を一気に見ない

  1. sealedと宣言された親型を1つ選ぶ。
  2. その親を直接継承・実装する型だけを拾う。
  3. permitsへの登録または同一コンパイル単位での推論を確認する。
  4. 各直接サブタイプがfinal・sealed・non-sealedのどれかを確認する。
  5. sealedが続く型について同じ確認を繰り返す。

この読み方なら、孫クラスを祖父のpermitsと比較する誤りを防げます。
