Java Silver Reference

型・変数・リテラル

プリミティブ型、参照型、初期値、識別子、null、参照コピーを、Java Silverでそのまま使える判定手順まで掘り下げます。

この単元で最初に固定する結論

型・変数・リテラルの問題は、暗記だけで解くと崩れます。毎回、次の順番で機械的に判定します。

  1. 変数の型を見る。プリミティブ型か参照型か。
  2. 右辺の式の型を見る。リテラル、演算式、メソッド戻り値、null のどれか。
  3. 代入・比較・メソッド呼び出しが可能かを見る。
  4. 最後に、問題が聞いているのが コンパイルエラー、実行時例外、表示結果 のどれかを分ける。
雑に読むと落ちる点: 「値としては入りそう」ではなく「その式の型として代入できるか」で判断します。Java Silverではここを狙われます。

型とは「値の扱い方」を決めるルール

Javaの変数は、必ず型を持ちます。型は単なる入れ物の名前ではなく、次の3つを決めます。

保持できるもの

int は整数値、boolean は true/false、参照型はオブジェクトへの参照を保持します。

できる操作

int は算術演算できますが、boolean は数値演算できません。参照型はメソッド呼び出しできます。

代入できる範囲

型が違えば、値が似ていても代入できない場合があります。暗黙変換できるかが判定軸です。

試験では「同じ値に見えるが型が違う」コードがよく出ます。例として 1、1L、1.0、'1'、"1" はすべて意味が違います。

プリミティブ型と参照型の根本差

分類代表例変数が直接持つものnull比較の注意
プリミティブ型byte short int long float double char boolean値そのもの不可== は値比較
参照型String、配列、自作クラス、ラッパークラス、Objectオブジェクトへの参照可== は参照先の同一性比較
 
int a = 10;
String s = "java";
Object o = new Object();

// a は 10 という値そのものを持つ
// s と o は、オブジェクトそのものではなく参照を持つ

参照型変数は「オブジェクトの実体」ではなく「オブジェクトへ到達するための参照」を持ちます。この理解がないと、==、ガベージコレクション、null、メソッド引数の問題で誤答します。

8つのプリミティブ型を試験用に整理

全部を均等に覚える必要はありません。Java Silverで特に問われるのは byte、short、char、int、long、float、double、boolean の互換性です。

booleanは数値ではない

C言語系の感覚で 0 を false、1 を true のように扱うとJavaでは即アウトです。

boolean a = true;
boolean b = false;

// boolean c = 1;    // NG
// int x = true;     // NG
// if (1) { }        // NG

if (a == true) {     // OK。ただし実務では if (a) が自然
    System.out.println("ok");
}

試験では if、while、三項演算子の条件部に数値を置くひっかけが出ます。条件部は必ず boolean です。

変数の宣言・初期化・代入を分ける

宣言、初期化、代入は似ていますが、試験では分けて読む必要があります。

int a;        // 宣言。ローカル変数ならまだ使えない
int b = 10;   // 宣言 + 初期化
b = 20;       // 代入
操作意味試験での注意
宣言変数名と型を決めるローカル変数は宣言だけでは使えない。
初期化最初の値を入れる使用前に必須。
代入既存変数に値を入れ直すfinal 変数は再代入不可。

フィールドとローカル変数の初期値

フィールドにはデフォルト値があります。ローカル変数にはありません。ここは非常に落としやすいです。

変数の種類宣言場所自動初期化例
フィールドクラス直下されるint は 0、boolean は false、参照型は null
ローカル変数メソッド・コンストラクタ・ブロック内されない使う前に全経路で初期化が必要
 
class Sample {
    int field;          // 0
    boolean flag;       // false
    String text;        // null

    void test() {
        int local;
        // System.out.println(local); // NG: 初期化されていない
        System.out.println(field);    // OK: 0
    }
}

確定代入: 分岐があるローカル変数

ローカル変数は「実行時にたまたま初期化されそう」では足りません。コンパイラが全経路で初期化済みだと判断できる必要があります。

void test(boolean condition) {
    int x;
    if (condition) {
        x = 10;
    }
    // System.out.println(x); // NG: condition が false の経路で未初期化
}
void test(boolean condition) {
    int x;
    if (condition) {
        x = 10;
    } else {
        x = 20;
    }
    System.out.println(x); // OK: どちらの経路でも初期化される
}
判断: ローカル変数の問題は「実行したらどうなるか」より前に、コンパイラが初期化済みと証明できるかを見ます。

参照型変数とnull

null は「参照先がない」ことを表す特別なリテラルです。参照型には代入できますが、プリミティブ型には代入できません。

String s = null;   // OK
Object o = null;   // OK
int[] array = null;// OK

// int n = null;   // NG
// boolean b = null; // NG

null を出力するだけなら "null" と表示されますが、メソッドを呼ぶと NullPointerException です。

String s = null;
System.out.println(s);  // null
// System.out.println(s.length()); // 実行時にNullPointerException

nullと空文字は完全に別物

値意味メソッド呼び出し表示
null参照先がない不可。呼ぶと NullPointerExceptionprintln では null と出る
""長さ0のStringオブジェクト可見た目は何も出ない
 
String a = null;
String b = "";

System.out.println(a);        // null
System.out.println(b);        // 空文字なので見た目は空
System.out.println(b.length());// 0
// System.out.println(a.length()); // NullPointerException

参照のコピー: オブジェクトはコピーされない

参照型変数を別の変数に代入しても、オブジェクト本体が複製されるわけではありません。参照がコピーされます。

StringBuilder a = new StringBuilder("A");
StringBuilder b = a;
b.append("B");
System.out.println(a); // AB

a と b は同じ StringBuilder を参照しています。これは配列、自作クラス、ArrayList でも同じです。

識別子: 変数名・クラス名に使えるもの

識別子は「名前」です。英字、数字、_、$ を使えますが、数字始まりは不可です。予約語も使えません。

例可否理由
int a1 = 10;OK数字始まりではない。
int $price = 100;OK$ は使える。ただし実務では避ける。
int _value = 1;OK_ を含む名前は可。
int 1st = 1;NG数字始まり。
int class = 1;NGclass は予約語。
int _ = 1;NG_ 単独は識別子として使えない。

予約語・文脈キーワードの読み方

予約語は識別子にできません。さらにJava 17では var、record、sealed、permits、non-sealed など、文脈によって特別な意味を持つ語があります。

// int class = 1;   // NG: 予約語
int var = 10;       // OK: var は予約語ではない
var value = 20;     // OK: ここではローカル変数型推論

// record は record 宣言では特別な意味を持つ
record Point(int x, int y) {}

var は「どこでも型推論できる魔法」ではありません。ローカル変数の宣言で、初期化式がある場合にだけ使うのが基本です。

charは文字だが、数値的にも扱われる

char は1文字を表しますが、内部的にはUnicodeコード値を持ちます。そのため、範囲内の整数定数を代入できる場合があります。

char a = 'A';   // OK
char b = 65;    // OK。'A' 相当
// char c = -1; // NG。char の範囲外

char x = 'A';
// char y = x + 1; // NG: x + 1 は int
char z = (char)(x + 1); // OK
注意: 'A' は char、"A" は String です。シングルクォートとダブルクォートの違いは頻出です。

配列も参照型

配列変数も参照型です。配列そのものではなく、配列オブジェクトへの参照を持ちます。

int[] a = {1, 2, 3};
int[] b = a;
b[0] = 99;
System.out.println(a[0]); // 99

a と b は同じ配列オブジェクトを参照します。配列問題で == が出たら、要素比較ではなく参照比較です。

ラッパークラス: プリミティブに見えて参照型

Integer、Long、Double、Boolean などはラッパークラスです。プリミティブ型に対応する参照型です。

Integer a = 10;  // オートボクシング
int b = a;       // アンボクシング

Integer n = null;
// int x = n;    // 実行時にNullPointerException

ラッパークラスは参照型なので null を持てます。しかし、int などに戻すアンボクシング時に null だと実行時例外になります。

よくある誤答パターン

値だけで判断する

byte b = 1 + 2; はOKですが、byte b = x + 2; は x が変数ならNGになりやすいです。式の型を見てください。

nullを文字列だと思う

null は文字列 "null" ではありません。参照先がない状態です。

参照コピーを値コピーと誤解する

参照型の代入ではオブジェクト本体は複製されません。同じオブジェクトを共有する場合があります。

ローカル変数に初期値があると思う

フィールドには初期値がありますが、ローカル変数にはありません。

ミニ演習

Q1. Object obj = null; System.out.println(obj); の結果は？

null と表示されます。println に null 参照を渡すこと自体は可能です。ただし obj.toString() を明示的に呼ぶと NullPointerException です。

Q2. char c = 65; はコンパイルできる？

できます。65 は char の範囲内の整数定数です。ただし int x = 65; char c = x; は暗黙には代入できません。

Q3. Integer n = null; int x = n; は？

コンパイルは通りますが、実行時に NullPointerException です。Integer から int へアンボクシングしようとして、参照先がないためです。

Q4. String a = ""; System.out.println(a.length()); は？

0 と表示されます。空文字は長さ0の String オブジェクトです。null とは違います。

  • null を代入できる型とできない型を即答できる。
  • フィールドとローカル変数の初期値の違いを判断できる。
  • "A" と 'A' の違いを説明できる。
  • 参照型変数の代入が「参照のコピー」であることを追跡できる。
  • 識別子として使える名前・使えない名前を判定できる。
