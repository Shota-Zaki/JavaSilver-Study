ひっかけ集

Java Silverで選択肢を誤らせる典型パターンを、コンパイル可否・実行時例外・出力追跡の順に整理します。

ひっかけ問題の正体

Java Silverのひっかけは、知識量ではなく判定順の崩れを狙っています。コードを見た瞬間に出力を追い始める人は落とされます。まずコンパイル可否、次に実行時例外、最後に出力です。

1コンパイルエラー候補を潰す

宣言、型、アクセス、override、例外処理、到達不能文。

2実行時例外候補を探す

null、境界外、キャスト、parse、変更不可List。

3初めて出力を追う

評価順、参照、動的束縛、finally、close順。

罠2：コンパイルできるが実行時例外

コンパイルできるコードでも、実行中に該当行へ到達した瞬間に止まることがあります。例外発生後の通常処理は実行されません。

IndexOutOfBoundsExceptionlist.get(0)、str.substring(5)ListやStringの範囲外。

罠3：実行できるが出力を誤る

コンパイルも実行時例外も問題ない場合だけ、出力を追います。ここからは「値」ではなく「評価順」と「参照」を見る段階です。

Stringは不変

String s = "abc";
s.toUpperCase();
System.out.println(s); // abc

戻り値を受け取らない限り、元のStringは変わりません。

StringBuilderは可変

StringBuilder sb = new StringBuilder("A");
sb.append("B");
System.out.println(sb); // AB

同じインスタンスの中身が変わります。

後置インクリメント

int a = 10;
int b = a++ + a;
System.out.println(b); // 21

後置は古い値を式に使い、その後で変数を更新します。

短絡評価

int a = 0;
if (false && ++a > 0) {}
System.out.println(a); // 0

&& は左がfalseなら右を評価しません。

罠4：参照型の代入を「コピー」と誤解する

参照型変数に入っているのは、インスタンスそのものではなく参照です。代入すると参照がコピーされ、同じインスタンスを見る変数が増えます。

class Box {
    int value;
}

public class Main {
    public static void main(String[] args) {
        Box a = new Box();
        Box b = a;
        b.value = 10;
        System.out.println(a.value);
    }
}

出力は 10 です。a と b は同じBoxインスタンスを参照しています。

見抜き方

new が何回出たかを数えます。new が1回なら、原則としてインスタンスは1個です。参照変数が複数あっても、インスタンスが増えるわけではありません。

罠5：メソッドは実体、フィールドは宣言型

ポリモーフィズム問題では、メソッドとフィールドを同じルールで見てはいけません。

class A {
    String name = "A";
    String getName() { return "A"; }
}
class B extends A {
    String name = "B";
    String getName() { return "B"; }
}
public class Main {
    public static void main(String[] args) {
        A obj = new B();
        System.out.println(obj.name);
        System.out.println(obj.getName());
    }
}
式結果理由
obj.nameAフィールドはoverrideされず、宣言型 A で決まる。
obj.getName()Boverrideされたインスタンスメソッドは実体 B で決まる。

罠6：overloadとoverrideの判定タイミングを混ぜる

override は実体で決まりますが、overload はコンパイル時の引数型で決まります。

class A {}
class B extends A {}

public class Main {
    static void test(A a) { System.out.println("A"); }
    static void test(B b) { System.out.println("B"); }

    public static void main(String[] args) {
        A obj = new B();
        test(obj);
    }
}

出力は A です。実体は B でも、変数 obj の宣言型は A なので、test(A) が選ばれます。

罠7：try-catch-finallyの出力順

例外処理の出力問題は、例外が出た時点でtry内の残りが飛ばされること、そしてfinallyが最後に実行されることを表で追います。

try {
    System.out.print("A");
    String s = null;
    s.length();
    System.out.print("B");
} catch (NullPointerException e) {
    System.out.print("C");
} finally {
    System.out.print("D");
}
順番処理出力
1System.out.print("A")A
2s.length() で例外なし
3catchへ移動C
4finallyD

出力は ACD です。B は実行されません。

罠8：try-with-resourcesのclose順

try-with-resourcesでは、リソースは宣言した順と逆順にcloseされます。さらに、closeはfinallyより前です。

try (Resource a = new Resource("A");
     Resource b = new Resource("B")) {
    System.out.print("T");
} finally {
    System.out.print("F");
}

closeが名前を出力するなら、順番は TBAF です。A から閉じるのではなく、後から作った B から閉じます。

罠9：選択肢の文言で誤る

選択肢の言い方注意点
「実行時に例外がスローされる」コンパイルエラーが先なら実行されない。実行時例外を選ばない。
「何も表示されない」正常終了で何も出ないのか、コンパイルエラーなのかを分ける。
「nullと表示される」println(null参照) は表示、null参照.メソッド() は例外。
「継承しているから使える」private とコンストラクタは引き継がない。
「同じ内容だからtrue」== なら参照比較。equals なら実装確認。

模擬問題前の10秒チェック

  1. 出力を追う前に、宣言と型を見たか。 ここを飛ばすとコンパイルエラー問題で落ちます。
  2. 例外が出る行に到達するか確認したか。 到達しない例外は発生しません。
  3. メソッドとフィールドの解決規則を分けたか。 メソッドは実体、フィールドは宣言型です。
  4. StringとStringBuilderを分けたか。 不変か可変かで結果が変わります。
  5. checked例外を処理しているか。 未処理ならコンパイルエラーです。

ミニ演習

Q1. String s = null; System.out.println(s); は例外？

例外ではありません。null と表示されます。s.length() なら NullPointerException です。

Q2. byte b = 1; b += 1; はコンパイルできる？

できます。複合代入には暗黙のキャストが含まれます。ただし b = b + 1 は int になるため不可です。

Q3. A a = new B(); test(a); のoverloadは実体Bで選ぶ？

選びません。overloadはコンパイル時の引数型で選びます。宣言型が A なら test(A) が優先されます。

Q4. try-with-resourcesで A、B の順に宣言したらcloseは？

B、A の順です。宣言と逆順です。
