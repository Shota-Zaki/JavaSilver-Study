Java Silver Study

Equality

同一性比較・同値比較

宣言の仕方で == の結果が変わる論点を、例題形式で整理します。

まず区別すること

用語見るもの使う演算/メソッド典型例
同一性比較同じインスタンスを参照しているか==a == b
同値比較値・内容が同じとみなせるかequals()a.equals(b)
プリミティブ比較値そのものが同じか==int a = 10; a == 10
  • 参照型の変数には、オブジェクト本体ではなく参照が入る。
  • == は参照型では原則「同じ場所を指しているか」を見る。
  • equals() はクラス側の実装次第。Stringは内容比較、StringBuilderは基本的に同一性比較のまま。

Stringリテラル同士

String a = "java";
String b = "java";
System.out.println(a == b);      // true
System.out.println(a.equals(b)); // true

同じ文字列リテラルは文字列プールの同じインスタンスを参照します。そのため == も equals も true です。

試験の見方：文字列リテラルだけで作られているなら、同じ内容のリテラルは同じ参照になりやすい。

new Stringを使った場合

String a = "java";
String b = new String("java");
System.out.println(a == b);      // false
System.out.println(a.equals(b)); // true

new String("java") は新しいStringインスタンスを作ります。内容は同じでも参照先が違うため、== は false です。

  • ==：参照が違うので false
  • equals：文字列内容が同じなので true

参照を代入した場合

String a = new String("java");
String b = a;
System.out.println(a == b);      // true
System.out.println(a.equals(b)); // true

b = a はStringを複製していません。参照をコピーしているだけです。したがって、aとbは同じインスタンスを指します。

試験の見方：new が1回だけで、その参照を別変数へ代入しているなら == は true になりやすい。

コンパイル時定数の連結

String a = "ja" + "va";
String b = "java";
System.out.println(a == b);      // true
System.out.println(a.equals(b)); // true

リテラル同士の連結はコンパイル時に "java" として扱われます。結果として文字列プールの同じ参照になります。

変数を使った連結

String part = "ja";
String a = part + "va";
String b = "java";
System.out.println(a == b);      // false
System.out.println(a.equals(b)); // true

変数を含む文字列連結は実行時に新しい文字列を作るため、リテラル "java" とは別参照になります。

落とし穴：見た目は "ja" + "va" と似ていても、変数が混ざると == の結果が変わります。

final変数とコンパイル時定数

final String part = "ja";
String a = part + "va";
String b = "java";
System.out.println(a == b);      // true
System.out.println(a.equals(b)); // true

final で、かつコンパイル時に値が確定するString変数は定数として扱われます。そのため part + "va" はコンパイル時に "java" になります。

final String part = new String("ja");
String a = part + "va";
String b = "java";
System.out.println(a == b);      // false
System.out.println(a.equals(b)); // true

同じ final でも、new String("ja") はコンパイル時定数ではありません。ここは雑に「finalならtrue」と覚えると間違えます。

internを使った場合

String a = new String("java");
String b = a.intern();
String c = "java";
System.out.println(a == c); // false
System.out.println(b == c); // true

intern() は、文字列プール上の同じ内容のStringへの参照を返します。newで作ったa自体がプールへ移るわけではありません。

  • a：ヒープ上のnew String
  • b：文字列プール上のString
  • c：文字列プール上のString

StringBuilderのequals

StringBuilder a = new StringBuilder("java");
StringBuilder b = new StringBuilder("java");
System.out.println(a == b);      // false
System.out.println(a.equals(b)); // false

StringBuilder はStringと違い、内容比較用に equals をオーバーライドしていません。したがって、別インスタンスなら中身が同じでも equals は false です。

StringBuilder sb = new StringBuilder("java");
String s = "java";
System.out.println(sb.equals(s));          // false
System.out.println(sb.toString().equals(s)); // true

StringBuilderの内容をStringと比較したいなら、toString() してからString同士で比較します。

配列の比較

int[] a = {1, 2};
int[] b = {1, 2};
int[] c = a;
System.out.println(a == b);      // false
System.out.println(a.equals(b)); // false
System.out.println(a == c);      // true

配列の equals は要素比較ではありません。基本的に参照の同一性を見ると考えます。

試験の見方：配列の中身が同じでも、別々に new された配列は == も equals も false。

ArrayList/Listの比較

ArrayList<String> a = new ArrayList<>();
a.add("java");
ArrayList<String> b = new ArrayList<>();
b.add("java");
System.out.println(a == b);      // false
System.out.println(a.equals(b)); // true

ArrayList の equals は、要素数と各要素の等価性を比較します。StringBuilderや配列とは違います。

List<String> a = List.of("java");
List<String> b = List.of("java");
System.out.println(a == b);      // 実装に依存させて考えない
System.out.println(a.equals(b)); // true

List.of の戻り値もリストです。内容比較は equals を使います。== で同一性を判定しない方が安全です。

自作クラス・recordの比較

class Item {
  int id;
  Item(int id) { this.id = id; }
}
Item a = new Item(1);
Item b = new Item(1);
System.out.println(a == b);      // false
System.out.println(a.equals(b)); // false

通常の自作クラスは、equals をオーバーライドしなければObject由来の同一性比較です。フィールド値が同じでも別インスタンスなら false です。

record Item(int id) {}
Item a = new Item(1);
Item b = new Item(1);
System.out.println(a == b);      // false
System.out.println(a.equals(b)); // true

record は成分に基づく equals が自動生成されます。Java Silverではrecordも出題対象になり得るため、通常クラスとの違いを押さえます。

ラッパークラスとオートボクシング

Integer a = 100;
Integer b = 100;
System.out.println(a == b);      // true
System.out.println(a.equals(b)); // true
Integer a = 128;
Integer b = 128;
System.out.println(a == b);      // false と考える
System.out.println(a.equals(b)); // true

Integer などのラッパークラスでは、一定範囲の値がキャッシュされるため == がtrueになる場合があります。Silver対策では、ラッパー同士の値比較は equals を使う、と整理するのが安全です。

Integer a = 10;
int b = 10;
System.out.println(a == b); // true

片方がプリミティブの場合はアンボクシングされ、値比較になります。ただし a がnullならNullPointerExceptionです。

nullとequals

String s = null;
System.out.println(s == null);     // true
System.out.println(s.equals("x")); // NullPointerException

null は参照先がない状態です。== null は可能ですが、null に対してメソッド呼び出しはできません。

String s = null;
System.out.println("x".equals(s)); // false

定数文字列側から equals を呼ぶとNullPointerExceptionを避けられます。

判定練習

例1String a = "A"; String b = "A";

a == b は true。リテラル同士。

例2String b = new String("A");

"A" == b は false。newで別インスタンス。

例3String x = "A"; x + "B"

変数を含む連結は実行時生成。== "AB" は false。

例4final String x = "A"; x + "B"

コンパイル時定数。== "AB" は true。

試験での判断順

  1. 型を見る：プリミティブか、参照型か。
  2. newの回数を見る：別々にnewされていれば、基本的に別参照。
  3. リテラルか連結かを見る：リテラル同士・コンパイル時定数ならプールを疑う。
  4. equalsの実装を見る：String/List/recordは内容比較、StringBuilder/配列/通常クラスは注意。
  5. nullを見る：== null は可能。null.equals(...) は例外。
