継承・インタフェース・sealed

多態性、インタフェース実装、オーバーロード、オーバーライド、sealedを整理します。

この章の見方

継承・インタフェース問題は、最初に「宣言型で何を呼べるか」、次に「実体でどのメソッドが動くか」を分けます。フィールド、static、private、コンストラクタは扱いが違うため、同じ継承でも一括りにしないことが重要です。

継承で実装されるもの・されないもの

対象扱い試験での注意
インスタンスメソッドアクセス可能なら継承され、オーバーライド可能呼び出しは宣言型で可否判定、実行は実体で決まる
フィールドアクセス可能なら使えるが多態的ではない同名フィールドは隠蔽。宣言型で決まる
privateメンバサブクラスから直接アクセス不可継承関係でもprivateは同じクラス内だけ
コンストラクタ継承されない子のコンストラクタは先頭でsuper(...)またはthis(...)を呼ぶ
staticメソッド隠蔽。オーバーライドではないインスタンス経由で呼べても、解決は型寄り

親型変数で子インスタンスを扱う

class Employee {
  void work() { System.out.println("employee"); }
}
class Engineer extends Employee {
  @Override
  void work() { System.out.println("engineer"); }
  void code() { System.out.println("code"); }
}

Employee e = new Engineer();
e.work(); // engineer
// e.code(); // コンパイルエラー。Employee型にcode()はない。

変数の宣言型で呼べるメソッドが決まります。呼べると決まったインスタンスメソッドについては、実体クラスのオーバーライドが動きます。

interfaceは「扱い方」の契約

interface Worker {
  void work(); // public abstract
}

class Engineer implements Worker {
  public void work() {}
}

interfaceは実体ではなく型です。Worker w = new Engineer(); のように扱えますが、Worker型で呼べるのはWorkerに定義されたメソッドです。

インタフェースの抽象メソッドはpublicなので、実装側もpublicにします。publicを省くとアクセスを弱める扱いになりコンパイルエラーです。

抽象クラスがinterfaceを実装する場合

interface Worker { void work(); }

abstract class Employee implements Worker {
  // work()を実装しなくてもよい
}

class Engineer extends Employee {
  public void work() {}
}

抽象クラスは抽象メソッドを残せます。通常クラスになった時点で、残っている抽象メソッドをすべて実装する必要があります。

defaultメソッドと衝突

interface A { default void test() {} }
interface B { default void test() {} }

class C implements A, B {
  public void test() {} // 自分で解決する
}

複数インタフェースから同じシグネチャのdefaultメソッドを受け取る場合、クラス側で明示的に実装して衝突を解決します。

sealed

sealed class A permits B, C {}
final class B extends A {}
non-sealed class C extends A {}

sealedの直接の子は許可リストに必要で、子側はfinal/sealed/non-sealedのいずれかを宣言します。
