判定フロー完全版

コンパイル可否、実行時例外、正常出力を分離して、出力問題の見落としを減らすための基準です。

結論：出力を追う前に、問題を3段階に分解する

Java Silverのコード問題で一番多い失敗は、いきなり値を追い始めることです。出力候補が見えても、コンパイルできなければ出力は存在しません。実行時例外が起きれば、正常な出力は最後まで出ません。

1コンパイル可否

型、宣言、アクセス、override、checked例外、到達不能catchを見る。

2実行時例外

null、配列境界、キャスト、parse、0除算、変更不可Listを見る。

3正常出力

評価順、参照、動的束縛、ループ回数、finally/close順を追う。

試験での現実：出力計算が合っていても、前段階のコンパイルエラーを見落としたら不正解です。出力問題ではなく「実行可能性判定問題」だと考えてください。

1段階目：コンパイル可否のチェック順

コンパイルエラーは、コードが実行される前に確定します。次の順に見ると漏れが減ります。

2段階目：実行時例外のチェック順

コンパイルできるコードだけが実行時例外の候補になります。例外が出た時点で、通常の後続処理は飛び、対応するcatchまたはfinallyへ移ります。

3段階目：正常出力を追うときの型

評価順型

++、--、代入式、短絡評価、三項演算子。式の途中で値が変わる。

参照型

a = b 後に、変数がどのインスタンスを指しているかを追う。

動的束縛型

オーバーライドメソッドは実体で決まる。フィールドは宣言型で決まる。

例外制御型

try内のどこで止まり、catch/finally/closeがどの順で動くかを見る。

判定例：宣言型と実体を分ける

class Parent { void run() { System.out.println("P"); } }
class Child extends Parent { void run() { System.out.println("C"); } void childOnly() {} }

public class Main {
    public static void main(String[] args) {
        Parent p = new Child();
        p.run();
        p.childOnly();
    }
}
  1. p.run() はコンパイル可。Parent に run がある。
  2. p.childOnly() はコンパイルエラー。実体が Child でも、宣言型 Parent にそのメソッドはない。
  3. したがって C が表示される、ではありません。コードは実行されません。

30秒チェックリスト

  • publicクラス名とファイル名は一致しているか。
  • package/importの順番は正しいか。
  • mainは public static void main(String[] args) か String... args か。
  • var をフィールド・引数・戻り値型に使っていないか。
  • ローカル変数を初期化前に読んでいないか。
  • byte/short/char の演算結果をそのまま狭い型に入れていないか。
  • interface実装メソッドに public があるか。
  • overrideでアクセスを狭めていないか。
  • checked例外の処理漏れはないか。
  • 親例外catchが先に来ていないか。
  • null参照に対してメソッドを呼んでいないか。
  • 配列・List・Stringのインデックス範囲は正しいか。
