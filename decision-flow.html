判定フロー

Java Silverのコード問題を、コンパイル可否、実行時例外、出力結果の順に切るための基本手順です。

最初に固定する判定順

出力問題で一番やってはいけないのは、コンパイルエラーを確認する前に値を追い始めることです。Java Silverでは「実行すればこうなる」は正しくても、そもそも実行できないコードが頻出します。

順番見るものここで落ちる典型
1クラス・ファイル・宣言publicクラス名とファイル名、package/importの順序、トップレベルクラスの修飾子
2メソッド・変数宣言mainの形、戻り値、引数、var使用場所、ローカル変数の初期化
3型変換・演算byte/short/charのint昇格、booleanと数値の非互換、戻り値型
4継承・overrideアクセスを狭くしていないか、戻り値型、throws、abstract実装漏れ
5例外処理catch順序、チェック例外の処理、finally/catchの並び
6実行時例外NullPointerException、ArrayIndexOutOfBoundsException、ClassCastExceptionなど
7出力追跡副作用、参照、短絡評価、例外発生後の処理スキップ

選択肢の切り方

選択肢に「コンパイルエラー」「実行時例外」「何も表示されない」「特定の出力」が混在する問題では、まずエラー系を検討します。

public class Main {
    public static void main(String[] args) {
        int a;
        System.out.println(a);
    }
}

このコードは出力を考える問題ではありません。ローカル変数 a が初期化されていないため、コンパイルエラーです。

実行時例外を判定する順序

コンパイルできると判断した後は、左から順に実行して、例外が発生した時点で通常の続きは実行されないと考えます。ただし finally は別です。

try {
    String s = null;
    System.out.println(s.length());
    System.out.println("A");
} catch (NullPointerException e) {
    System.out.println("B");
} finally {
    System.out.println("C");
}

s.length() で例外が発生するため A は表示されません。対応する catch の B、その後 finally の C が表示されます。

20秒チェックリスト

  1. トップレベルクラスに使えない修飾子がないか。
  2. main が public static void main(String[] args) または String... args か。
  3. ローカル変数が使用前に確実に初期化されているか。
  4. var がフィールド・引数・戻り値・null 初期化に使われていないか。
  5. 小さい整数型の演算結果をそのまま小さい型へ戻していないか。
  6. overrideでアクセスを狭くしていないか。
  7. チェック例外を処理または宣言しているか。

ミニ演習

Q. 選択肢に出力とコンパイルエラーが両方ある場合、最初に何を見る？

最初にコンパイル可否です。出力を追うのは、コンパイルできることを確認した後です。

Q. try内で例外が出た後、tryの残りは実行される？

実行されません。対応するcatchへ移動します。finallyがあれば最後に実行されます。

実戦での読み順テンプレート

問題文が長いときほど、コードを上から読む前に「問われている結果」を確認します。選択肢にコンパイルエラーがあるなら宣言・型・例外処理を先に見ます。選択肢がすべて出力なら、コンパイル可否の確認を短く済ませて追跡に入ります。

選択肢の構成優先する確認
コンパイルエラーを含む宣言、型、override、catch順序
実行時例外を含むnull、範囲外、キャスト、parse
出力だけ副作用、参照、分岐、ループ
複数選択正しい説明だけでなく、誤説明の根拠も切る

自分に問いかける質問

  • このコードはclassファイルを作れるか。
  • 実行されるmainは本当に存在するか。
  • 例外が出る行に到達するか。
  • 例外が出た場合、それはcatchされるか。
  • 出力される前に処理が止まっていないか。
  • 最後まで正常終了する場合、printlnに渡る値は何か。
