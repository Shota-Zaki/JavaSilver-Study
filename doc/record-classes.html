レコードクラス

ヘッダーから生成されるメンバー、標準コンストラクタ、値比較、可変コンポーネントまで、レコード固有の規則だけを段階的に扱います。

レコード宣言を「省略されたクラス」として展開する

recordは、フィールドを少なく書くための記号ではありません。ヘッダーに並べた値を、そのクラスの状態として固定する特殊なクラス宣言です。まず短い宣言を、通常クラスに近い形へ展開すると、何が自動生成されるかを見失いません。

record Product(String name, int price) {}

概念上は、次の要素を持つクラスとして読みます。完全に同じソースへ置換できるという意味ではなく、メンバーの役割を理解するための展開です。

final class Product extends java.lang.Record {
    private final String name;
    private final int price;

    public Product(String name, int price) {
        this.name = name;
        this.price = price;
    }

    public String name() { return name; }
    public int price() { return price; }

    // equals、hashCode、toStringも値に基づく形で生成される
}
ヘッダーの記述生成されるもの読み違えやすい点
String nameprivate finalのコンポーネントフィールドproduct.nameでは直接参照できない
String namepublic String name()getName()ではない
全コンポーネント標準コンストラクタ宣言順と引数順が対応する
全コンポーネントequals、hashCode、toString==の意味は変わらない

生成直後の状態を決めるのは標準コンストラクタ

レコードの中心は「全コンポーネントを一度に確定する経路」です。この経路が標準コンストラクタです。通常形式ではフィールド代入まで自分で記述し、コンパクト形式では検証・加工だけを書き、代入は本体終了後に暗黙実行されます。

通常形式

public record Range(int min, int max) {
    public Range(int min, int max) {
        if (min > max) {
            throw new IllegalArgumentException();
        }
        this.min = min;
        this.max = max;
    }
}

全コンポーネントを明示的に代入します。1つでも未代入ならコンパイルできません。

コンパクト形式

public record User(String name) {
    public User {
        if (name == null) {
            throw new IllegalArgumentException();
        }
        name = name.strip();
    }
}

仮引数リストを書きません。最後にthis.name = name;相当が補われるため、引数を正規化してから保存できます。

コンパクトコンストラクタの本体で、コンポーネントフィールドへ直接代入することはできません。

record User(String name) {
    User {
        // this.name = name; // コンパイルエラー
        name = name.strip(); // 仮引数の値を変更するのは可
    }
}

また、標準コンストラクタのアクセス範囲をレコード本体より狭くできません。public recordの標準コンストラクタをパッケージアクセスに落とすコードは不正です。

追加コンストラクタは必ず標準経路へ合流する

引数の少ない補助コンストラクタを追加できます。ただし、レコードの状態確定を複数経路に分散させないため、追加コンストラクタは先頭で同じレコードの別コンストラクタを呼びます。

record Point(int x, int y) {
    Point() {
        this(0, 0);       // 標準コンストラクタへ委譲
    }

    Point(int value) {
        this(value, value);
    }
}
record Point(int x, int y) {
    Point() {
        // super();       // レコードの追加コンストラクタでは不可
        // System.out.println("start");
        this(0, 0);       // 明示的コンストラクタ呼出しが先頭
    }
}

読むときは、呼び出されたコンストラクタを順にたどり、最後に全コンポーネントが確定する標準コンストラクタへ到達することを確認します。

==、equals、配列コンポーネントを分ける

record Item(String name, int price) {}

Item a = new Item("pen", 100);
Item b = new Item("pen", 100);

System.out.println(a == b);       // false
System.out.println(a.equals(b));  // true

==は参照の同一性を比較するため、別々に生成した2個のレコードはfalseです。生成されたequalsは、対応するコンポーネント値を比較します。

ただし、コンポーネントが配列の場合は注意が必要です。配列自身のequalsは内容比較ではなく参照比較です。

record Data(int[] values) {}

Data x = new Data(new int[]{1, 2});
Data y = new Data(new int[]{1, 2});

System.out.println(x.equals(y));  // false

「レコードなら必ず深い内容比較になる」と一般化してはいけません。コンポーネント型が持つ比較規則まで見ます。

finalなのは参照であり、参照先の可変性までは止めない

コンポーネントフィールドはfinalですが、可変オブジェクトを保持した場合、その内部状態は変更できます。

record Scores(java.util.List<Integer> values) {}

var list = new java.util.ArrayList<Integer>();
list.add(10);
Scores scores = new Scores(list);
list.add(20);
System.out.println(scores.values()); // [10, 20]

不変な値オブジェクトとして使うなら、コンストラクタで防御的コピーを作ります。

record Scores(java.util.List<Integer> values) {
    Scores {
        values = java.util.List.copyOf(values);
    }
}

この設計上の注意は、コンパイル可否とは別問題です。コード読解では「代入できるか」、設計では「外部から状態を変えられないか」を分けて確認します。

レコード本体に書けるもの・書けないもの

宣言可否根拠
インスタンスメソッド可コンポーネントを使う振る舞いを追加できる
staticフィールド・メソッド可各インスタンス固有の追加状態ではない
追加インスタンスフィールド不可インスタンス状態はヘッダーのコンポーネントで表す
インスタンス初期化子不可初期化は標準コンストラクタへ集約する
インターフェース実装可implementsを使用できる
別クラスの継承不可暗黙にjava.lang.Recordを継承し、レコード自体もfinal
interface Printable {
    void print();
}

record User(String name) implements Printable {
    static int count;

    @Override
    public void print() {
        System.out.println(name());
    }

    // int age; // コンパイルエラー
}

コード問題は「ヘッダー→生成メンバー→独自宣言」の順で読む

次のコードでは、最初にヘッダーから自動生成されるメンバーを確定し、その後で本体の宣言がそれらと衝突しないかを調べます。

record Sample(int value) {
    public int value() {
        return value * 2;
    }
}

アクセサを明示的に宣言すること自体は可能です。名前はコンポーネント名、引数なし、戻り値型はコンポーネント型、アクセスはpublicという形を守ります。この例のnew Sample(5).value()は10です。一方、getValue()を追加しても、それは自動生成アクセサの置換ではなく別メソッドです。

レコード問題の着眼点：通常クラスの常識を当てはめる前に、ヘッダーが生成するフィールド・アクセサ・標準コンストラクタを紙上に書き出します。
