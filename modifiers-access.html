修飾子・アクセス制御

public/protected/default/private、abstract、final、static、sealedを、使える場所とアクセス範囲で整理します。

アクセス修飾子の範囲

修飾子同じクラス同じパッケージ別パッケージのサブクラス別パッケージの非サブクラス
private可不可不可不可
指定なし可可不可不可
protected可可条件付きで可不可
public可可可可

protected は「同じパッケージなら可」に加えて、「別パッケージでも継承関係があれば条件付きで可」です。ここを単に“サブクラスなら全部OK”と覚えると落ちます。

トップレベルクラスに使える修飾子

トップレベルクラスには public、指定なし、abstract、final、sealed、non-sealed などが関係します。private や protected、static はトップレベルクラスには使えません。

public class A { }      // OK
class B { }             // OK
// private class C { }  // NG: トップレベルでは不可
// static class D { }   // NG: トップレベルでは不可

abstractとfinalは同時に使えない

abstract は継承して実装されることを前提にします。final は継承禁止です。したがって同時には使えません。

abstract class A { }
final class B { }
// abstract final class C { } // NG

privateメンバと継承

privateメンバはサブクラスから直接アクセスできません。また、privateメソッドはoverride対象でもありません。

class A {
    private void test() {}
}
class B extends A {
    void test() {} // overrideではなく、別メソッド
}

sealed関連修飾子

sealed は継承・実装できる相手を制限します。sealedクラスを継承したクラスは、final、sealed、non-sealed のいずれかで今後の継承方針を明示します。

sealed class A permits B, C { }
final class B extends A { }
non-sealed class C extends A { }

ミニ演習

Q. protectedは「同じパッケージだけ」？

違います。同じパッケージなら継承関係なしでもアクセス可能です。別パッケージでは継承関係を使ったアクセスが論点になります。

Q. privateメソッドをサブクラスで同名定義したらoverride？

overrideではありません。privateメソッドはサブクラスから見えないためです。

protectedの別パッケージアクセス

別パッケージのサブクラスからprotectedメンバへアクセスする場合、「サブクラスのインスタンスとして」アクセスする必要があります。親型の別インスタンス経由では落ちる形があります。

package p1;
public class A { protected int value; }

package p2;
public class B extends p1.A {
    void test(B b, p1.A a) {
        b.value = 1; // OK
        // a.value = 1; // NGになりうる形
    }
}
