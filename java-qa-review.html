質問整理まとめ

最近確認した論点を、試験で使える判断基準として再整理します。用語暗記ではなく、コードを読んだときの失点箇所を潰すためのページです。

interface Worker { void work(); } はどういう状態か

これは「Workerとして扱うなら work() を呼べる」という契約だけを定義している状態です。実装はありません。interface内の抽象メソッドは暗黙的に public abstract です。

interface Worker {
    void work(); // public abstract void work(); と同じ扱い
}

class Engineer implements Worker {
    public void work() {
        System.out.println("work");
    }
}

実装クラス側で public を付け忘れると、interfaceのpublicメソッドよりアクセスを狭めることになりコンパイルエラーです。

抽象メソッド

抽象メソッドは、本体を持たない未完成のメソッドです。抽象メソッドを含むクラスはabstractでなければなりません。具象クラスにするなら、継承した抽象メソッドをすべて実装する必要があります。

abstract class Employee {
    abstract void work();
}

class Engineer extends Employee {
    @Override
    void work() {
        System.out.println("engineer");
    }
}
抽象メソッドは「まだ処理が決まっていないメソッド」です。呼び出し方だけ先に決め、具体的な処理はサブクラスに任せます。

Engineerにwork()がなくても、親にwork()があればOKか

はい、条件付きでOKです。具象親クラスが work() を実装済みなら、子クラスはそのメソッドを継承しているため、interfaceの契約を満たせます。

interface Worker {
    void work();
}

class Employee {
    public void work() {
        System.out.println("employee");
    }
}

class Engineer extends Employee implements Worker {
    // work()を書かなくても、Employeeから継承したpublic work()で契約を満たす
}

ただし、親の work() がprivateだったり、戻り値や引数が違ったり、アクセスがpublicでなかったりすると満たせません。

extends / implements / abstract の整理

extends

クラスなら1つだけ継承。interface同士なら複数継承可能です。

implements

クラスがinterfaceの契約を満たすときに使います。複数実装可能です。

abstract class

未完成のクラス。抽象メソッドを持てますが、直接newできません。

interface

外から見た操作契約。抽象メソッドはpublic abstract扱いです。

オーバーライドとオーバーロード

オーバーライドは親のメソッドを子で再定義すること。オーバーロードは同じ名前で引数リストが違うメソッドを複数用意することです。

class A {
    void test(int n) { }
    void test(String s) { } // overload
}

class B extends A {
    @Override
    void test(int n) { }    // override
}

override判定

同じシグネチャか。戻り値は同じか共変戻り値か。アクセスを狭めていないか。

overload判定

引数リストが違うか。戻り値だけ違うものは不可。

@Override

付けなくてもオーバーライドは成立しますが、付けておくと誤りをコンパイラが検出します。

static

staticメソッドはオーバーライドではなく隠蔽です。

record

recordは値の組を表すクラスを短く書く構文です。アクセサは getName() ではなく name() です。継承不可、追加インスタンスフィールド不可、interface実装可、という制限を押さえます。

record Person(String name, int age) {}

Person p1 = new Person("A", 10);
Person p2 = new Person("A", 10);

System.out.println(p1.name());     // A
System.out.println(p1 == p2);      // false
System.out.println(p1.equals(p2)); // true

sealed class / sealed interface

sealedは継承先・実装先を限定する仕組みです。親の permits で許可された直接サブタイプだけが継承・実装できます。子側は final、sealed、non-sealed のどれかを必ず示します。

sealed interface Service permits FileService {}

final class FileService implements Service { }
// class NetService implements Service { } // NG。permitsされていない
sealedは「newできない」ではありません。newできないのはabstractです。ここを混同すると選択肢で落ちます。

try-catch-finally の流れ

try内で例外が発生した時点で、それ以降のtry内処理はスキップされ、対応するcatchへ移ります。finallyは例外の有無に関係なく基本的に実行されます。

try {
    System.out.println("A");
    int[] x = new int[0];
    x[0] = 10;
    System.out.println("B"); // 実行されない
} catch (ArrayIndexOutOfBoundsException e) {
    System.out.println("C");
} finally {
    System.out.println("D");
}
// A C D

try-with-resources と close()

try-with-resourcesは、AutoCloseable を実装したリソースを自動的に閉じます。closeは宣言した順番の逆順に呼ばれ、finallyはその後です。

try (A a = new A("A"); A b = new A("B")) {
    System.out.println("body");
} finally {
    System.out.println("finally");
}
// body B A finally

RuntimeException

RuntimeException 系は非チェック例外です。catchやthrowsをコンパイラに強制されませんが、実行時に発生すれば通常の例外として制御が移ります。

NullPointerException

null参照に対してメソッドやフィールドへアクセス。

ArrayIndexOutOfBoundsException

配列の範囲外アクセス。

NumberFormatException

数値変換できない文字列をparse。

ClassCastException

実体と互換性のない型へのキャスト。

トップレベルクラス・サブクラス・インナークラス

トップレベルクラスはファイル直下に書くクラスです。トップレベルに使えるアクセス修飾子はpublicかデフォルトだけです。サブクラスは継承関係にある子クラスです。インナークラスはクラスの中に定義されたクラスです。

public class Outer {          // トップレベルクラス
    class Inner { }           // インナークラス
    static class Nested { }    // staticネストクラス
}

class Child extends Outer { }  // サブクラス
