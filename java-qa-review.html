混ざりやすい疑問をQ&A形式で整理し、試験で使える判断基準に落とし込みます。

このページの位置づけ

ここは、学習中に混ざりやすい質問をQ&A形式で整理するページです。結論だけ覚えるのではなく、なぜその結論になるかを判定順で確認してください。

アクセス修飾子・継承

Q. publicクラスなら、別パッケージでも中のメンバを自由に使える？

使えません。public クラスであることは「そのクラス型を外部から見える」という意味です。フィールドやメソッドを使えるかは、それぞれのメンバのアクセス修飾子で別に判定します。

たとえば public class A の中に defaultアクセスのフィールド int value; がある場合、別パッケージから a.value は使えません。

Q. protectedコンストラクタはサブクラスの super() から呼べる？

呼べます。protected は同じパッケージ、または継承関係にあるサブクラスからのアクセスを許します。別パッケージのサブクラスでも、サブクラスのコンストラクタ内で super() として呼ぶのは可能です。

ただし、別パッケージの無関係なクラスが new A() のように呼ぶことはできません。

Q. privateメソッドはサブクラスで同じ名前を書いたらoverride？

overrideではありません。private メソッドはそのクラスの外から見えないため、サブクラスの同名メソッドは「親のメソッドを再定義したもの」ではなく、別メソッドです。@Override を付けるとコンパイルエラーになります。

Q. override時にアクセス修飾子はどう変えられる？

同じか、より広くできます。狭くはできません。protected を public にするのはOKですが、public を protected にするのはNGです。

ポリモーフィズム・キャスト

Q. A a = new B(); のとき、Bだけにあるメソッドは呼べる？

呼べません。コンパイル時に呼べるメソッドは宣言型 A で決まります。実体が B でも、A 型にそのメソッドがなければコンパイルエラーです。

Q. ではoverrideされたメソッドはAとBのどちらが動く？

実体である B 側が動きます。ここが重要です。呼べるかどうかは宣言型、override後にどの実装が動くかは実体です。

Q. フィールドもoverrideされる？

されません。フィールドはメソッドと違い、多態的に解決されません。A a = new B(); a.value のように直接フィールドへアクセスした場合、参照変数の型 A のフィールドが使われます。

Q. キャストすれば何でも呼べる？

コンパイル上は通ることがありますが、実体が一致していなければ実行時に ClassCastException です。Object o = "abc"; Integer i = (Integer)o; はコンパイル可能でも実行時例外です。

Q. null をキャストしたら例外？

例外ではありません。String s = (String) null; は可能で、結果は null です。ただし、その後 s.length() のようにメソッドを呼ぶと NullPointerException です。

String・StringBuilder

Q. substring や replace を呼んだのに文字列が変わらないのはなぜ？

String は不変だからです。substring や replace は、元の文字列を変更せず、新しい文字列を返します。戻り値を代入しなければ元の変数は変わりません。

Q. StringBuilder はなぜ参照コピーで影響が出る？

StringBuilder は可変です。複数の変数が同じ StringBuilder インスタンスを参照している場合、一方から append すると同じ実体が変わるため、もう一方から見ても変わっています。

Q. StringBuilder.equals() は内容比較？

違います。String の equals は内容比較ですが、StringBuilder は内容比較用に equals を上書きしていません。別インスタンスなら同じ文字列を持っていても false です。

Q. str.substring(x + 3); だけ書いたら str は変わる？

変わりません。戻り値を捨てています。str = str.substring(x + 3); と代入して初めて変数 str の参照先が変わります。

配列・List

Q. int[] a; と書いた時点で配列はある？

ありません。これは配列を参照する変数を宣言しただけです。new int[3] や {1,2,3} で配列インスタンスを作って初めて要素が存在します。

Q. 長さ0の配列は作れる？

作れます。new int[0] は有効です。ただし要素がないので a[0] は ArrayIndexOutOfBoundsException です。

Q. List.of と new ArrayList は同じように変更できる？

できません。List.of が返すリストは変更不可です。add や remove を呼ぶと UnsupportedOperationException です。

Q. remove(1) は値1を消す？

List<Integer> では特に注意が必要です。remove(1) は通常、インデックス1の要素を削除します。値としての Integer.valueOf(1) を消したいなら remove(Integer.valueOf(1)) のようにします。

例外

Q. RuntimeExceptionかどうかは何で判断する？

クラス階層で判断します。RuntimeException のサブクラスなら非チェック例外です。代表例は NullPointerException、ArrayIndexOutOfBoundsException、ClassCastException、NumberFormatException、ArithmeticException です。

Q. チェック例外は全部覚えるしかない？

代表例は覚える必要があります。ただし試験上は、Exception 直下寄り、IOException 系はチェック例外、RuntimeException 系は非チェック例外と切るのが実用的です。

Q. try内で例外が起きたら、下の行は実行される？

されません。例外が発生した時点で、対応する catch へ制御が移ります。対応する catch がなくても、finally があればそれを実行してから外へ送出されます。

Q. catchは広い例外から書いていい？

だめです。Exception を先に書くと、その後の IOException などは到達不能になりコンパイルエラーです。狭い例外から広い例外へ書きます。

Q. try-with-resourcesのclose順は？

宣言と逆順です。try (A a = ...; B b = ...) なら、先に b.close()、次に a.close() です。その後に finally が実行されます。

Java 17

Q. recordのアクセサはgetter形式？

違います。record User(String name) のアクセサは name() です。getName() は自動生成されません。

Q. recordのフィールドは変更できる？

基本的にできません。recordコンポーネントに対応するフィールドは private final です。値を保持するための簡潔なクラスとして扱います。

Q. sealedクラスを継承したクラスは普通に書ける？

書けません。許可されたサブクラス側も、final、sealed、non-sealed のいずれかで今後の継承方針を明示する必要があります。

Q. switch式では break で値を返す？

基本は -> 形式で値を返すか、ブロック内で yield を使って値を返します。従来のswitch文とは別に考えるべきです。

最後に残りやすい誤解

  • 「実体がBだからBのメソッドを呼べる」は誤り。呼べるかは宣言型。
  • 「Stringのメソッドを呼べば元の文字列が変わる」は誤り。Stringは不変。
  • 「publicクラスなら中身もpublic」は誤り。メンバごとに判定。
  • 「catchすればtryの残りに戻る」は誤り。戻らない。
  • 「フィールドもoverrideされる」は誤り。メソッドと分ける。
  • 「RuntimeExceptionはcatch必須」は誤り。非チェック例外。
