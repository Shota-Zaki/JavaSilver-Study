質問整理まとめ

最近確認した論点を、試験で使える判断基準として整理します。用語の暗記ではなく、コードを読んだときにどこで落とされるかを判断できる状態を目標にします。

record

recordは「データを入れるだけのクラス」を短く書くための仕組みです。Java Silver 17では、通常クラスとの違い、生成されるメンバ、変更できる点とできない点が狙われます。

public record Person(String name, int age) { }

この1行で、private final相当のフィールド、コンストラクタ、name()、age()、equals、hashCode、toString が用意されます。アクセサ名は getName() ではなく name() です。ここを曖昧にすると選択肢で簡単に落ちます。

本質

recordはデータキャリアです。状態を持たせるための一般的なクラスではなく、決まった値の組を扱うための型です。

継承

recordは暗黙的に java.lang.Record を継承するため、他のクラスを extends できません。

変更不可

コンポーネント由来のフィールドは実質finalです。後から値を書き換えるsetterは基本的に作りません。

追加可能

メソッド、staticフィールド、コンパクトコンストラクタは定義できます。ただしインスタンスフィールドを自由に増やす発想は危険です。

recordを「getter/setterを自動生成するクラス」と覚えるのは弱いです。正しくは「コンポーネントから、値を表す型を作る構文」です。

interface / implements / abstract / extends

extends は既存の型を拡張する、implements はinterfaceの契約を満たす、という違いです。ここを言葉だけで覚えると、抽象クラス・interface・多重継承の問題で混乱します。

interface Worker {
    void work();
}

class Engineer implements Worker {
    public void work() {
        System.out.println("work");
    }
}

interfaceの抽象メソッドは暗黙的に public abstract です。実装クラスで public を付け忘れると、アクセス修飾子を弱めたことになりコンパイルエラーです。

class extends class

クラスの継承。Javaではクラスの多重継承はできません。

class implements interface

interfaceの実装。複数のinterfaceを同時に実装できます。

interface extends interface

interface同士の継承。複数継承できます。

abstract class

未完成のクラス。抽象メソッドを残せますが、直接 new はできません。

未実装の抽象メソッドが残るクラスは、必ずabstractにする必要があります。逆に、具象クラスにするなら全抽象メソッドを実装しなければなりません。

override と overload

オーバーライドとオーバーロードは名前が似ていますが、判定基準が別物です。試験では、戻り値・アクセス修飾子・引数リストを混ぜて誤答を作ってきます。

overload

同じメソッド名で、引数リストが違うものを複数定義すること。呼び出し先はコンパイル時に決まります。

override

親のインスタンスメソッドを子で再定義すること。実行時の実体クラスで呼び出し先が決まります。

戻り値

overloadは戻り値だけ違っても不可。overrideは同じ型または共変戻り値が必要です。

アクセス修飾子

overrideでは親より狭くできません。public を package-private にするような変更は不可です。

class A {
    Number value() { return 1; }
    void test(int n) { }
    void test(String s) { } // overload
}

class B extends A {
    @Override
    Integer value() { return 1; } // covariant return OK

    @Override
    void test(int n) { }          // override
}
判定順は、まず「同じシグネチャか」を見る。同じならoverride候補、違うならoverload候補です。戻り値だけ見て判断するのは危険です。

sealed class / sealed interface

sealedは、継承・実装できる直接のサブタイプを限定する仕組みです。sealed は「newできない」という意味ではありません。newできないのは abstract です。

public sealed class Animal permits Dog, Cat { }
public final class Dog extends Animal { }
public non-sealed class Cat extends Animal { }

sealedクラスを継承した直接の子クラスは、必ず final、sealed、non-sealed のどれかで方針を示します。

final

ここで継承を止めます。

sealed

さらに次の継承先を限定します。

non-sealed

ここから先の継承制限を解除します。

permits

直接の許可クラスを列挙します。許可されていないクラスは継承できません。

sealed問題では、親側の permits と子側の final/sealed/non-sealed の両方を見る。片方だけ見て正誤判断すると外します。

try-with-resources と close()

try-with-resourcesは、リソースを自動で閉じる構文です。対象は AutoCloseable を実装しているオブジェクトです。

try (A a = new A("A"); A b = new A("B")) {
    System.out.println("body");
} finally {
    System.out.println("finally");
}

この場合、出力順は body → B → A → finally です。closeは宣言した順番の逆順に呼ばれます。finallyはその後です。

宣言順

A a、次に A b の順で作られます。

close順

閉じるときは逆順。先に b.close()、次に a.close()。

finally

リソースのclose後に実行されます。

例外

close() がチェック例外を投げる可能性があるなら、catchまたはthrowsが必要です。

RuntimeException

RuntimeException とそのサブクラスは非チェック例外です。コンパイラはcatchやthrowsを強制しません。ただし、実行時に発生すれば当然プログラムの流れは変わります。

NullPointerException

null参照に対してメソッドやフィールドへアクセスしたとき。

ArrayIndexOutOfBoundsException

配列の範囲外インデックスへアクセスしたとき。

NumberFormatException

Integer.parseInt("abc") のように数値変換できないとき。

ClassCastException

実体と互換性のない型へキャストしたとき。

例外問題は、出力順を追う前にコンパイル可否を確認します。コンパイルできないコードに「実行時例外」は発生しません。この順番を崩すと、実行結果系の問題で誤答します。
