Java 17論点

record、sealed、switch式、instanceofパターンマッチングを、試験で失点しない判定順に落とし込みます。

このページの使い方

Java 17論点は、単に「新しい書き方を知っているか」ではありません。試験では、その構文がどこまで自動生成するか、どの修飾子が必須か、どの範囲で変数が使えるかを狙ってきます。

1構文として成立するか

record、sealed、switch式、instanceofパターンの形を確認する。

2暗黙に補われるものを読む

recordのフィールド・アクセサ、interfaceの修飾子など。

3実行時の型・出力を追う

toString、equals、switchの戻り値、pattern変数のスコープ。

Java 17論点は、クラス・static・record、修飾子・アクセス制御、ポリモーフィズム・キャストと強くつながっています。

recordの基本：データ保持用クラスを短く書く

record は、値のまとまりを表すクラスを簡潔に書くための仕組みです。次の1行だけで、コンストラクタ、アクセサ、equals、hashCode、toString などが自動的に用意されます。

record Item(String name, int price) {}

public class Main {
    public static void main(String[] args) {
        Item item = new Item("apple", 100);
        System.out.println(item.name());
        System.out.println(item.price());
        System.out.println(item);
    }
}
自動生成されるもの試験での注意点
private final フィールド外部から直接 item.name のようには読めない。
標準コンストラクタnew Item("apple", 100) のように全コンポーネントを渡す。
アクセサgetName() ではなく name()。
equals/hashCode同じコンポーネント値なら同値と判定される。
toStringクラス名とコンポーネント名付きで表示される。

特にアクセサ名の罠は頻出です。JavaBeans風の getName() を勝手に想像すると、存在しないメソッド呼び出しになりコンパイルエラーです。

recordの同値比較とtoString

recordは equals を自動生成します。通常クラスで equals をオーバーライドしていない場合と違い、コンポーネント値で比較される点が重要です。

record Item(String name, int price) {}

public class Main {
    public static void main(String[] args) {
        Item a = new Item("apple", 100);
        Item b = new Item("apple", 100);
        System.out.println(a == b);
        System.out.println(a.equals(b));
        System.out.println(a);
    }
}
式結果理由
a == bfalse別々に生成したインスタンスなので参照が違う。
a.equals(b)truerecordの自動生成 equals がコンポーネント値を比較する。
System.out.println(a)Item[name=apple, price=100] 形式record用の toString が自動生成される。

== と equals の違いはrecordでも変わりません。recordだからといって == が内容比較になるわけではありません。

recordの制約

recordは普通のクラスより制約があります。制約を知らないと「それっぽく見えるコード」に引っかかります。

項目可否理由・注意
別クラスを extends不可recordは暗黙的に java.lang.Record を継承する。
interface を implements可通常クラスと同じく実装できる。
追加のインスタンスフィールド不可recordの状態はコンポーネントで表す。
static フィールド・メソッド可インスタンス状態ではないため定義できる。
インスタンスメソッド追加可コンポーネントを使った処理を追加できる。
暗黙的な finalありrecordをさらに継承することはできない。
 
interface Printable {
    void print();
}

record User(String name) implements Printable {
    static int count = 0;          // OK
    public void print() {         // OK
        System.out.println(name);
    }
    // int age;                   // NG: 追加インスタンスフィールド不可
}

// class SubUser extends User {}  // NG: recordは継承不可

recordのコンストラクタ：標準形式とコンパクト形式

recordには、全コンポーネントに対応するコンストラクタがあります。処理を追加したい場合、標準コンストラクタまたはコンパクトコンストラクタを書けます。

標準コンストラクタ

record User(String name) {
    public User(String name) {
        if (name == null) {
            throw new IllegalArgumentException();
        }
        this.name = name;
    }
}

コンパクトコンストラクタ

record User(String name) {
    public User {
        if (name == null) {
            throw new IllegalArgumentException();
        }
        name = name.trim();
    }
}

コンパクトコンストラクタでは、引数リストを書きません。処理後にコンポーネントへの代入が行われるため、引数値の検証や加工を書くのに向いています。

試験での見方

recordのコンストラクタ問題では、通常クラスのコンストラクタ問題と同じく「戻り値型を書いていないか」「アクセス修飾子は不自然でないか」「コンポーネント名と一致しているか」を確認します。

sealed class / sealed interface

sealed は、継承または実装できる相手を制限する仕組みです。許可リストに書かれていないクラスは、継承・実装できません。

sealed class Animal permits Dog, Cat {}

final class Dog extends Animal {}
non-sealed class Cat extends Animal {}

// class Bird extends Animal {} // NG: permitsにない

sealedの出題では、親側と子側の両方を見ます。

確認箇所見る内容
親側sealed が付いているか。permits に許可クラスが列挙されているか。
子側許可されているか。さらに final / sealed / non-sealed のいずれかが付いているか。
孫クラス以降final ならそこで終了。non-sealed なら通常の継承に戻る。

sealedでコンパイルエラーになる典型

permitsにない

sealed class A permits B {}
final class B extends A {}
final class C extends A {} // NG

C は許可されていないためコンパイルエラーです。

子側の修飾子忘れ

sealed class A permits B {}
class B extends A {} // NG

sealedの直接サブクラスは、継承方針を明示する必要があります。

普通のクラスにnon-sealed

class A {}
non-sealed class B extends A {} // NG

non-sealed はsealed階層の制限を解除するための指定です。普通の継承には使えません。

sealedを継続するのにpermitsなし

sealed class A permits B {}
sealed class B extends A {} // 状況によりNG

B をさらにsealedにするなら、次に誰を許可するかを設計上明確にする必要があります。試験ではサブクラス指定の有無を確認します。

switch式：値を返すswitch

従来の switch 文は処理を分岐します。一方、switch 式は値を返します。

int n = 2;
String result = switch (n) {
    case 1 -> "one";
    case 2 -> "two";
    default -> "other";
};
System.out.println(result);
観点switch文switch式
値を返すか返さない返す
代入できるか不可可
->使用可使用可
ブロックで値を返す通常の処理を書くyield を使う
網羅性必須ではない値を返すため全経路が必要

switch式のyield

case -> の右側に単一式を書くならその値が返ります。複数行の処理を書いて値を返す場合は yield を使います。

int score = 80;
String rank = switch (score / 10) {
    case 10, 9 -> "A";
    case 8 -> {
        System.out.println("border");
        yield "B";
    }
    default -> "C";
};

return と yield は違います。yield はswitch式の値を返すためのものです。メソッドから戻るわけではありません。

switchで狙われる型・caseラベル

Java Silverでは、switchの中身そのものより、switchに使える型か、caseラベルが定数かを問われやすいです。

項目要点
使える主な型byte, short, char, int, 対応ラッパー, String, enum など。
long, float, double, booleanswitch式・文の制御式として不適切。
caseラベルコンパイル時定数である必要がある。
final 変数初期化値がコンパイル時定数ならcaseに使える。
 
final int A = 1;
int b = 2;

switch (A) {
    case A: System.out.println("A"); break;
    // case b: System.out.println("B"); break; // NG: 定数ではない
}

instanceof パターンマッチング

Java 17では、instanceof の判定と同時に変数を宣言できます。trueと判定された範囲内で、その変数をキャスト済みの型として使えます。

Object obj = "abc";

if (obj instanceof String s) {
    System.out.println(s.length());
}

従来の書き方では、判定後に明示的なキャストが必要でした。

if (obj instanceof String) {
    String s = (String) obj;
    System.out.println(s.length());
}

パターンマッチングは、このキャストを省略する構文です。obj 自体の宣言型が変わるわけではありません。

パターン変数のスコープ

パターン変数は、型判定がtrueだと保証できる範囲でだけ使えます。

コード可否理由
if (obj instanceof String s) { s.length(); }OKifブロック内ではStringと確定している。
if (obj instanceof String s && s.length() > 0)OK&& の右側は左側がtrueのときだけ評価される。
if (obj instanceof String s || s.length() > 0)NG|| の右側では s が存在する保証がない。
System.out.println(s); をif外に書く原則NGifがfalseの場合、s は宣言されていない。
 
Object obj = "abc";

if (obj instanceof String s && s.length() > 2) {
    System.out.println(s.toUpperCase());
}

ミニ演習

Q1. record Item(String name) { int count; } はコンパイルできる？

できません。recordに追加のインスタンスフィールドは定義できません。static フィールドなら定義できます。

Q2. new Item("A").getName() は呼べる？

recordのアクセサは name() です。getName() は自動生成されません。

Q3. sealedクラスを継承した直接サブクラスに何も修飾子を付けないと？

コンパイルエラーです。final、sealed、non-sealed のいずれかを明示します。

Q4. switch式のブロック内で値を返すキーワードは？

yield です。return ではありません。

Q5. if (obj instanceof String s || s.length() > 0) はなぜだめ？

|| の右側は、左側がfalseのときに評価されます。その場合、s は宣言されていません。
