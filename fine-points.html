Java Silver Reference

細かい挙動・境界ルール集

yield、switch式、戻り値、評価順、境界値、暗黙変換など、1文字差で正誤が変わる規則を例題で整理します。

このページの目的

ここでは、問題文では一見小さく見えるが、Java Silverで正誤を分ける細かい規則をまとめます。特に、switch式のyield、式と文の違い、セミコロン位置、戻り値、暗黙変換、評価順、null、例外発生タイミングを重点的に扱います。

  • コード問題では、まず「これは文か、式か」を分ける。
  • 式なら「値を返すか」「その値をどこで使うか」を見る。
  • 文なら「実行されるだけで、値を返さない」ものとして扱う。
  • Java Silverでは、処理内容よりも「文法として成立するか」で落とされる問題が多い。

switch式：yieldは「switch式の返り値」を指定する

int n = 2;
String result = switch (n) {
  case 1 -> "one";
  case 2 -> {
    System.out.println("two block");
    yield "two";
  }
  default -> "other";
};
System.out.println(result);

// 出力
// two block
// two

yield は、switch式全体の値を返します。メソッドの return ではありません。上の例では yield "two"; によって、switch (n) { ... } という式全体の値が "two" になります。

判定
switch式の -> { ... } ブロックで値を返すなら、原則 yield 値; が必要です。

switch式：最後のセミコロンはswitch式全体に必要

int n = 1;
String s = switch (n) {
  case 1 -> "A";
  default -> "B";
}; // ここに ; が必要

System.out.println(s); // A

これは代入文です。右辺にswitch式を書いて、その結果を s に代入しています。したがって、最後に代入文を終わらせる ; が必要です。

int n = 1;
switch (n) {
  case 1 -> System.out.println("A");
  default -> System.out.println("B");
} // switch文として単独で使う場合、ここに ; は不要

一方、switch文として単独で使っている場合は、ブロックで完結します。ここを混同すると、問題文の } の後の ; の有無で誤判定します。

switch式：yieldとreturnの違い

static String test(int n) {
  String result = switch (n) {
    case 1 -> "A";
    case 2 -> {
      yield "B";   // switch式の値を返す
    }
    default -> "C";
  };
  return result;   // メソッドの戻り値を返す
}
  • yield：switch式の値を決める。
  • return：メソッドそのものを終了して、メソッドの戻り値を返す。
  • yield をメソッド外で使えるわけではない。switch式の中で使う。

Java Silverでは、yield を return のように読んでしまうと崩れます。yieldは「switch式の右辺値を作るもの」と固定してください。

switch式：arrow形式とcolon形式の違い

int n = 2;
String a = switch (n) {
  case 1 -> "A";
  case 2 -> "B";
  default -> "C";
};

String b = switch (n) {
  case 1:
    yield "A";
  case 2:
    yield "B";
  default:
    yield "C";
};

-> 形式では、右側に式・ブロック・throwを書けます。単純な値なら yield は不要です。: 形式のswitch式では、値を返すために yield が必要になります。

switch文の case 1: と、switch式の case 1: は似ていますが、switch式は最終的に値を返さなければいけません。

switch式：全分岐が値を返す必要がある

int n = 3;
String s = switch (n) {
  case 1 -> "A";
  case 2 -> "B";
  // default がないため、値を返せない可能性がある
}; // コンパイルエラー

switch式は値を返す式なので、どの分岐でも値が決まる必要があります。int など全パターンを列挙しきれない型では、基本的に default が必要です。

enum Color { RED, BLUE }
Color c = Color.RED;
String s = switch (c) {
  case RED -> "R";
  case BLUE -> "B";
}; // enumの全値を網羅していればOKになり得る

enumでは全定数を網羅すればdefaultなしで成立するケースがあります。ただし、試験では「すべての値に対して返り値があるか」を見るだけで十分です。

switch：selectorに使える型とnull

String s = null;
switch (s) {
  case "A": System.out.println("A"); break;
  default: System.out.println("default");
}
// 実行時に NullPointerException

参照型をswitchの対象にしたとき、対象が null なら実行時に NullPointerException です。これはコンパイルエラーではありません。

Java Silverでは、String のswitch、Integer などラッパー型のswitchでnullが絡む問題に注意します。

式と文：値を返すもの、返さないもの

分類例値を返すか注意
式a + b返す計算結果を代入・比較・出力に使える
式a++返す返す値と更新後の値が違う
式switch (...) { ... }返すswitch式の場合。全分岐が値を返す
文if (...) { ... }返さない変数へ代入できない
文for (...) { ... }返さないループは値ではない
文switch (...) { ... }返さないswitch文の場合。breakの有無を見る
int x = if (true) { 1; } else { 2; }; // コンパイルエラー
int y = true ? 1 : 2;                // OK。三項演算子は式

do-while・break・continue・++の詳細

ループはswitch式と同じくらい、1文字差で結果が変わります。特に do-while は「最低1回実行」、continue は「forなら更新式へ進む」、break は「更新式を実行せず抜ける」を分けてください。

ループ専用ページで確認する

三項演算子：両方の分岐の型を見る

int a = 10;
String s = a > 0 ? "plus" : "zero"; // OK

Object o = a > 0 ? "plus" : 0;      // OK。String と int は共通上位のObjectとして扱える

// int x = a > 0 ? 1 : "zero";      // コンパイルエラー

三項演算子は式です。結果を代入するなら、2つ目と3つ目の式の型が代入先に収まるかを見ます。片方だけ見てはいけません。

println：出力される値と戻り値を混同しない

System.out.println("A"); // Aを表示する。戻り値はvoid

// String s = System.out.println("A"); // コンパイルエラー

println は表示するだけで、値を返しません。戻り値は void です。メソッド呼び出しの「副作用」と「戻り値」を分けて見ます。

Stringメソッド：戻り値を受け取らないと元は変わらない

String s = "abc";
s.toUpperCase();
System.out.println(s); // abc

s = s.toUpperCase();
System.out.println(s); // ABC

String は不変です。replace、substring、trim、toUpperCase などは、新しい文字列を返すだけです。戻り値を代入しなければ元の変数の参照は変わりません。

StringBuilderメソッド：自分自身を変更して返すものが多い

StringBuilder sb = new StringBuilder("abc");
StringBuilder ret = sb.append("d");
System.out.println(sb);       // abcd
System.out.println(ret);      // abcd
System.out.println(sb == ret); // true

StringBuilder は可変です。append、insert、delete、reverse などは、基本的に自分自身を変更します。戻り値は同じインスタンスへの参照であることが多いので、== がtrueになるケースがあります。

substring・charAt・indexOfの境界

String s = "abcde";
System.out.println(s.charAt(0));      // a
System.out.println(s.charAt(4));      // e
// System.out.println(s.charAt(5));   // StringIndexOutOfBoundsException

System.out.println(s.substring(1, 3)); // bc
System.out.println(s.indexOf("cd"));  // 2
System.out.println(s.indexOf("x"));   // -1
  • charAt の添字は 0 から length() - 1 まで。
  • substring(begin, end) は beginを含み、endを含まない。
  • indexOf は見つからないと -1 を返す。例外ではない。

配列・List：lengthとsize()を混同しない

int[] array = {1, 2, 3};
System.out.println(array.length); // 3

ArrayList<String> list = new ArrayList<>();
list.add("A");
System.out.println(list.size());  // 1

// array.length()  // コンパイルエラー
// list.length     // コンパイルエラー

配列はフィールド風の length、Stringはメソッドの length()、Listは size() です。この3つは頻出です。

ArrayList：addとsetの違い

ArrayList<String> list = new ArrayList<>();
list.add("A");        // [A]
list.add(0, "B");     // [B, A]
list.set(1, "C");     // [B, C]

// list.set(2, "D");  // IndexOutOfBoundsException
// list.add(3, "E");  // IndexOutOfBoundsException
  • add(index, value) は指定位置に挿入する。indexは 0 から size() まで。
  • set(index, value) は既存要素を置き換える。indexは 0 から size() - 1 まで。
  • set は要素数を増やさない。

拡張for文：変数に代入しても配列要素は置き換わらない

int[] nums = {1, 2, 3};
for (int n : nums) {
  n = 10;
}
System.out.println(nums[0]); // 1

拡張for文の変数 n は、要素の値を受け取るローカル変数です。プリミティブ配列では、n に代入しても元の配列要素は変わりません。

StringBuilder[] arr = { new StringBuilder("A") };
for (StringBuilder sb : arr) {
  sb.append("B");
}
System.out.println(arr[0]); // AB

参照型の場合、変数に入るのは参照のコピーです。参照先のオブジェクトを変更すれば、元の配列から見える中身も変わります。

for文：初期化・条件・更新式は省略できる

for (;;) {
  break;
}

int i = 0;
for (; i < 3; ) {
  System.out.println(i);
  i++;
}

for(初期化; 条件; 更新) の各部分は省略できます。ただしセミコロン2つは必要です。for(;;) は無限ループです。

do-while：最後のセミコロンが必要

int i = 0;
do {
  System.out.println(i);
  i++;
} while (i < 1); // ; が必要

do-while は最後に ; が必要です。while 文や for 文のブロック終端とは扱いが違います。

ラベル付きbreak/continue

outer:
for (int i = 0; i < 3; i++) {
  for (int j = 0; j < 3; j++) {
    if (j == 1) break outer;
    System.out.println(i + ":" + j);
  }
}
// 0:0

ラベル付き break は指定ラベルの文を抜けます。ネストしたループ問題では、通常のbreakなのかラベル付きbreakなのかを必ず確認します。

final変数：定数式かどうかで結果が変わる

final int a = 10;
byte b = a; // OK。コンパイル時定数でbyte範囲内

int x = 10;
// byte y = x; // コンパイルエラー

final が付いていて、コンパイル時に値が確定していて、代入先の範囲内なら狭い型へ代入できる場合があります。final があれば何でもOKではありません。

final int a;
a = 10;
// byte b = a; // 文脈によってはコンパイル時定数として扱えない

「宣言時にリテラルで初期化されたfinal変数」かどうかを確認します。

複合代入：暗黙キャストが入る

byte b = 127;
b += 1;
System.out.println(b); // -128

// b = b + 1; // コンパイルエラー。b + 1 はint

b += 1 は b = (byte)(b + 1) に近い動きをします。コンパイルは通りますが、オーバーフローは起きます。

charの演算：結果は基本int

char c = 'A';
System.out.println(c + 1); // 66

// char d = c + 1; // コンパイルエラー
char e = (char)(c + 1); // OK
System.out.println(e); // B

char は文字ですが、数値として演算できます。ただし算術演算では基本的に int へ昇格します。

ラッパークラス：==とアンボクシング

Integer a = 100;
Integer b = 100;
System.out.println(a == b); // trueになり得る。キャッシュ範囲

Integer x = 1000;
Integer y = 1000;
System.out.println(x == y); // falseになり得る

System.out.println(x.equals(y)); // true

ラッパークラスで == を使うと、参照比較になる場合があります。数値として比較したいなら、基本は equals かアンボクシング後の比較を考えます。

Integer n = null;
// int x = n; // NullPointerException。アンボクシング時にnull参照

nullのラッパーをプリミティブに代入すると、アンボクシングで実行時例外です。

メソッド呼び出し：オーバーロードはコンパイル時に決まる

static void test(Object o) { System.out.println("Object"); }
static void test(String s) { System.out.println("String"); }

Object x = "abc";
test(x); // Object

test("abc"); // String

オーバーロードは、実引数の実体ではなく、コンパイル時の型で選ばれます。オーバーライドとは別物です。

オーバーライド：実行時の実体で決まる

class A {
  void test() { System.out.println("A"); }
}
class B extends A {
  @Override void test() { System.out.println("B"); }
}
A a = new B();
a.test(); // B

オーバーライドされたインスタンスメソッドは、実行時の実体で呼び出し先が決まります。フィールドやstaticメソッドとは違います。

staticメソッド：参照型で解決される

class A {
  static void test() { System.out.println("A"); }
}
class B extends A {
  static void test() { System.out.println("B"); }
}
A a = new B();
a.test(); // A。非推奨な書き方だが、参照型Aで解決

staticメソッドはオーバーライドされません。隠蔽です。変数の宣言型で解決される点がインスタンスメソッドと違います。

フィールド：宣言型で参照される

class A { String name = "A"; }
class B extends A { String name = "B"; }

A a = new B();
System.out.println(a.name); // A

フィールドはオーバーライドされません。参照変数の型でどのフィールドを見るかが決まります。

コンストラクタ：this() と super() は先頭だけ

class A {
  A() { }
}
class B extends A {
  B() {
    super(); // OK。先頭
  }
}
class B extends A {
  B() {
    System.out.println("B");
    // super(); // コンパイルエラー。先頭でなければならない
  }
}

コンストラクタ内で this(...) または super(...) を明示する場合、必ず最初の文です。両方を同時に書くこともできません。

try-catch-finally：returnがあってもfinallyは動く

static int test() {
  try {
    return 1;
  } finally {
    System.out.println("finally");
  }
}
System.out.println(test());

// 出力
// finally
// 1

return が実行される場合でも、メソッドを抜ける前に finally が実行されます。

static int test() {
  try {
    return 1;
  } finally {
    return 2;
  }
}
System.out.println(test()); // 2

finally内でreturnすると、try側のreturnを上書きします。ただし、実務では避けるべき書き方です。試験では出力判定の罠として出ます。

catch順序：広い例外を先に書くとコンパイルエラー

try {
  throw new java.io.IOException();
} catch (Exception e) {
} catch (java.io.IOException e) { // コンパイルエラー。到達不能
}

catchは上から評価されます。先に広い型で捕まえると、後ろの狭い型のcatchには到達できません。

try-with-resources：closeは逆順

class R implements AutoCloseable {
  String name;
  R(String name) { this.name = name; }
  public void close() { System.out.println(name); }
}

try (R a = new R("A"); R b = new R("B")) {
  System.out.println("body");
}

// 出力
// body
// B
// A

リソースは宣言した順ではなく、逆順にcloseされます。finallyがある場合、closeの後にfinallyが実行されます。

null：コンパイルエラーではなく実行時例外になりやすい

String s = null;
// System.out.println(s.length()); // NullPointerException

System.out.println(s); // null と表示される

null参照に対してメソッドやフィールドへアクセスすると NullPointerException です。ただし、System.out.println(s) はnullを文字列として表示するため例外になりません。

record：equalsは中身比較が自動生成される

record Item(String name, int price) {}

Item a = new Item("apple", 100);
Item b = new Item("apple", 100);
System.out.println(a == b);      // false
System.out.println(a.equals(b)); // true

recordは、全コンポーネントを使った equals、hashCode、toString が自動生成されます。通常クラスのデフォルトequalsとは違います。

sealed：継承許可とサブクラス修飾子を両方見る

sealed class A permits B, C {}
final class B extends A {}
non-sealed class C extends A {}

// class D extends A {} // コンパイルエラー。permitsにない

sealedクラスを継承できるのは permits に列挙されたクラスだけです。また、直接のサブクラスは final、sealed、non-sealed のいずれかで修飾します。
