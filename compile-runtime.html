Compile vs Runtime

コンパイルエラーと例外の違い

Java Silverで最初に判定すべきなのは出力ではありません。まずコンパイルできるか。コンパイルできるなら実行時例外があるか。最後に出力です。

最重要フロー

  1. コンパイルエラー：Javaの文法・型・宣言ルールに違反している。実行されない。
  2. 実行時例外：コンパイルは通るが、実行中に問題が起きる。
  3. 正常出力：コンパイルも実行も通った後に初めて出力を追う。

コンパイルエラーになりやすいもの

分類例見抜き方
構文ミス;不足、{}不一致、package/import順序違いまず記号を確認。特に問題文の行番号に注意。
型不一致longをintへ代入、doubleをfloatへ代入、booleanに数値代入先の型と式全体の型を見る。
未初期化ローカル変数int x; System.out.println(x);ローカル変数は自動初期化されない。
到達不能コードreturn後の通常文コンパイラが絶対に実行不能と判断するとエラー。
override違反戻り値不正、アクセスを狭くする、throwsを広げる@Overrideがあると検出しやすい。
catch順序Exceptionの後にIOExceptionをcatch親例外を先にcatchすると子が到達不能。
checked例外未処理IOExceptionをcatch/throwsしないRuntimeException以外の例外は処理義務がある。
static文脈staticメソッドから非staticフィールドへ直接アクセスインスタンスがない場所からインスタンスメンバは触れない。
var誤用フィールドvar、var a;、var b=null;varは初期化ありのローカル変数だけ。

実行時例外になりやすいもの

例外典型例試験での見方
NullPointerExceptionnull.toString(), null.equals(...)null変数からインスタンスメソッド/フィールド参照。
ArrayIndexOutOfBoundsExceptionarray[array.length]添字は0〜length-1。
StringIndexOutOfBoundsExceptioncharAt(length), substring範囲外Stringのindexも0始まり。
ArithmeticExceptionint x = 10 / 0;整数の0除算。浮動小数点ではInfinity/NaNになり得る。
ClassCastException親型参照を実体と無関係な子型へキャストコンパイル時の型変換可能性と実体の型は別。
NumberFormatExceptionInteger.parseInt("abc")文字列が数値形式でない。
UnsupportedOperationExceptionList.of(...).add(...)不変コレクションの変更操作。

コンパイルエラーではなく例外になる例

String s = null;
System.out.println(s.length()); // コンパイル可。実行時にNullPointerException
Object obj = "abc";
Integer i = (Integer) obj; // コンパイル可。実行時にClassCastException
int[] a = new int[2];
System.out.println(a[2]); // コンパイル可。実行時にArrayIndexOutOfBoundsException

例外ではなくコンパイルエラーになる例

int x;
System.out.println(x); // ローカル変数未初期化
float f = 1.0; // doubleをfloatへ暗黙代入不可
try {
} catch (Exception e) {
} catch (RuntimeException e) { // 到達不能
